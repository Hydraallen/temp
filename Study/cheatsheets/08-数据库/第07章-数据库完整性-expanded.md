## 一、本章概览

本章解决的核心问题是：如何保证数据库中数据的**正确性、有效性和相容性**，防止不合语义的数据进入数据库。内容分为六大块：完整性基础概念（完整性 vs 安全性、DBMS 三大功能）、三大完整性约束（实体完整性、参照完整性、用户自定义完整性）、完整性约束命名与域、触发器。本章是本学科**最高频考点**（考试信号频次 44 次，全学科第一），几乎所有 SQL 约束关键字、触发器语法细节均可出题。

## 二、完整性基础概念

### 2.1 完整性的定义 ⭐

**数据的正确性**是指数据是符合现实世界语义、反映当前实际状况的。*(讲义 p68)*

**数据的相容性**是指数据库同一对象在不同关系表中的数据是符合逻辑的。*(讲义 p68)*

> Caveat: 数据库的完整性 = 正确性 + 相容性，二者缺一不可。注意"相容性"不是"符合现实世界语义"——那是"正确性"的定义。*(第七章-判断-2)*

### 2.2 完整性与安全性 ⭐

| 维度 | 完整性 | 安全性 |
|---|---|---|
| 目的 | 防止数据库中存在**不符合语义**的数据（不正确的数据） | 保护数据库防止**恶意破坏和非法存取** |
| 防范对象 | 不合语义的、不正确的数据 | 非法用户和非法操作 |
| 防止结果 | 防止它们进入数据库 | 防止对数据库数据的非法存取 |

*(讲义 p68)*

> Caveat: 完整性防范"不合语义的、不正确的数据"（多选选 AC），安全性防范"非法用户和非法操作"（多选选 AD），二者防范对象完全不同，不可混淆。*(第七章-多选-1, 第七章-多选-2)*

### 2.3 DBMS 的完整性功能 ⭐

DBMS 的完整性功能包括三个方面：*(讲义 p68)*

| 功能 | 说明 |
|---|---|
| **提供定义完整性约束条件的机制** | 完整性约束条件（也称完整性规则）是数据库中的数据必须满足的语义约束条件，表达数据及其联系的制约和依存规则，限定符合数据模型的数据库状态及状态变化 |
| **提供完整性检查的方法** | 一般在 `INSERT`、`UPDATE`、`DELETE` 语句执行后开始检查，也可以在事务提交时检查 |
| **进行违约处理** | 若违背约束条件，采取拒绝执行该操作或级联（CASCADE）执行其他操作 |

> Caveat: "预防非法用户侵入"属于安全性功能，不属于完整性功能（多选 BCD，不选 A）。*(第七章-多选-3)*

## 三、实体完整性

### 3.1 定义实体完整性 ⭐

关系模型的实体完整性在 `CREATE TABLE` 中用 `PRIMARY KEY` 定义。*(讲义 p68)*

对单属性构成的码有**两种**说明方法：*(讲义 p68)*
1. 定义为**列级约束条件**
2. 定义为**表级约束条件**

对多个属性构成的码只有**一种**说明方法，即定义为**表级约束条件**。*(讲义 p68)*

> Caveat (第七章-单选-4): 多个属性构成的码**只能**定义为表级约束，**不能**定义为列级约束。

**示例**（列级约束，主键直接跟在列名后）：*(讲义 p68)*

```sql
CREATE TABLE STUDENT
(SNO   CHAR(9)  PRIMARY KEY,
 SNAME CHAR(20) NOT NULL,
 SSEX  CHAR(2),
 SAGE  SMALLINT,
 SDEPT CHAR(20));
```

> Caveat: 上例中 `SNO CHAR(9) PRIMARY KEY` 直接定义在列名之后，属于**列级约束**。*(第七章-单选-9)*

### 3.2 实体完整性约束条件 ⭐

用 `PRIMARY KEY` 定义主码后，每当用户程序对基本表**插入**一条记录或对主码列进行**更新**操作时，DBMS 自动检查：*(讲义 p68)*

1. **检查主码值是否唯一**，如果不唯一则拒绝插入或修改。
2. **检查主码的各个属性是否为空**，只要有一个为空就拒绝插入或修改。

> Caveat: 实体完整性要求主属性**非空且唯一**。主码不能为 NULL（插入主码为 NULL 的记录会被拒绝）。*(第七章-单选-6, 第七章-单选-8)*

> Caveat: 实体完整性是针对**主码**进行的操作，不是对外码。*(第七章-判断-3)*

检查主码值是否唯一的一种方法是**全表扫描**，依次判断表中每一条记录的主码值与将插入记录的主码值（或修改的新主码值）是否相同。*(讲义 p68)*

## 四、参照完整性

### 4.1 定义参照完整性 ⭐

参照完整性在 `CREATE TABLE` 中用 `FOREIGN KEY` 短语定义哪些列为外码，用 `REFERENCES` 短语指明这些外码参照哪些表的主码。*(讲义 p68)*

**语法格式**：*(讲义 p68)*

```sql
FOREIGN KEY <外码列名> REFERENCES <被参照表名>(<主码列名>)
```

> Caveat: 参照完整性可以通过建立**主键约束和外键约束**来实现（单选选 B）。*(第七章-多选-4)*

> Caveat: 被参照表的列应具有 `PRIMARY KEY` 约束。*(第七章-单选-12)*

### 4.2 参照完整性约束条件 ⭐

参照完整性将两个表中的相应元组联系起来，因此对**被参照表**和**参照表**进行增、删、改操作时有可能破坏参照完整性，必须进行检查以保证这两个表的相容性。*(讲义 p68)*

**可设置的级联操作选项**：*(讲义 p68)*

| 级联操作 | 是否支持 |
|---|---|
| 级联更新相关字段 | ✅ 支持 |
| 级联删除相关字段 | ✅ 支持 |
| 级联插入相关字段 | ❌ 不存在 |
| 级联查询相关字段 | ❌ 不存在 |

> Caveat: 级联操作只有**级联更新**和**级联删除**两种，没有级联插入、级联查询。*(第七章-多选-5)*

> Caveat: 保证相关表之间数据的一致性，必须在关联表中定义**主键和外键**。*(第七章-判断-6)*

## 五、用户自定义完整性

### 5.1 用户自定义完整性概念 ⭐

用户定义的完整性就是针对某一具体应用的数据必须满足的语义要求。*(讲义 p68)*

目前的关系 DBMS 都提供了定义和检验这类完整性的机制，使用了和实体完整性、参照完整性**相同的技术和方法**来处理它们，而不必由应用程序承担这一功能。*(讲义 p68-69)*

### 5.2 属性上的约束条件 ⭐

**（1）定义方式**：用 `CHECK` 短语指定列值应该满足的条件。*(讲义 p69)*

**（2）检查和违约处理**：当往表中插入元组或修改属性的值时，DBMS 检查属性上的约束条件是否被满足，如果不满足则操作被**拒绝执行**。*(讲义 p69)*

> Caveat: 可用于定义属性上约束条件的短语包括 `NOT NULL`、`UNIQUE`、`CHECK`，**不包括** `HAVING`（HAVING 用于 GROUP BY 后的条件语句）。*(第七章-多选-7)*

> Caveat: `UNIQUE` 约束强制了数据的属性值限制。*(第七章-单选-15)*

### 5.3 元组上的约束条件 ⭐

**（1）定义**：在 `CREATE TABLE` 语句中用 `CHECK` 短语定义元组上的约束条件（元组级的限制）。与属性值限制相比，元组级的限制可以设置**不同属性之间的取值的相互约束条件**。*(讲义 p69)*

**（2）检查和违约处理**：当往表中插入元组或修改属性的值时，DBMS 检查元组上的约束条件是否被满足，如果不满足则操作被**拒绝执行**。*(讲义 p69)*

> Caveat: 元组级的限制可以设置**不同属性之间**的取值的相互约束条件，这是与属性级约束的关键区别。*(第七章-单选-14)*

## 六、完整性约束命名与域

### 6.1 完整性约束命名子句 ⭐

SQL 在 `CREATE TABLE` 语句中提供 `CONSTRAINT` 子句，用来对完整性约束条件命名，从而可以灵活地增加、删除一个完整性约束条件。*(讲义 p69)*

**语法格式**：*(讲义 p69)*

```sql
CONSTRAINT <完整性约束条件名> <完整性约束条件>;
```

`<完整性约束条件>` 包括：`NOT NULL`、`UNIQUE`、`PRIMARY KEY`、`FOREIGN KEY`、`CHECK` 等。*(讲义 p69)*

**修改表中的完整性限制**：可以使用 `ALTER TABLE` 语句修改表中的完整性限制。*(讲义 p69)*

> Caveat: 完整性约束命名子句的关键字是 `CONSTRAINT`。*(第七章-单选-16)*

### 6.2 域中的完整性限制 ⭐

域是一组具有相同数据类型的值的集合。SQL 支持域的概念，可以用 `CREATE DOMAIN` 语句建立一个域以及该域应该满足的完整性约束条件，然后就可以用域来定义属性。*(讲义 p69)*

**优点**：数据库中不同的属性可以来自同一个域，当域上的完整性约束条件改变时只要修改域的定义即可，而不必一一修改域上的各个属性。*(讲义 p69)*

> Caveat: 域完整性又称为**列完整性**。*(第七章-单选-17)*

> Caveat: 域完整性**能够**有效控制列的数据的数据类型、格式和有效取值范围（判断题说"无法约束"是错误的）。*(第七章-判断-7)*

> Caveat: 保证数据列的取值范围正确的是**域完整性**和**参照完整性**，不涉及实体完整性；默认值无法控制取值范围。*(第七章-多选-8)*

## 七、触发器

### 7.1 触发器概念 ⭐

任何用户对表的增、删、改操作均由服务器自动激活相应的触发器。*(讲义 p69)*

触发器可以实施更为复杂的检查和操作，具有更精细和更强大的数据控制能力。*(讲义 p69)*

触发器又叫做**事件-条件-动作规则**：当特定的系统事件发生时，对规则的条件进行检查，如果条件成立则执行规则中的动作，否则不执行该动作。规则中的动作体可以很复杂，可以涉及其他表和其他数据库对象，通常是一段 SQL 存储过程。*(讲义 p69)*

### 7.2 定义触发器 ⭐

**CREATE TRIGGER 语法格式**：*(讲义 p69)*

```sql
CREATE TRIGGER <触发器名>
{BEFORE | AFTER} <触发事件> ON <表名>
REFERENCING NEW|OLD ROW AS <变量>
FOR EACH {ROW | STATEMENT};
[WHEN <触发条件>] <触发动作体>
```

**语法详细说明**：*(讲义 p69-70)*

| 要素 | 说明 |
|---|---|
| **权限** | 只有表的拥有者（创建表的用户）才可以在表上创建触发器；一个表上只能创建一定数量的触发器 |
| **触发器名** | 可以包含模式名，也可以不包含模式名；同一模式下触发器名必须唯一；触发器名和表名必须在同一模式下 |
| **表名** | 触发器**只能定义在基本表上**，不能定义在视图上；该表也称触发器的目标表 |
| **触发事件** | 可以是 `INSERT`、`DELETE` 或 `UPDATE`，也可以是组合；还可以是 `UPDATE OF <触发列,…>` |
| **触发时机** | `AFTER`：触发事件操作执行**之后**激活；`BEFORE`：触发事件操作执行**之前**激活 |
| **触发器类型** | **行级触发器**（`FOR EACH ROW`）和**语句级触发器**（`FOR EACH STATEMENT`） |
| **触发条件** | `WHEN` 条件为真时才执行动作体；省略 `WHEN` 则激活后立即执行 |
| **触发动作体** | 可以是匿名 PL/SQL 过程块或对已创建存储过程的调用；执行失败则激活事件终止，目标表不发生变化 |

> Caveat: 触发器名**可以**包含模式名，也**可以不**包含模式名（判断题说"必须包含模式名"是错误的）。*(第七章-判断-9)*

> Caveat: 触发器**只能定义在基本表上**，不能定义在视图上。*(第七章-单选-19)*

> Caveat: 触发器按照触发动作的间隔尺寸分为**行级触发器**和**语句级触发器**。*(第七章-单选-20)*

> Caveat: 触发器可以完成的功能包括：强制执行远复杂于 CHECK 约束的限制约束、数据库中相关表的级联操作、跨数据库创建触发器、触发器递归调用（ABCD 均正确）。*(第七章-多选-9)*

### 7.3 激活触发器 ⭐

触发器的执行由触发事件激活，并由**数据库服务器自动执行**。*(讲义 p70)*

同一个表上多个触发器激活时的**执行顺序**：*(讲义 p70)*

| 执行顺序 | 操作 |
|---|---|
| ① | 执行该表上的 **BEFORE** 触发器 |
| ② | 激活触发器的 **SQL 语句** |
| ③ | 执行该表上的 **AFTER** 触发器 |

对于同一个表上的多个 BEFORE（AFTER）触发器，遵循**"谁先创建谁先执行"**的原则，即按照触发器创建的时间先后顺序执行。有些 DBMS 按照触发器名称的字母排序顺序执行。*(讲义 p70)*

> Caveat: 触发器的执行由触发事件激活，并由**数据库服务器自动**执行（不是 DBA 启动、不是操作系统自动、不是用户启动）。*(第七章-多选-10)*

### 7.4 删除触发器

**删除触发器的 SQL 语法**：*(讲义 p70)*

```sql
DROP TRIGGER <触发器名> ON <表名>;
```

触发器必须是一个已经创建的触发器，并且只能由具有相应权限的用户删除。*(讲义 p70)*

> Caveat: 删除触发器使用 `DROP TRIGGER`，**不是** `DELETE TRIGGER`。*(第七章-判断-10)*

### 7.5 慎重使用触发器 ⚠

触发器是一种功能强大的工具，但在使用时要慎重，因为在每次访问一个表时都可能触发一个触发器，这样会**影响系统的性能**。*(讲义 p70)*

## 八、三大完整性对比表 ⭐

| 对比维度 | 实体完整性 | 参照完整性 | 用户自定义完整性 |
|---|---|---|---|
| **针对对象** | 主码 | 外码 | 具体应用的语义要求 |
| **SQL 关键字** | `PRIMARY KEY` | `FOREIGN KEY` + `REFERENCES` | `CHECK`、`NOT NULL`、`UNIQUE` 等 |
| **约束内容** | 主码值唯一且非空 | 外码值必须在被参照表主码中存在或为空 | 列值/元组值满足特定条件 |
| **定义位置** | `CREATE TABLE` 中 | `CREATE TABLE` 中 | `CREATE TABLE` 中 |
| **检查时机** | 插入/更新时 | 增/删/改（被参照表和参照表）时 | 插入/修改属性值时 |
| **违约处理** | 拒绝插入或修改 | 拒绝执行或级联操作 | 拒绝执行 |

*(讲义 p68-69)*

## 九、可应用于单一列的约束辨析 ⚠

| 约束 | 能否应用于单一列 | 说明 |
|---|---|---|
| `UNIQUE` | ✅ 可以 | 唯一约束 |
| `CHECK` | ✅ 可以 | 检查约束 |
| `PRIMARY KEY` | ✅ 可以 | 主键约束（单属性时） |
| `FOREIGN KEY` | ❌ 不可以 | 至少涉及两个列（外码列 + 被参照主码列） |

> Caveat: `FOREIGN KEY` 至少涉及了两个列，不能仅应用在单一列上。*(第七章-多选-6)*

## 十、高频考点与易错提醒

1. **完整性 = 正确性 + 相容性**（高频填空/单选）。正确性 = 符合现实世界语义；相容性 = 同一对象在不同表中符合逻辑。*(第七章-单选-1, 第七章-判断-1, 第七章-判断-2)*
2. **完整性 vs 安全性**的防范对象是必考对比题。完整性防"不合语义、不正确的数据"；安全性防"非法用户和非法操作"。*(第七章-多选-1, 第七章-多选-2)*
3. **多属性主码只能用表级约束**定义，单属性主码既可列级也可表级。*(第七章-单选-4)*
4. **主码不能为 NULL**，插入主码为 NULL 的记录违反实体完整性，会被拒绝。*(第七章-单选-8)*
5. **级联操作只有级联更新和级联删除**，没有级联插入和级联查询。*(第七章-多选-5)*
6. **HAVING 不能用于定义属性约束**，属性约束用 `NOT NULL`、`UNIQUE`、`CHECK`。*(第七章-多选-7)*
7. **域完整性 = 列完整性**，能控制数据类型、格式和有效取值范围。*(第七章-单选-17, 第七章-判断-7)*
8. **触发器只能定义在基本表上**，不能定义在视图上。*(第七章-单选-19)*
9. **删除触发器用 `DROP TRIGGER`**，不是 `DELETE TRIGGER`。*(第七章-判断-10)*
10. **触发器名可以不包含模式名**（非必须）。*(第七章-判断-9)*
11. **触发器由数据库服务器自动执行**，不需要 DBA/用户/操作系统启动。*(第七章-多选-10)*
12. **表只能有一个主码，但可以有任意个 UNIQUE 约束**。*(第七章-判断-4)*

## 十一、一句话记忆要点

- **完整性** = 正确性（符合现实语义）+ 相容性（不同表中逻辑一致）。
- **完整性 vs 安全性**：完整性防"坏数据"，安全性防"坏人/坏操作"。
- **DBMS 三大功能**：定义约束 → 检查约束 → 违约处理（拒绝/级联）。
- **实体完整性**：`PRIMARY KEY`，主码唯一且非空；多属性码只能表级定义。
- **参照完整性**：`FOREIGN KEY ... REFERENCES`，级联只有更新和删除。
- **用户自定义完整性**：`CHECK`/`NOT NULL`/`UNIQUE`；元组级可约束不同属性间关系。
- **CONSTRAINT**：给约束命名，方便增删；**CREATE DOMAIN**：定义域，改一处即生效。
- **触发器** = 事件-条件-动作规则；只能建在基本表上；`DROP TRIGGER` 删除；慎用以免影响性能。
- **触发器执行顺序**：BEFORE → SQL 语句 → AFTER；同类按创建时间先后执行。

## Worked Examples

### 例 1：三大完整性类型判定 *(来自本章 expanded 三大完整性对比表)*
*Tags: 实体完整性, 参照完整性, 用户定义完整性*
**问题**：判断下列约束分别属于哪一类完整性：
（1）学生表中"学号"字段定义为主码；
（2）选课表中"学号"定义为外码，参照学生表的"学号"；

### 例 2：触发器与多属性主码定义 *(来自本章 expanded 触发器与实体完整性语法)*
*Tags: 触发器, CREATE TRIGGER, 多属性主码, 表级约束*
**问题**：以下两段 SQL 是否正确？说明理由。
（1）在视图 V 上创建触发器：`CREATE TRIGGER tri_v AFTER INSERT ON V ...`
（2）在学生选课表 SC 中定义主码 (SNO, CNO)：`SNO CHAR(9) PRIMARY KEY, CNO CHAR(4) PRIMARY KEY`*(题目第七章-单选-4,19)*
**解答**：`[PROPOSED — verify]`
1. 语句（1）错误：触发器**只能定义在基本表上**，不能定义在视图上。*(讲义 p69)*
2. 语句（2）错误：多个属性构成的码**只能定义为表级约束**，不能定义为列级约束。*(讲义 p68)*
3. 正确写法（表级）：
   ```sql
   CREATE TABLE SC (
     SNO CHAR(9),
     CNO CHAR(4),
     PRIMARY KEY (SNO, CNO)
   );
   ```
4. 删除触发器用 `DROP TRIGGER`（不是 `DELETE TRIGGER`）；触发器由数据库服务器自动执行，按 BEFORE → SQL 语句 → AFTER 顺序激活。*(讲义 p70)*
