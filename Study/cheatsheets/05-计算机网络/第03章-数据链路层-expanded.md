## 基本概念与模型
**链路 (Link)**：一条无源的点到点的物理线路段，中间没有任何交换结点，也称物理链路。*(讲义 p51)*
**数据链路 (Data Link)**：也称逻辑链路，是指物理链路加上必要的通信规程。*(讲义 p51)*
$$数据链路 / 逻辑链路 = (物理)链路 + 通信规程$$*(讲义 p51)*

**简化通信模型**：把数据链路层以上的各层抽象为一个主机，把物理层和传输媒体抽象成一条简单的链路。早期通信常简称为“结点”与“链路”。*(讲义 p51)*

## 数据链路层的功能 ⭐
数据链路层负责在物理层提供的服务基础上向网络层提供服务，其核心功能包括：*(讲义 p52)*
1. **链路管理**：数据链路的建立、维持和释放。在 OSI/RM 中须提供面向连接的可靠服务；但在 TCP/IP 体系下（因网际层 IP 不可靠），数据链路层并不提供可靠的服务。*(讲义 p52)*
2. **帧同步**：收方应当能从收到的比特流中准确地区分出一帧的开始和结束。主要方法有：字节计数法、字符填充首尾定界符法、**比特填充首尾定界符法**（HDLC 采用）、违法编码法。*(讲义 p52)*
3. **流量控制**：控制发送方发送数据的速率，使收发双方达到匹配。在数据链路层上，控制的是**相邻两节点之间**数据链路上的流量。*(讲义 p52)*
4. **差错控制**：计算机通信最常用的差错控制方法是 ARQ（自动反馈重发）。分为：空闲重发（停止等待法）和连续重发（分为 Go-Back-N 策略和选择重发策略）。*(讲义 p52)*
5. **透明传输**：不管所传数据是什么样的比特组合，都应当能够在链路上传送，使收方不会将数据误认为是控制信息。常用方法如零比特填充法和字符填充法。*(讲义 p52)*
6. **寻址**：多点式链路传输数据时，必须保证每一帧都能送到正确的接收方，接收方也应知道发送方是哪一个。*(讲义 p52)*

> Caveat: 数据链路层不提供“电路管理功能”，电路管理属于物理层职责。*(讲义 p62-课后习题2)*

## 数据链路层提供的服务
数据链路层为网络层提供的服务分为以下三种：*(讲义 p53)*
1. **无确认、无连接服务**：目的结点不作确认，差错由上层负责。适用于信道比较可靠并且对通信实时性要求高的网络。*(讲义 p53)*
2. **有确认的无连接服务**：目的节点对收的帧作确认。*(讲义 p53)*
3. **面向连接服务**：可靠地传送数据的服务，具有建立连接、维持连接、释放连接的过程。*(讲义 p53)*

## 流量控制与可靠传输机制
实现可靠传输的协议前提是打破“理想信道”和“接收方无限缓存”这两个假定。解决流量控制与差错控制的基本方案分为等待发送（停止等待协议）和连续发送两类。*(讲义 p54)*

### 1. 简单的停止等待协议 ⭐
**前提**：链路是理想的信道，数据既不会出错也不会丢失。*(讲义 p54)*
**方案**：仅解决流量控制问题。发送端每发完一帧必须停下来，只有收到接收端的确认（ACK）才能继续发送下一帧；接收端每正确收到一帧才发送一个确认。*(讲义 p54)*

### 2. 实用的停止等待协议
**前提**：不满足理想信道假定，需考虑数据帧出错、数据帧丢失、应答帧丢失。*(讲义 p54)*
**解决方法**：*(讲义 p54)*
- **数据帧出错**：对数据帧进行校验，出错重发。
- **数据帧丢失**：设置超时定时器，超时重发。
- **应答帧丢失**：设置超时定时器，以序号标识数据帧（解决重复帧问题）。
**特点**：优点是比较简单，缺点是信道利用率不高。*(讲义 p55)*

### 3. 连续 ARQ (Go-Back-N) 协议 ⭐
**工作原理**：发送端每发送完一个数据帧后，不停下来等待确认，而是可以连续再发送若干帧。若收到确认，继续发送；若收到否认帧或超时，则**从该帧开始的后续帧全部重发**。*(讲义 p55)*
**缓冲区要求**：要求发送端的发送缓冲区必须能缓存多个数据帧，而接收端的缓冲区只需要能容纳**一个**数据帧。*(讲义 p55)*
**特点**：因连续发送提高了传输效率，但在重传时必须把原来已正确传送过的数据帧进行重传，使传输效率降低。若信道传输质量很差、误码率较高时，不一定优于停止等待协议。*(讲义 p55)*

### 4. 选择重传 ARQ (Selective Repeat) 协议 ⭐
**工作原理**：设法只重传出错的数据帧或者是定时器超时的数据帧。*(讲义 p55)*
**缓冲区要求**：必须加大接收端的缓冲区使其能容纳**多个**数据帧，以便接收端缓存正确但未按序到达的数据帧（发送序号可能是不连续的），等到所缺序号的数据帧收到后再一并送交主机。要求收发双方的缓冲区都能缓存多个帧。*(讲义 p56)*
**特点**：可避免重传那些已经正确到达接收端的数据帧；代价是接收端需要设置具有相当容量的缓存空间，不够经济。*(讲义 p56)*

### 5. 滑动窗口协议 ⭐
发送方和接收方各自维持一个随时间变化可向前滑动的窗口（发送窗口和接收窗口）。它允许发送方发送多个分组而不需等待确认。TCP 的滑动窗口是以字节为单位的。*(讲义 p56)*
**窗口内的四类数据（以发送窗口为例）**：*(讲义 p56)*
1. 已发送并收到确认的数据（不在发送窗口内）。
2. 已发送但未收到确认的数据（位于窗口内）。
3. 允许发送但尚未发送的数据（位于窗口内）。
4. 暂时不允许发送的数据（位于窗口外）。

**滑动规则**：*(讲义 p57)*
1. 凡是已经发送过的数据，在未收到确认之前，都必须暂时保留，以便超时重传时使用。
2. 只有当发送方收到了接收方的确认报文段时，发送方窗口才可以向前滑动几个序号。
3. 当发送方发送的数据经过一段时间没有收到确认（超时计时器控制），就要使用回退 N 步协议，回到最后接收到确认号的地方，重新发送这部分数据。

**基于滑动窗口的协议等价关系** ⚠*(讲义 p63-课后习题9)*
- 发送窗口 = 1，接收窗口 = 1：相当于停止-等待协议。
- 发送窗口 > 1，接收窗口 = 1：回退 N 帧 (GBN) ARQ 协议。
- 发送窗口 > 1，接收窗口 > 1：选择重传 SR 协议。

## 高级数据链路控制协议 (HDLC)

### 概述
HDLC (High level Data Link Control) 是一个在同步网上传输数据、**面向比特**的数据链路层协议。由 IBM 提出的 SDLC 研发而来。*(讲义 p58)*
HDLC 具有两种实现方式：HDLC NRM（正常响应模式，具有永久基站）和 HDLC LAPB（链路访问过程平衡，支持对等链路）。HDLC 是 X.25 栈的一部分。*(讲义 p58)*

### 帧结构与透明传输 ⭐
HDLC 帧结构包含：标志(F)、地址(A)、控制(C)、信息、帧检验序列(FCS)、标志(F)。*(讲义 p58)*
| 字段 | 长度 | 说明与功能 |
|---|---|---|
| **标志 F** | 8 bit | `01111110`（即 6 个连续 1 加两边各一个 0）。用于首尾定界。连续传输两帧时，前一帧的结束 F 可兼作后一帧的起始 F。 |
| **地址 A** | 8 bit | 多终端线路区分终端；点到点线路区分命令和响应。若地址是接收站的地址则为命令帧；若是发送站的地址则为响应帧。 |
| **控制 C** | 8 bit | 最复杂的字段，实现 HDLC 许多重要功能（决定帧类型）。 |
| **信息** | 可变 | 透明传输区间。 |
| **FCS** | 16 bit | 检验区间从地址字段的第一个比特起，到信息字段的最末一个比特为止。 |
*(讲义 p58)*

**透明传输实现（零比特填充法 / 位填充法）** ⚠：*(讲义 p58)*
- 发送端：当一串比特流数据中有 **5 个连续 1** 时，就立即在其后填入一个 **0**。
- 接收端：找到 F 字段确定帧的边界后扫描比特流，每当发现 5 个连续 1 时，就将其后的一个 0 删除，还原原来的比特流。
- 采用此法可传送任意组合的比特流，实现数据链路层的透明传输。

### 控制域与帧类型
控制域（C）共 8 bit，根据第 1 或第 1~2 个比特的不同，划分出三种类型的帧：*(讲义 p59)*
1. **信息帧**：第 1 位为 `0`。包含序号，使用滑动窗口技术，3 位序号，**发送窗口大小为 7**。捎带确认表示第一个未收到的帧序号。
2. **监控帧**：前两位为 `10`。用于差错控制和流量控制。
   - Type `0`：确认帧 RR (RECEIVE READY)
   - Type `1`：否定性确认帧 REJ (REJECT)（返回 N 帧重传）
   - Type `2`：接收未准备好 RNR (RECEIVE NOT READY)（要求发方停止）
   - Type `3`：选择拒绝 SREJ (SELECTIVE REJECT)（选择性重传）
3. **无序号帧**：前两位为 `11`。可以用来传控制信息，也可在不可靠无连接服务中传数据。
*(讲义 p59)*

**探询 / 结束 (P/F 位)**：命令帧置 “P”，响应帧置 “F”。多终端系统中，计算机置 “P” 允许终端发送数据；终端发向计算机的帧中，最后一个帧置为 “F”，其它置为 “P”。*(讲义 p59)*

## 点对点协议 (PPP) ⭐
全世界使用得最多的数据链路层协议，用于路由器到路由器、家庭用户到 ISP 的点对点连接。*(讲义 p61)*

### 组成部分
1. 一个将 IP 数据报封装到串行链路的方法。
2. 链路控制协议 LCP (Link Control Protocol)：用于建立、配置、测试和管理数据链路。
3. 网络控制协议 NCP (Network Control Protocol)：用于建立和配置不同的网络层协议。
*(讲义 p61)*

### 帧格式
PPP 的帧格式包含：标志、地址、控制、协议、净荷、校验和、标志。*(讲义 p61)*
| 字段 | 长度 | 默认值与说明 |
|---|---|---|
| **标志** | 8 bit | `01111110` (`0x7E`)。首尾定界。 |
| **地址** | 8 bit | `11111111` (`0xFF`)。表示所有站都可以接受该帧。 |
| **控制** | 8 bit | `00000011` (`0x03`)。表明此帧为无编号的帧。 |
| **协议** | 2 字节 | 指明净荷中是哪一种分组（如 `0x0021` 为 IP 数据报，`0xC021` 为 LCP 数据，`0x8021` 为 NCP 数据）。可通过协商变为 1 字节。 |
| **净荷** | 变长 | 默认值 1500 字节，最多可达商定的最大值（≤500 字节[原文此处如此]）。 |
| **校验和** | 2 字节 | 使用 CRC 的帧校验序列 FCS，可通过协商使用 4 字节。 |
*(讲义 p61-62)*

### PPP 协议的特点 ⚠
PPP 协议是不可靠的传输协议。默认不使用序号和确认机制实现可靠传输。*(讲义 p62)*
不提供可靠传输的原因是：*(讲义 p62)*
1. 在数据链路层出现差错概率不大时，使用简单的 PPP 较合理。
2. 在因特网环境下，PPP 的信息字段放入的是 IP 数据报。数据链路层的可靠传输并不能保证网络层的传输也是可靠的。
3. 帧检验序列 FCS 字段可保证无差错接受（具有差错检测能力）。

> Caveat (课后习题12): PPP 支持动态分配 IP 地址和身份验证，并不仅支持 IP 协议（支持多协议复用）。*(讲义 p62-课后习题12)*
> Caveat (课后习题14): PPP 是面向字节的协议，而 HDLC 协议是面向比特的协议。*(讲义 p65-课后习题14)*

### PPP 的透明传输实现 ⚠
当 PPP 用在同步传输链路时，采用硬件完成**比特填充**（和 HDLC 做法一样）；*(讲义 p62)*
当 PPP 用在异步传输时，使用特殊的**字节填充法**。*(讲义 p62)*

## Worked Examples

### 例 1：滑动窗口与 ARQ 协议判定 *(来自 exam-signals 候选 / 本章课后习题9)*
*Tags: 滑动窗口, ARQ, 停止-等待*
**问题**：从滑动窗口的观点看，当发送窗口为 1，接收窗口也为 1 时，相当于 ARQ 的哪种方式？*(讲义 p63-课后习题9)*
**解答**：`[PROPOSED — verify]`
1. 分析停止-等待协议的工作原理：发送方每发送一帧，都要等待接收方的应答信号，之后才能发送下一帧。
2. 此时发送方连续发送的帧数为 1（发送窗口为 1）。
3. 接收方每次也只能接收并缓存当前的一个帧，若未按序到达或未收到确认，不能提前接收后续帧（接收窗口为 1）。
4. 因此，发送窗口=1，接收窗口=1 等价于停止-等待方式。选 C。

### 例 2：PPP 协议透明传输填充方法 *(来自 exam-signals 候选 / 本章课后习题13)*
*Tags: PPP, 透明传输, 填充法*
**问题**：为实现透明传输，PPP 使用的填充方法是？*(讲义 p65-课后习题13)*
**解答**：`[PROPOSED — verify]`
1. PPP 协议在实际应用中根据传输方式的不同采取不同的透明传输填充机制。
2. 若 PPP 运行在同步传输链路上，采用硬件来完成比特填充（和 HDLC 类似）。
3. 若 PPP 运行在异步传输链路上，采用字节填充法。
4. 考试常考“根据传输模式区分填充法”的选项，因此正确表述应能反映出这种根据同步/异步环境切换策略的特性。如果选项为单一策略，需仔细审题；此处强调 PPP 对两种填充方法的使用场景。
