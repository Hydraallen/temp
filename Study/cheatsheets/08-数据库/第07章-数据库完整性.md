## 一、本章概览

本章围绕**数据库完整性**展开，核心问题是：如何保证数据库中的数据始终符合语义约束、正确且相容。内容分为四大块：（1）完整性基础概念及其与安全性的区别；（2）三大完整性机制——实体完整性、参照完整性、用户自定义完整性——的定义与检查；（3）完整性约束的命名、域级限制等进阶手段；（4）触发器的定义、激活、删除及使用注意事项。本章是数据库约束体系的核心章节，与前面关系模型、SQL 语言章节紧密衔接，也是考试高频出题区。

## 二、知识点详解

### 1. 完整性 ⭐（练习单选1，判断1，判断2）

- **定义**：
  - 数据的**正确性**是指数据是符合现实世界语义、反映当前实际状况的。
  - 数据的**相容性**是指数据库同一对象在不同关系表中的数据是符合逻辑的。
- **要点**：完整性 = 正确性 + 相容性，二者缺一不可。

### 2. 完整性和安全性 ⭐（练习单选2，多选1，多选2）

- **完整性**：为了防止数据库中存在不符合语义的数据，也就是防止数据库中存在不正确的数据。
  - 完整性检查和控制的防范对象是**不合语义的、不正确的数据**，防止它们进入数据库。
- **安全性**：保护数据库防止恶意破坏和非法存取。
  - 安全性控制的防范对象是**非法用户和非法操作**，防止他们对数据库数据的非法存取。
- **要点对比**：完整性防"坏数据"，安全性防"坏人/坏操作"。

### 3. 数据库管理系统的完整性功能 ⭐（练习单选3，多选3）

DBMS 的完整性功能包括三个方面：

1. **提供定义完整性约束条件的机制**
   - 完整性约束条件也称为完整性规则，是数据库中的数据必须满足的语义约束条件。它表达了给定的数据模型中数据及其联系所具有的制约和依存规则，用以限定符合数据模型的数据库状态以及状态的变化，以保证数据的正确、有效和相容。

2. **提供完整性检查的方法**
   - DBMS 中检查数据是否满足完整性约束条件的机制称为完整性检查。一般在 `INSERT`、`UPDATE`、`DELETE` 语句执行后开始检查，也可以在事务提交时检查。检查这些操作执行后数据库中的数据是否违背了完整性约束条件。

3. **进行违约处理**
   - DBMS 若发现用户的操作违背了完整性约束条件将采取一定的动作，如**拒绝执行该操作**或**级联（CASCADE）执行其他操作**，进行违约处理以保证数据的完整性。

### 4. 定义实体完整性 ⭐（练习单选4，单选7，单选9）

- **定义方式**：关系模型的实体完整性在 `CREATE TABLE` 中用 `PRIMARY KEY` 定义。
- **单属性构成的码**：有两种说明方法——
  - 定义为**列级约束条件**
  - 定义为**表级约束条件**
- **多个属性构成的码**：只有一种说明方法——定义为**表级约束条件**。

### 5. 实体完整性的约束条件 ⭐⚠（练习单选5，单选6，单选8，判断3，判断4）

用 `PRIMARY KEY` 短语定义了关系的主码后，每当用户程序对基本表插入一条记录或对主码列进行更新操作时，关系数据库管理系统将按照实体完整性规则自动进行检查。包括：

1. **检查主码值是否唯一**，如果不唯一则拒绝插入或修改。
2. **检查主码的各个属性是否为空**，只要有一个为空就拒绝插入或修改。

- **检查方法**：检查记录中主码值是否唯一的一种方法是进行**全表扫描**，依次判断表中每一条记录的主码值与将插入记录的主码值（或者修改的新主码值）是否相同。
- **⚠易错提醒**：实体完整性是针对**主码**进行的操作，不是外码（练习判断3）。表只能有一个主码，但可以有任意个 `UNIQUE` 约束（练习判断4）。

### 6. 定义参照完整性 ⭐（练习单选10，单选11，单选12，多选4，多选6，判断5，判断6）

- **定义方式**：参照完整性在 `CREATE TABLE` 中用 `FOREIGN KEY` 短语定义哪些列为外码，用 `REFERENCES` 短语指明这些外码参照哪些表的主码。
- **语法要点**：
  ```sql
  FOREIGN KEY (<外码列>) REFERENCES <被参照表名>(<主码列>)
  ```
- **实现方式**：参照完整性可以通过建立**主键约束和外键约束**来实现（练习多选4）。

### 7. 参照完整性约束条件 ⭐（练习单选13，多选5）

- 参照完整性将两个表中的相应元组联系起来，因此，对**被参照表**和**参照表**进行增、删、改操作时有可能破坏参照完整性，必须进行检查以保证这两个表的相容性。
- **级联操作**：在设置或编辑表之间的主、外键时，可设置的选项包括**级联更新相关字段**和**级联删除相关字段**（练习多选5）。没有级联插入、级联查询（练习多选5）。

### 8. 用户自定义完整性

- **定义**：用户定义的完整性就是针对某一具体应用的数据必须满足的语义要求。
- **要点**：目前的关系数据库管理系统都提供了定义和检验这类完整性的机制，使用了和实体完整性、参照完整性相同的技术和方法来处理它们，而不必由应用程序承担这一功能。

### 9. 属性上的约束条件 ⭐（练习单选15，多选7）

1. **用 `CHECK` 短语指定列值应该满足的条件**。
2. **属性上约束条件的检查和违约处理**：当往表中插入元组或修改属性的值时，关系数据库管理系统将检查属性上的约束条件是否被满足，如果不满足则操作被拒绝执行。
- **可用于定义属性约束的短语**：`NOT NULL`、`UNIQUE`、`CHECK`（练习多选7）。`HAVING` 短语用于 `GROUP BY` 语句后的条件语句，不能用于定义属性约束（练习多选7）。

### 10. 元组上的约束条件（练习单选14）

1. **元组上约束条件的定义**：与属性上约束条件的定义类似，在 `CREATE TABLE` 语句中可以用 `CHECK` 短语定义元组上的约束条件，即元组级的限制。同属性值限制相比，**元组级的限制可以设置不同属性之间的取值的相互约束条件**。
2. **元组上约束条件的检查和违约处理**：当往表中插入元组或修改属性的值时，关系数据库管理系统将检查元组上的约束条件是否被满足，如果不满足则操作被拒绝执行。

### 11. 完整性约束命名子句（练习单选16）

SQL 在 `CREATE TABLE` 语句中提供了完整性约束命名子句 `CONSTRAINT`，用来对完整性约束条件命名，从而可以灵活地增加、删除一个完整性约束条件。

1. **完整性约束命名子句**：
   ```sql
   CONSTRAINT <完整性约束条件名> <完整性约束条件>;
   ```
   - `<完整性约束条件>`包括 `NOT NULL`、`UNIQUE`、`PRIMARY KEY`、`FOREIGN KEY`、`CHECK` 等。

2. **修改表中的完整性限制**：可以使用 `ALTER TABLE` 语句修改表中的完整性限制。

### 12. 域中的完整性限制 ⭐⚠（练习单选17，多选8，判断7，判断8）

- **域的定义**：一般地，域是一组具有相同数据类型的值的集合。
- **SQL 支持**：SQL 支持域的概念，并可以用 `CREATE DOMAIN` 语句建立一个域以及该域应该满足的完整性约束条件，然后就可以用域来定义属性。
  ```sql
  CREATE DOMAIN <域名> <数据类型> [CHECK <约束条件>];
  ```
- **优点**：数据库中不同的属性可以来自同一个域，当域上的完整性约束条件改变时只要修改域的定义即可，而不必一一修改域上的各个属性。
- **域完整性**：又称为**列完整性**（练习单选17），能够有效控制列的数据的数据类型、格式和有效取值范围（练习判断7）。

### 13. 触发器（练习多选9）

- 任何用户对表的增、删、改操作均由服务器自动激活相应的触发器。
- 触发器可以实施更为复杂的检查和操作，具有更精细和更强大的数据控制能力。
- 触发器可以完成的任务包括：强制执行远复杂于 `CHECK` 约束的限制约束、数据库中相关表的级联操作、跨数据库创建触发器、触发器递归调用（练习多选9）。

### 14. 定义触发器 ⭐⚠（练习单选18，单选19，单选20，判断9）

- **概念**：触发器又叫做**事件-条件-动作规则**。当特定的系统事件发生时，对规则的条件进行检查，如果条件成立则执行规则中的动作，否则不执行该动作。规则中的动作体可以很复杂，可以涉及其他表和其他数据库对象，通常是一段 SQL 存储过程。

- **CREATE TRIGGER 语法格式**：
  ```sql
  CREATE TRIGGER <触发器名>
  {BEFORE | AFTER} <触发事件> ON <表名>
  REFERENCING NEW|OLD ROW AS <变量>
  FOR EACH {ROW | STATEMENT};
  [WHEN <触发条件>] <触发动作体>
  ```

- **语法详细说明**：
  1. 只有表的拥有者即创建表的用户才可以在表上创建触发器，并且一个表上只能创建一定数量的触发器。触发器的具体数量由具体的关系数据库管理系统在设计时确定。
  2. **触发器名**：可以包含模式名，也可以不包含模式名。同一模式下，触发器名必须是唯一的，并且触发器名和表名必须在同一模式下。
  3. **表名**：触发器只能定义在**基本表**上，不能定义在视图上。当基本表的数据发生变化时，将激活定义在该表上相应触发事件的触发器，因此该表也称为触发器的**目标表**。
  4. **触发事件**：可以是 `INSERT`、`DELETE` 或 `UPDATE`，也可以是这几个事件的组合，还可以是 `UPDATE OF <触发列，…>`，即进一步指明修改哪些列时激活触发器。`AFTER`/`BEFORE` 是触发的时机。`AFTER` 表示在触发事件的操作执行之后激活触发器；`BEFORE` 表示在触发事件的操作执行之前激活触发器。
  5. **触发器类型**：按照所触发动作的间隔尺寸可以分为**行级触发器**（`FOR EACH ROW`）和**语句级触发器**（`FOR EACH STATEMENT`）。
  6. **触发条件**：触发器被激活时，只有当触发条件为真时触发动作体才触发，否则不执行。如果省略 `WHEN` 触发条件，则触发动作体在触发器激活后立即执行。
  7. **触发动作体**：既可以是一个匿名 PL/SQL 过程块，也可以是对已创建存储过程的调用。如果触发动作体执行失败，激活触发器的事件（即对数据库的增、删、改操作）就会终止执行，触发器的目标表或触发器可能影响的其他对象不发生任何变化。

- **⚠易错提醒**：触发器名可以包含模式名，也可以不包含模式名（练习判断9）。触发器只能定义在基本表上，不能定义在视图上（练习单选19）。

### 15. 激活触发器（练习多选10）

- 触发器的执行，是由触发事件激活的，并由**数据库服务器自动执行**。
- 同一个表上的多个触发器激活时遵循如下执行顺序：
  1. 执行该表上的 **BEFORE** 触发器。
  2. 激活触发器的 **SQL 语句**。
  3. 执行该表上的 **AFTER** 触发器。
- 对于同一个表上的多个 BEFORE（AFTER）触发器，遵循**"谁先创建谁先执行"**的原则，即按照触发器创建的时间先后顺序执行。有些关系数据库管理系统是按照触发器名称的字母排序顺序执行触发器。

### 16. 删除触发器 ⚠（练习判断10）

- **删除触发器的 SQL 语法**：
  ```sql
  DROP TRIGGER <触发器名> ON <表名>;
  ```
- 触发器必须是一个已经创建的触发器，并且只能由具有相应权限的用户删除。
- **⚠易错提醒**：删除触发器使用 `DROP TRIGGER`，不是 `DELETE TRIGGER`（练习判断10）。

### 17. 要慎重使用触发器

- 触发器是一种功能强大的工具，但在使用时要慎重，因为在每次访问一个表时都可能触发一个触发器，这样会影响系统的性能。

## 三、公式 / 算法 / 定理速查表

| 功能 | SQL 语法 / 关键短语 |
|---|---|
| 定义实体完整性（列级） | `PRIMARY KEY`（直接写在列定义之后） |
| 定义实体完整性（表级） | `PRIMARY KEY (<属性1>, <属性2>, …)`（写在所有列定义之后） |
| 定义参照完整性 | `FOREIGN KEY (<外码列>) REFERENCES <被参照表名>(<主码列>)` |
| 属性上约束条件 | `CHECK <条件>`（写在列定义中） |
| 元组上约束条件 | `CHECK <条件>`（写在 `CREATE TABLE` 中，可涉及多个属性） |
| 完整性约束命名 | `CONSTRAINT <约束名> <约束条件>`（约束条件可为 `NOT NULL`/`UNIQUE`/`PRIMARY KEY`/`FOREIGN KEY`/`CHECK`） |
| 修改完整性限制 | `ALTER TABLE …` |
| 建立域 | `CREATE DOMAIN <域名> <数据类型> [CHECK <约束条件>]` |
| 定义触发器 | `CREATE TRIGGER <触发器名> {BEFORE\|AFTER} <触发事件> ON <表名> REFERENCING NEW\|OLD ROW AS <变量> FOR EACH {ROW\|STATEMENT}; [WHEN <触发条件>] <触发动作体>` |
| 删除触发器 | `DROP TRIGGER <触发器名> ON <表名>;` |

**触发器执行顺序**：
```
① BEFORE 触发器  →  ② 激活触发器的 SQL 语句  →  ③ AFTER 触发器
```
> 同类多个触发器：按创建时间先后执行（或按名称字母排序）。

## 四、重点对比与易混辨析

### 1. 完整性 vs 安全性

| 对比项 | 完整性 | 安全性 |
|---|---|---|
| 目的 | 防止数据库中存在不符合语义的数据（不正确的数据） | 保护数据库防止恶意破坏和非法存取 |
| 防范对象 | 不合语义的、不正确的数据 | 非法用户和非法操作 |

### 2. 三类完整性对比

| 对比项 | 实体完整性 | 参照完整性 | 用户自定义完整性 |
|---|---|---|---|
| 定义对象 | 主码 | 外码 | 具体应用的语义要求 |
| SQL 关键字 | `PRIMARY KEY` | `FOREIGN KEY` … `REFERENCES` | `CHECK`、`NOT NULL`、`UNIQUE` 等 |
| 检查内容 | 主码值唯一且非空 | 被参照表与参照表的相容性 | 属性/元组是否满足自定义条件 |
| 违约处理 | 拒绝插入或修改 | 拒绝执行或级联执行 | 拒绝执行 |

### 3. 属性约束 vs 元组约束

| 对比项 | 属性上的约束条件 | 元组上的约束条件 |
|---|---|---|
| 定义方式 | `CHECK` 短语（写在列定义中） | `CHECK` 短语（写在 `CREATE TABLE` 中） |
| 约束范围 | 单个属性的取值 | 可以设置**不同属性之间**取值的相互约束 |
| 检查时机 | 插入元组或修改属性值时 | 插入元组或修改属性值时 |
| 违约处理 | 操作被拒绝执行 | 操作被拒绝执行 |

### 4. 列级约束 vs 表级约束（实体完整性定义）

| 对比项 | 列级约束条件 | 表级约束条件 |
|---|---|---|
| 适用：单属性码 | ✅ 可以 | ✅ 可以 |
| 适用：多属性码 | ❌ 不可以 | ✅ 唯一方法 |

## 五、高频考点与易错提醒

1. ⭐ **完整性 = 正确性 + 相容性**是高频考点（练习单选1，判断1）。注意区分：正确性=符合现实世界语义；相容性=同一对象在不同表中数据符合逻辑（练习判断2）。
2. ⭐ **完整性 vs 安全性的防范对象**是高频对比考点：完整性防"不合语义的、不正确的数据"；安全性防"非法用户和非法操作"（练习单选2，多选1，多选2）。
3. ⭐ **DBMS 完整性功能三要素**：定义机制、检查方法、违约处理。注意"预防非法用户侵入"属于安全性而非完整性（练习多选3）。
4. ⭐ **多属性码只能用表级约束**定义，单属性码既可列级也可表级（练习单选4）。
5. ⚠ **实体完整性检查**：主码值必须唯一且非空，任一为空即拒绝。注意是针对**主码**而非外码（练习单选5，单选6，单选8，判断3）。
6. ⚠ **表只能有一个主码，但可以有任意个 `UNIQUE` 约束**（练习判断4）。
7. ⭐ **参照完整性通过主键约束和外键约束实现**（练习多选4）。`FOREIGN KEY` 用来实现参照完整性（练习单选11）。
8. ⚠ **级联操作只有级联更新和级联删除**，没有级联插入、级联查询（练习多选5）。
9. ⚠ **`FOREIGN KEY` 至少涉及两个列**，不能作为单一列上的列约束使用（练习多选6）。
10. ⭐ **定义属性约束的短语**：`NOT NULL`、`UNIQUE`、`CHECK`；`HAVING` 不能用于定义属性约束（练习多选7）。
11. ⭐ **元组级约束**可设置不同属性之间的取值相互约束，这是与属性级约束的关键区别（练习单选14）。
12. ⭐ **`CONSTRAINT` 子句**用于对完整性约束条件命名，可灵活增删（练习单选16）。
13. ⚠ **域完整性又称列完整性**，能控制列的数据类型、格式和有效取值范围（练习单选17，判断7）。域的优点：修改域定义即可，不必逐个修改属性（练习判断8）。
14. ⭐ **触发器只能定义在基本表上，不能定义在视图上**（练习单选19）。
15. ⭐ **触发器分为行级触发器和语句级触发器**（练习单选20）。
16. ⚠ **触发器名可以包含模式名，也可以不包含**（练习判断9）。
17. ⚠ **删除触发器用 `DROP TRIGGER`，不是 `DELETE TRIGGER`**（练习判断10）。
18. ⭐ **触发器由数据库服务器自动执行**，非 DBA、非操作系统、非用户启动（练习多选10）。

## 六、一句话记忆要点

1. **完整性** = 正确性（符合现实语义）+ 相容性（不同表中同一对象数据符合逻辑）。
2. **完整性 vs 安全性**：完整性防坏数据，安全性防坏人/坏操作。
3. **DBMS 完整性三功能**：定义约束 → 检查约束 → 违约处理（拒绝或级联）。
4. **实体完整性**：`PRIMARY KEY`，主码唯一且非空；多属性码只能表级定义。
5. **参照完整性**：`FOREIGN KEY … REFERENCES`，级联只有更新和删除。
6. **用户自定义完整性**：`CHECK`/`NOT NULL`/`UNIQUE`，属性级管单列，元组级管多列关系。
7. **`CONSTRAINT`** 给约束起名字，方便增删改。
8. **域**：`CREATE DOMAIN`，改域定义即改所有基于该域的属性约束。
9. **触发器** = 事件-条件-动作规则，只能建在基本表上，分行级/语句级，服务器自动执行。
10. **触发器执行顺序**：BEFORE → SQL 语句 → AFTER；同类按创建先后执行。
11. **删除触发器**：`DROP TRIGGER <名> ON <表>`，不是 `DELETE`。
12. **慎用触发器**：每次访问表都可能触发，影响性能。
