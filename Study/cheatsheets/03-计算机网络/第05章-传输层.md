## 一、本章概览

传输层是 OSI 模型中最重要、最关键的一层，是唯一负责总体数据传输和数据控制的层次。它向上为会话层等高三层提供可靠的传输服务，向下对网络层提供可靠的目的地站点信息，核心职责是提供**端到端**（应用进程之间）的数据交换机制。本章围绕两大传输层协议展开：**TCP**（面向连接、可靠）和 **UDP**（无连接、不可靠），分别讲解其封装格式、连接管理、流量控制、拥塞控制、差错控制、计时器机制及典型应用端口。传输层在网络层"点到点、不可靠"的基础上，为应用层构建"端到端、可选可靠"的通信服务。

## 二、知识点详解

### 1. 传输层概述与功能  ⭐（练习1, 8, 16, 17）

- **定义**：传输层是 OSI 中最重要、最关键的一层，是唯一负责总体的数据传输和数据控制，传输层提供端到端的交换数据的机制，传输层对会话层等高三层提供可靠的传输服务，对网络层提供可靠的目的地站点信息。
- **功能要点**：
  - 网络层协议只提供了**点到点**的连接，而传输层协议提供一种**端到端**的服务，即应用进程之间的通信。
  - 网络层协议提供**不可靠、无连接和尽力投递**的服务，因此，如果对于可靠性要求很高的上层协议，就需要在传输层实现可靠性的保障。
- **两大协议**：
  - **TCP**（Transmission Control Protocol）：可靠的、面向连接的协议。允许网络间两台主机之间无差错的信息传输。TCP 协议还进行流量控制，以避免发送过快而发生拥塞。这一切对用户都是透明的。
  - **UDP**（User Datagram Protocol）：采用无连接的方式传送数据，发送端不关心发送的数据是否到达目标主机，数据是否出错等。收到数据的主机也不会告诉发送方是否收到了数据，它的可靠性由上层协议来保障。
- **选用原则**：
  - TCP 效率较低，但可靠性高，适合于网络链路不好或可靠性要求高的环境。
  - UDP 面向非连接，不可靠，但效率较高，常用于一些实时业务，也用于一些对差错不敏感的应用。

### 2. TCP 的封装格式  ⭐（练习6, 12）

TCP 协议为实现可靠的数据传输而提供了一系列方法和手段，其各种功能的实现依赖于**首部数据结构**。如果不计任选字段，TCP 首部通常是 **20 个字节**。

**TCP 首部数据格式：**

| 行 | 字段 | 位数 |
|----|------|------|
| 第1行 | 源端口地址 / 目的端口地址 | 各16位 |
| 第2行 | 序列号 | 32位 |
| 第3行 | 确认号 | 32位 |
| 第4行 | 首部长度(4位) + 保留(6位) + 控制位(6位) + 窗口大小(16位) | 32位 |
| 第5行 | 校验和(16位) + 紧急指针(16位) | 32位 |
| 第6行 | 选项与填充 | 可变 |
| 第7行 | 数据 | 可变 |

**各字段详解：**

- **源端口号（0~15位，16位）**：发送方进程对应的端口号，由发送方进程产生的随机数，唯一地标识了发送端的一个进程。
- **目标端口号（16~31位，16位）**：对应接收端的进程，接收端根据此端口号确定把数据递送给哪个应用程序的进程。
- **序列号（第2行，32位）**：提供 $0 \sim 2^{32}-1$ 范围内的一个数字。TCP 用这个数字给数据段打上标记，接收端按序列号把数据重新排列，保证数据的正确性。
- **确认序列号（第3行，32位）**：提供 $0 \sim 2^{32}-1$ 范围内的一个数字。确认号是对发送端的确认信息，告诉发送端这个序号之前的数据段都收到了。比如确认号是 $X$，就是表示前 $X-1$ 个数据段都收到了。如果一个数据段一直没有被确认，它将会被要求重发。
- **首部长度（第4行第1部分，4位）**：确定首部数据结构的字节长度。4位最大值 $1111$ 换算成十进制是 15，表示首部长度为 15 行，每行 32 位即 4 字节，所以首部长度最大为 $15 \times 4 = 60$ 字节。
- **保留（第4行第2部分，6位）**：作为今后扩展功能用，现在还没有使用到。
- **控制位（第4行第3部分，6位）**：TCP 的连接、传输和断开都受这 6 个控制位指挥：

  | 控制位 | 含义 |
  |--------|------|
  | URG | 紧急指针有效位，与16位紧急指针配合使用。URG=1 时，TCP 根据16位紧急指针确定紧急数据的最后一个字节的位置，确保紧急数据的即时到达 |
  | ACK | 只有 ACK=1 时确认序列号字段才有效。ACK=0 时确认号无效 |
  | PSH | 标志位为1时要求接收方尽快将数据段送达应用层，加快特殊数据的处理速度 |
  | RST | 值为1时通知重新建立 TCP 连接 |
  | SYN | 同步序号位。TCP 需要建立连接时将此位置为1 |
  | FIN | 发端完成发送任务位，完成数据传输需要断开连接时，提出断开连接的一方将此位置为1 |

- **窗口大小（第4行第4部分，16位）**：说明本地可接收数据段的数目，值的大小是可变的。网络通畅时窗口值变大以加快传输速度，网络不稳定时减小此值可保证可靠传输。TCP 协议中的流量控制机制就是依靠变化窗口的大小实现。
- **校验和（第5行，16位）**：差错控制。发送端计算 TCP 数据段所有字节的校验和，到达目的地再计算一次。若两次校验和一致则说明数据基本正确，否则认为数据已被破坏，接收端将抛弃该数据。
- **紧急指针（第5行，16位）**：和 URG 配合使用，当 URG=1 时有效，说明紧急数据的末尾字节的位置。
- **选项与填充（第6行）**：只有当4位首部长度大于20时才有效。
- **数据（第7行）**：由应用层的数据分段而得到的一部分数据，TCP 协议在传输前会给这部分数据加上一个序号。

**TCP 连接的标识**：一个 TCP 连接不仅需要端口，还需要 IP 地址来确定通信的主机。发送端 IP 地址 + 发送端端口号 = 连接的发送端；目标端 IP 地址 + 接收端端口号 = 连接的接收端。这样就唯一地确定了一个 TCP 连接。

**流式传输**：TCP 被认为是一种流式传输层服务。TCP 发送端从应用程序接收到字符流，并从这个流中提取适当的长度创建数据段，然后将其发送到网络上。TCP 接收端接收数据段，从中提取数据，若没有按序号到达还要对其进行排序，并将其作为字符流交付给接收端应用程序。发送 TCP 使用发送缓冲来存储从发送应用程序截取的数据。

### 3. TCP 的连接与断开  ⭐⚠（练习3）

TCP 是面向连接的服务，在数据通信之前，发送端与接收端要先建立连接。数据发送结束后，双方再断开连接。

#### 3.1 TCP 建立连接——三次握手  ⭐

在计算机上可以通过命令 `netstat -n` 来查看目前存在的连接进程。TCP 建立连接的过程称为**3次握手**。

```
Host A                                    Host B
  ① 发送 SYN，请求建立连接
     (seq=100, ctl=SYN)
                →
                发送 SYN、ACK
                (seq=300, ack=101,
                 ctl=SYN、ACK)           ②
                ←
  ③ 发送 ACK
     (seq=101, ack=301,
      ctl=ACK)
                →
```

- **第1次握手**：主机A发送含有 SYN 标志位的数据段给主机B。告知两点：①主机A希望建立连接请求，主机B应答；②主机A告诉主机B使用哪个序列号作为数据传输时数据段的起始号。
- **第2次握手**：主机B发送带有 ACK 和 SYN 标志位的数据段响应主机A。两个目的：①发送 ACK 通知主机A收到了数据段；②通知A从哪个序列号开始给数据段做标记。
- **第3次握手**：主机A再次发送数据段，确认收到了主机B的数据段，并可以开始传送实际数据。
- **特点**：没有应用层数据；SYN 标志位只有 TCP 建立连接时才被置为1，握手完成后 SYN 标志位被置为0。

#### 3.2 TCP 断开连接——四次挥手  ⭐⚠

TCP 终止一个连接要经过**4次**。这是由 TCP 的**半关闭**（half-close）造成的。

> **半关闭**：TCP 连接是全双工的（数据可在两个方向上同时传递），关闭时每个方向必须单独进行关闭。单方向的关闭称为半关闭。关闭方法是一方完成数据发送任务后，发送一个 FIN 来通告将要终止这个方向连接。

```
Host A                                    Host B
  ① 发送 FIN，请求断开连接
     (seq=101, ack=301,
      ctl=FIN, ACK)
                →
                发送 ACK
                (seq=301, ack=102,
                 ctl=ACK)                ②
                ←
                发送 FIN，请求断开连接
                (seq=301, ack=102,
                 ctl=FIN, ACK)           ③
                ←
  ④ 发送 ACK
     (seq=102, ack=302,
      ctl=ACK)
                →
```

四个步骤：
1. TCP 连接的一端 A 将控制位 FIN 置为1，提出停止 TCP 连接的请求。
2. 对端 B 收到 FIN 后对其做出响应，确认这一方向上的 TCP 连接将关闭。
3. 由 B 端再提出反方向的关闭要求，将 FIN 置为1。
4. 由 A 端对 B 提出的关闭做出应答，双方方向的关闭结束。

> ⚠ 练习3答案表明"只有客户端可以主动提出释放连接的请求"是**错误**描述——服务端也可以主动提出释放连接。正文中使用"TCP连接的一端A"而非"客户端"，说明任一端均可主动发起断开。

### 4. TCP 的流控机制——滑动窗口  ⭐（练习2）

为了解决发送与接收双方数据通信能力存在差异而带来的数据丢失问题，TCP 引入了流量控制机制。TCP 使用**滑动窗口**来实现流量控制。

**工作原理**：
- 在 TCP 建立连接时，双方进行窗口大小的协商，单位是字节。
- 接收方根据缓冲区空间大小宣告窗口大小。
- 发送方根据接收方宣告的窗口大小来发送数据。
- 如果接收方缓冲区大小为0，向发送方宣告窗口大小为0，发送方停止发送，等待接收方宣告一个非0的窗口时再发送数据。

**示例流程**（以缓冲区大小为3为例）：
1. 主机B缓冲区大小为3，B向A发送确认时宣告窗口大小为3。
2. 主机A发送3个数据段，主机B接收后缓冲区由空变为满。
3. 应用程序读取1个数据段，缓冲区清空大小为1的空间。
4. 主机B向A确认收到的数据，并根据缓冲区空间宣告窗口为1。
5. 主机A根据B宣告的窗口大小来发送数据。

### 5. TCP 的拥塞控制  ⭐（练习2）

接收方可以根据其缓冲区大小来指定窗口大小，发送方遵守此限制，接收端不会发生缓冲区溢出的问题。但可能由于中间传输网络的带宽较小，导致拥塞的发生。

**拥塞窗口（cwnd）概念**：
- TCP 引入拥塞窗口（cwnd）的概念。
- **实际发送数据的窗口 = 发送方和接收方协商的窗口与拥塞窗口中的最小值**，即：

$$\text{发送窗口} = \min(\text{接收方允许的窗口}, \ \text{拥塞窗口 cwnd})$$

- 发送方要维护两个窗口：①接收方允许的窗口；②拥塞窗口（cwnd）。

**示例**：主机A与B协商窗口大小为3，A发送3个数据段，B只接收到1个数据段，发送确认号为102的确认。在重传计时器超时前，A没有收到来自B的确认，A认为数据段102和103丢失，A根据接收到的确认将拥塞窗口 cwnd 减小。

### 6. TCP 的差错控制  ⚠（练习7）

TCP 提供差错控制来保证可靠性，包括：检测受损数据段、丢失的数据段、失序的数据段和重复的数据段，以及检测出差错后的纠错机制。

**差错检验的3种方式**：

| 方式 | 说明 |
|------|------|
| 校验和 | 每一个数据段都包含校验和字段，用来检测受损数据段。若数据段受到损伤，由目的TCP将其丢弃 |
| 确认 | TCP使用确认的方式证实收到了某些数据段，它们已经无损伤地到达了目的TCP |
| 超时 | 若一个报文在超时前未被确认，则被认为是受到损伤或已丢失 |

**5种差错情况的处理**：

1. **受损数据段**：到达目标站点被丢弃，目标站点不认为自己已收到该受损数据段，故而会请求重新发送。
2. **丢失的数据段**：和受损数据段情况完全一样，只不过受损数据段是被目的站丢弃的，而丢失的数据段是被中间的节点丢弃的。TCP 并不认为收到过这些数据段。
3. **重复的数据段**：可能由发送端TCP产生（超时截止期到了而确认还没收到，发送端会重发，重发后的数据段和先前的数据段可能都到达了）。目的TCP只要丢弃这个重复的数据段即可。
4. **失序的数据段**：TCP 使用IP的服务，IP是不可靠的网络层协议，每个IP数据包是独立实体，可能沿不同路径到达导致失序。处理方法：**对失序的数据段不确认，直到收到所有它以前的数据段为止**。若确认晚了，源TCP的失序数据段的计时器会到期而重新发送该数据段。目标TCP会丢弃重复的数据段。
5. **丢失的确认**：在TCP确认机制中，丢失的确认甚至不会被源TCP发现。TCP 使用**累计确认系统**，每一个确认证实由确认号指明的字节之前的所有字节都已经收到了。例如，目标站点发送了两个ACK确认号分别是1601和1801，1801的确认号证实了字节1801以前的字节都收到了。如果1601这个确认号丢失了，而1801的确认号收到了，则系统会明白1801之前的字节流都收到了，1601的确认号已变得没有意义了。

### 7. TCP 的计时器  ⚠

TCP 协议设计了 **4种计时器**：

#### (1) 重传计时器
- **作用**：控制丢失的或丢弃的数据段。
- **机制**：当TCP发送数据段时，创建该特定数据段的重传计时器。
  - 若在计时器截止时间之前收到了对此数据段的确认，则撤销此计时器。
  - 若在收到确认之前计时器截止到期，则重传此数据段，并将计时器复位。
- **时间值**：设一次数据段传输的往返时间为 RTT，则：

$$\text{重传时间} = 2 \times \text{RTT}$$

#### (2) 坚持计时器
- **作用**：解决窗口大小为零时的死锁问题。
- **背景**：在TCP中对确认是不需要确认的。若接收端TCP宣布窗口大小为零，发送端TCP停止传送。若接收端的确认丢失，双方进入死锁等待。
- **机制**：当发送端收到窗口大小为零的确认时，启动坚持计时器。期限到时，发送端发送一个**探测数据段**（只有一个字节的数据，序号永远不需要确认），提醒接收端重传确认。
- **时间值**：初始设置为重传时间的数值。若没有收到响应，发送另一个探测数据段，并将坚持计时器的值**加倍和复位**，直到增大到极限值（通常是**60秒**）。此后每隔60秒发送一个探测数据段，直到窗口重新打开。

#### (3) 保活计时器
- **作用**：防止两个TCP之间的连接长时间空闲。
- **机制**：每当服务器收到客户的信息就将保活计时器复位。超时通常设为**2小时**。若服务器过了2小时没有收到客户的信息，发送探测数据段。若发送**10个探测数据段**（每个相隔**75秒**）还没有响应，就假定客户出了故障，中止该连接。

#### (4) 时间等待计时器
- **作用**：保证TCP断开连接的可靠性。
- **背景**：如果A发送完最后一个ACK后就立即关闭连接，而此时这个ACK数据段丢失了，B无法判断是FIN丢失还是ACK丢失，B会重传FIN数据段，而A已经关闭了连接，B永远也无法收到A的ACK了。
- **机制**：A在发送了最后一个ACK报文后，并不立即关闭连接，而是经过一个时间等待计时器的时间再关闭。这个时间可以保证A能收到重复的FIN数据段。

### 8. TCP 的应用与常用端口  ⭐（练习12）

TCP 主要用在对于数据传输可靠性要求高的环境，如远程登录程序 TELNET、网页浏览（HTTP）等。

**表 6-1 TCP 端口及应用：**

| 端口 | 协议 | 说明 |
|------|------|------|

### 9. UDP 的封装  ⭐⚠（练习4, 6）

UDP 是一个简单的面向数据包的传输层协议，进程的每个输出操作都恰好产生一个UDP数据段，并组装成一份待发送的IP数据包。UDP 不提供可靠性。

**UDP 首部各字段（共8字节）：**

| 位数 | 字段 |
|------|------|
| 0~15 | 16位源端口号 |
| 16~31 | 16位目标端口号 |
| 0~15（第2行） | 16位UDP长度 |
| 16~31（第2行） | 16位UDP校验和 |
| 第3行 | 数据 |

- **源端口号（16位）**：标识数据发送端的进程。
- **目标端口号（16位）**：标识数据接收端的进程。
- **UDP长度（16位）**：指出UDP数据段中所包含数据的大小，通过这个值可以准确地计算出数据的结束位置。
- **UDP校验和（16位）**：用来完成对UDP数据的差错检验，这是UDP协议提供的**唯一可靠机制**。UDP的校验和是**可选的**。
- **数据**：来自应用程序，大小可由16位的UDP长度减去UDP首部长度后得到。

> ⚠ 练习4答案B表明"UDP长度字段包括伪首部的长度"是**错误**描述；练习6答案D表明UDP数据报首部**不包含**"UDP数据报首部长度"字段（UDP首部长度固定为8字节，无需额外字段指示）。

### 10. UDP 的应用与常用端口  ⭐（练习5, 13, 18）

UDP 协议能高速地传输数据，在某些方面有着TCP协议不可比拟的优势。对于可靠性要求高的数据传输选择TCP方式；对于传输率要求高的数据传输使用UDP方式。

**UDP 常见端口：**

| 端口 | 协议 | 说明 |
|------|------|------|
| 69 | TFTP | 简单文件传输协议 |
| 53 | DNS | 域名服务 |
| 123 | NTP | 网络时间协议 |
| 111 | RPC | 远程过程调用 |

**DNS 的双协议支持**：DNS服务器支持TCP和UDP两种协议的查询方式，而且端口都是53。大多数的查询都是UDP查询。一般需要TCP查询的两种情况：
- 当查询数据较大以至于产生了数据分段，这时需要利用TCP的分片能力来进行数据传输。
- 当主服务器和辅服务器之间进行数据同步通信的时候。

**UDP 的局限性**：
- UDP 没有流控机制，当来到的报文太多时，接收端可能会溢出。
- 除校验和外，UDP也没有差错控制机制，发送端并不知道数据是丢失了还是重复交付了。当接收端使用校验和检测出差错时，就会悄悄地将此用户数据丢掉。
- 缺少流控制和差错控制就表示使用UDP的进程必须要提供这些机制。例如TFTP协议提供分块传输、分块确认的机制，保证数据传输的可靠性。

## 四、重点对比与易混辨析

### TCP vs UDP 对比

| 特征 | TCP | UDP |
|------|-----|-----|
| 连接方式 | 面向连接 | 无连接 |
| 可靠性 | 可靠（无差错传输） | 不可靠（可靠性由上层协议保障） |
| 流量控制 | 有（滑动窗口机制） | 无 |
| 拥塞控制 | 有（拥塞窗口 cwnd） | 无 |
| 差错控制 | 校验和 + 确认 + 超时 | 仅校验和（可选） |
| 首部大小 | 通常 20 字节（最大 60 字节） | 8 字节（固定） |
| 传输方式 | 流式传输（字节流） | 面向数据包 |
| 传输效率 | 较低 | 较高 |
| 适用场景 | 可靠性要求高的环境（链路不好等） | 实时业务、对差错不敏感的应用、简短数据传输 |
| 典型应用 | TELNET, HTTP, FTP, SMTP, DNS(TCP模式) | TFTP, DNS(UDP模式), NTP, RPC |
| 连接管理 | 三次握手建立、四次挥手断开 | 无连接管理 |
| 缓冲机制 | 发送TCP使用发送缓冲 | [原文未述] |

### TCP 四种计时器对比

| 计时器 | 作用 | 关键参数 |
|--------|------|----------|
| 重传计时器 | 控制丢失/丢弃的数据段 | 重传时间 $= 2 \times \text{RTT}$ |
| 坚持计时器 | 解决窗口为零时的死锁 | 初始=重传时间值，加倍至极限60秒，发送1字节探测数据段 |
| 保活计时器 | 防止连接长时间空闲 | 超时2小时；10个探测段，每75秒一个 |
| 时间等待计时器 | 保证断开连接可靠性 | A发最后ACK后等待，确保能收到重复FIN |

### TCP 差错控制的5种情况

| 差错类型 | 原因 | 处理方式 |
|----------|------|----------|
| 受损数据段 | 传输中数据被破坏 | 目的站丢弃，不认为收到，请求重发 |
| 丢失的数据段 | 被中间节点丢弃 | 与受损相同，TCP不认为收到过 |
| 重复的数据段 | 超时重发导致重复到达 | 目的TCP丢弃重复数据段 |
| 失序的数据段 | IP数据包走不同路径 | 对失序数据段不确认，直到收到所有以前的数据段 |
| 丢失的确认 | 确认数据段丢失 | 累计确认系统，后续确认可覆盖前一个 |

## 五、高频考点与易错提醒

1. ⭐ **传输层的核心定位**：传输层提供端到端服务（应用进程之间通信），网络层只提供点到点服务；传输层处理可靠性、流量控制和错误校正（练习1, 8, 16, 17）。
2. ⭐ **TCP 发送窗口大小**取决于接收方允许的窗口和拥塞窗口中的**最小值**（练习2，答案C）。
3. ⚠ **TCP 断开连接**：任一端均可主动提出释放连接请求，并非只有客户端可以（练习3，答案C指出"只有客户端可以主动提出"是错误描述）。
4. ⚠ **UDP 长度字段**：UDP长度字段是UDP数据报的长度，**不包括伪首部的长度**（练习4，答案B指出"包括伪首部"是错误描述）。
5. ⚠ **UDP 首部不包含"首部长度"字段**：UDP首部固定8字节，无需首部长度字段（练习6，答案D）。
6. ⭐ **使用UDP的网络应用，数据传输可靠性由应用层负责**（练习5，答案B）。
7. ⚠ **接收端收到有差错的UDP用户数据时直接丢弃**，不请求重传，不做差错校正（练习7，答案A）。
8. ⭐ **面向连接的服务保证可靠和顺序的交付**；无连接的服务只能提供不可靠的服务（练习9, 10）。
9. ⭐ **熟知端口号范围为 0~1023**（练习11，答案D）。
10. ⚠ **FTP 端口为 21**，不是24（练习12，答案D指出"FTP:24"是错误的）。
11. ⭐ **TFTP 基于UDP**（练习13，答案C）。
12. ⭐ **TCP和UDP都是传输层协议**，这是二者相似之处（练习17，答案C）。
13. ⚠ **TCP和UDP分别拥有自己的端口号，互不干扰，可以共存于同一台主机**（练习19，答案A）。
14. ⚠ **TCP套接字由四元组标识**（源IP地址、源端口号、目的IP地址、目的端口号），是考点方向（例题1）。
15. ⚠ **TCP快速重传**：收到3个重复ACK时触发快速重传，是考点方向（例题2，反推）。
16. ⚠ **UDP套接字由目的IP地址和目的端口号二元组标识**，是考点方向（例题3，反推）。
17. ⚠ **停止等待协议**在源主机和目的主机距离远且传输速率高时效率很低，是考点方向（例题4，反推）。

## 六、一句话记忆要点

1. **传输层**：OSI 最关键一层，端到端服务，网络层点到点不可靠 → 传输层补可靠。
2. **TCP vs UDP**：TCP 面向连接可靠效率低（20B首部）；UDP 无连接不可靠效率高（8B首部）。
3. **TCP 首部**：20字节常规，最大60字节（4位首部长度×4）；6个控制位 URG/ACK/PSH/RST/SYN/FIN 各司其职。
4. **三次握手**：SYN → SYN+ACK → ACK，无应用层数据，SYN 仅建连时置1。
5. **四次挥手**：FIN → ACK → FIN → ACK，全双工需半关闭，任一端可主动发起。
6. **滑动窗口**：接收方宣告窗口大小，发送方据此控制发送量，窗口为0则停止发送。
7. **拥塞控制**：发送窗口 = min(接收方窗口, cwnd)，丢包时减小 cwnd。
8. **差错控制**：校验和 + 确认 + 超时；累计确认使丢失的确认可被后续确认覆盖。
9. **四种计时器**：重传（2×RTT）、坚持（窗口为零防死锁，极限60秒）、保活（2小时+10×75秒）、时间等待（防最后ACK丢失）。
10. **UDP 校验和**：唯一可靠机制，可选；UDP 无流控、无差错控制，进程自行保障。
11. **DNS 双协议**：端口53，大多数UDP查询；大数据分段或主辅同步用TCP。
12. **常用端口**：TCP—FTP(21)/Telnet(23)/SMTP(25)/DNS(53)/HTTP(80)；UDP—TFTP(69)/DNS(53)/NTP(123)/RPC(111)。
