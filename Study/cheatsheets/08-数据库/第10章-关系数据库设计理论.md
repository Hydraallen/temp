## 一、数据库设计基础

**数据库设计**：广义地讲，是数据库及其应用系统的设计；狭义地讲，是设计数据库本身（设计各级模式并建立数据库）。对于给定的应用环境，构造（设计）优化的数据库逻辑模式和物理结构，使之能够有效地存储和管理数据。

**数据库设计的目标**：为用户和各种应用系统提供一个信息基础设施和高效的运行环境（包括存取效率、存储空间的利用、运行管理的效率等）。

**数据库建设的基本规律**：三分技术，七分管理，十二分基础数据。
- > Caveat (第十章-判断-1): 基本规律是“三分技术，七分管理，十二分基础数据”，切勿将“技术”和“管理”的比例颠倒。

**数据库设计方法**：主要包括新奥尔良方法、基于E-R模型的设计方法、3NF的设计方法、面向对象的数据库设计方法、统一建模语言（UML）方法等。

## 二、数据库设计六大步骤 ⭐

数据库设计主要包含以下六个阶段，设计完善的应用系统往往是这些阶段的不断反复：
1. **需求分析阶段**
2. **概念结构设计阶段**
3. **逻辑结构设计阶段**
4. **物理结构设计阶段**
5. **数据库实施阶段**
6. **数据库运行和维护阶段**

## 三、需求分析阶段

**需求分析的任务**：通过详细调查现实世界要处理的对象，充分了解原系统工作概况，明确用户的各种需求并确定新系统功能。新系统必须充分考虑今后可能的扩充和改变。

**常用调查方法**：跟班作业、开调查会、请专人介绍、询问、设计调查表请用户填写、查阅记录。（调查往往需要多种方法结合使用）

**数据字典**：关于数据库中数据的描述（即**元数据**），而不是数据本身。在需求分析阶段建立。

| 组成部分 | 含义定义 |
| :--- | :--- |
| **数据项** | 不可再分的数据单位 |
| **数据结构** | 反映了数据之间的组合关系 |
| **数据流** | 数据结构在系统内传输的路径 |
| **数据存储** | 数据结构停留或保存的地方，也是数据流的来源和去向 |
| **处理过程** | 具体处理逻辑一般用判定表或判定树来描述 |

## 四、概念结构设计 ⭐

**概念结构设计**：将需求分析得到的用户需求抽象为信息结构（即概念模型）的过程，是整个数据库设计的**关键**。

**概念模型的主要特点**：能真实反映现实世界、易于理解（便于和不熟悉计算机的用户交换意见）、易于更改、易于向关系/网状/层次等数据模型转换。

### 1. 实体-联系模型 (E-R模型) ⭐
E-R图提供了表示实体型、属性和联系的方法：
| 图形元素 | 代表含义 | 说明细节 |
| :--- | :--- | :--- |
| **矩形** | 实体型 | 矩形框内写明实体名 |
| **椭圆形** | 属性 | 用无向边与相应的实体型连接 |
| **菱形** | 联系 | 菱形框内写明联系名，用无向边与有关实体型连接，旁标联系类型(1:1, 1:n, m:n) |

### 2. 实体之间的联系
实体型之间的联系分为一对一(1:1)、一对多(1:n)、多对多。两个以上实体型之间及单个实体型内部也可以存在这三种联系。

- > Caveat (第十章-单选-10~12): 解题时需结合实际业务语义分析双向映射基数（如“双向均为一对多”即为多对多），不能盲目推测传递依赖。

### 3. 实体与属性的划分原则
为了简化E-R图，现实世界的事物能作为属性对待的，尽量作为属性。需满足两条准则：
1. 作为属性，不能再具有需要描述的性质（必须是不可分的数据项）。
2. 属性不能与其他实体具有联系（E-R图中所表示的联系只能是实体之间的联系）。

### 4. E-R图的集成
一般分两步：（1）**合并**：解决各分E-R图之间的冲突，生成初步E-R图。（2）**修改和重构**：消除不必要的冗余，生成基本E-R图。

### 5. 冲突的类型 ⚠
各子系统的E-R图之间主要存在三类冲突：
| 冲突类型 | 产生原因及表现 |
| :--- | :--- |
| **属性冲突** | 同一属性在不同局部应用中类型不同（如学号定义为字符型vs整型） |
| **命名冲突** | 同名不同义、同义不同名等 |
| **结构冲突** | [原文未述具体产生原因，仅作分类罗列] |

- > Caveat (第十章-单选-13): 同一属性（如学号）在不同局部应用中数据类型不同，属于**属性冲突**；若为“同名不同义”，则属于**命名冲突**。

## 五、逻辑结构设计 ⭐

**逻辑结构设计的任务**：把概念结构设计阶段的基本E-R图转换为与选用DBMS产品所支持的数据模型相符合的逻辑结构（关系数据库中即E-R图向关系数据模型转换）。

### 1. E-R图向关系模型转换规则 ⭐⚠
实体与联系向关系模式的转换遵循以下规则：
| 原始结构 | 转换规则描述 | 关系模式生成与合并原则 |
| :--- | :--- | :--- |
| **实体** | 1个实体 $\rightarrow$ 1个独立关系模式 | |
| **1:1 联系** | 可转为独立关系模式，也可与任意一端实体合并 | 与任意一端实体对应的关系模式合并 |
| **1:n 联系** | 可转为独立关系模式，也可与n端实体合并 | 合并到n端对应的关系模式最合适 |
| **m:n 联系** | 必须转换为1个独立关系模式 | 不可与实体合并 |
| **多元联系(3个及以上)** | 必须转换为1个独立关系模式 | 不可与实体合并 |
| **相同码的模式** | 具有相同码的关系模式可合并 | 减少冗余关系 |

- > Caveat (第十章-单选-17): “班级与学生”是1:n联系，合并到n端实体（学生端）能减少表数量，最为合适，此时主码即为n端的主码。

**数据模型的优化**：逻辑设计结果并不唯一，通常以**规范化理论**为指导修改、调整数据模型结构，提高系统性能。

**设计用户子模式**：将概念模型转换为全局逻辑模型后，根据局部应用需求设计外模式，注重考虑用户的习惯与方便（如使用别名、定义视图保证安全性、简化系统使用）。

## 六、物理结构设计

**物理结构设计**：为给定的逻辑数据模型选取最适合应用要求的物理结构（存储结构与存取方法），依赖选定DBMS。

**物理结构设计步骤**：
1. 确定数据库的物理结构（主要指存取方法和存储结构）。
2. 对物理结构进行评价（重点是时间和空间效率）。

**关系模式存取方法**：快速存取数据库中数据的技术。常用方法为**索引方法**和**聚簇方法**。

## 七、数据库实施与运行维护

**数据库实施阶段的工作**：
1. **数据的载入**：将各类源数据抽取、分类转换，综合成符合新设计结构的格式输入数据库（费时费力）。
2. **应用程序的编码和调试**：与数据库设计同时进行。

**数据库试运行**：应用程序调试完成、部分数据入库后进行联合调试。包括**功能测试**（执行操作测试功能）和**性能测试**（测量指标分析是否符合目标）。

**数据库运行和维护阶段的主要工作**（由DBA完成）：
1. 数据库转储和恢复
2. 数据库安全性、完整性控制
3. 数据库性能监督、分析和改进
4. 数据库重组织与重构造

## Worked Examples

### 例 1：E-R图向关系模型转换数目计算 *(来自 exam-signals 候选 2)*
*Tags: E-R模型, 逻辑结构设计*
**问题**：E-R 模型转换为关系模型后，关系的数目不可能为（ ）个。（图示含4个实体，2个1:N联系，1个M:N联系）
**解答**：`[PROPOSED — verify]`
1. 实体转关系：4个实体必须转换为 4 个关系模式。
2. 联系转关系：1个 M:N 联系必须转换为 1 个独立关系模式，共 1 个。
3. 1:N联系处理：2个 1:N 联系既可以向 N 端实体合并（转 0 个），也可以各自独立转换（转 2 个），数目范围为 0~2。
4. 总计：4 + 1 + (0~2) = 5~7个。因此，关系的数目不可能为 4 个。
