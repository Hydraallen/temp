## 2. 反复出现的题型模式
- **数据流图符号与数据字典内容匹配**：题干问"正方形/圆角矩形/开口矩形代表什么"或"数据字典不包括什么" → 切记四元素对应关系（正方形=源点/终点，圆角矩形=处理，开口矩形=存储，箭头=数据流），且数据字典只含数据流/数据项/数据存储/处理，**不含外部实体**。
- **白盒测试覆盖标准强度排序**：题干给出若干覆盖标准要求比较或选择 → 直接默写强度顺序：语句覆盖 < 判定覆盖 < 条件覆盖 < 判定条件覆盖 < 条件组合覆盖 < 路径覆盖。
- **集成测试策略与程序类型绑定**：题干提及"自顶向下"或"自底向上"集成 → 自顶向下**需存根程序**（不需驱动），自底向上**需驱动程序**（不需存根）。
- **软件维护四类活动定义辨析**：题干描述具体维护场景（改错/适应环境/扩充功能/预防） → 对应改正性/适应性/完善性/预防性维护；若问占比，完善性维护最大。
- **面向对象三大模型与工具映射**：题干问"描述系统数据结构/控制结构/功能" → 对应对象模型（类图）/动态模型（状态图、顺序图）/功能模型（数据流图、用例图）。
- **估算规模 vs 估算工作量方法辨析**：题干列举功能点/COCOMO2等模型问分类 → 代码行、功能点属于**规模估算**；COCOMO2、静态/动态多变量模型属于**工作量估算**。

## 3. 技巧 / 易错点
- > Caveat (第一章判断6): 软件工程方法学3要素是"方法、工具和**过程**"，题干若写"方法、工具和模型"是错的。
- > Caveat (第四章单选3/判断3): "成功的测试"是指**发现了至今未发现的错误**，而不是"未发现错误"或"证明程序正确"。
- > Caveat (第二章单选7): 数据字典由4类元素组成，**不包括外部实体**（外部实体是数据流图的成分）。
- > Caveat (第二章单选10): 数据流图是**功能模型**工具，不是数据模型工具；类图、ER图、数据字典才是数据模型工具。
- > Caveat (第二章判断7): 需求分析只解决"做什么"的问题，**不考虑"怎么做"**（怎么做是设计阶段）。
- > Caveat (第三章单选8/9): 耦合度最高的是**内容耦合**（如直接访问另一模块内部数据），内聚度最高的是**功能内聚**。
- > Caveat (第六章单选4): 软件项目开发时间最多只能减少到正常时间的**75%**，不是50%或25%。
- > Caveat (第一章单选11/第五章单选1): 软件生命周期中工作量最大、持续时间最长的阶段都是**维护阶段**。
- > Caveat (第四章单选12): 自底向上集成**不需要存根程序**，但**需要驱动程序**；自顶向下正好相反。

## 5. Worked Example 候选（适合放进最终 cheatsheet 末尾）
### 候选 1：第三章单选18 · McCabe环路复杂性计算 `[PROPOSED — verify]`
- 概念 tag：程序复杂程度的定量度量
- 题干（精简）：采用 McCabe 度量法计算下列程序图的环路复杂性为（ ）。已知节点数为9，边数为10。
- 解题骨架（3-8 步）：
  1. 识别 McCabe 环路复杂性公式：`V(G) = E - N + 2`（E为边数，N为节点数）或 `V(G) = P + 1`（P为判定节点数）。
  2. 代入已知条件：E = 10，N = 9。
  3. 计算：V(G) = 10 - 9 + 2 = 3。
  4. 若用判定节点公式，需数出有分支的节点数P，则 V(G) = P + 1。
  5. 得出结论答案为3。

### 候选 2：第四章单选16 · 白盒测试逻辑覆盖强度排序 `[PROPOSED — verify]`
- 概念 tag：白盒测试、逻辑覆盖
- 题干（精简）：一般而言，覆盖强度由低到高分别是？
- 解题骨架（3-8 步）：
  1. 识别白盒测试的6种逻辑覆盖标准。
  2. 按照覆盖粒度从小到大排序：语句覆盖（最弱） < 判定覆盖 < 条件覆盖 < 判定条件覆盖 < 条件组合覆盖 < 路径覆盖（最强）。
  3. 记忆口诀：语判条判组路。
  4. 注意区分"条件覆盖"（每个条件取真/假）与"判定覆盖"（每个判定取真/假）。

### 候选 3：第四章单选11/12 · 集成测试策略与程序依赖 `[PROPOSED — verify]`
- 概念 tag：集成测试、驱动程序、存根程序
- 题干（精简）：自顶向下和自底向上集成测试各需要开发什么辅助程序？
- 解题骨架（3-8 步）：
  1. 理解集成测试方向：自顶向下从主控模块开始往下集成；自底向上从底层模块开始往上集成。
  2. 自顶向下：上层调用的下层模块未开发时，需用**存根程序**代替底层模块接收调用并返回假数据。
  3. 自底向上：下层模块开发好但无上层调用时，需用**驱动程序**作为主程序调用底层模块并传数据。
  4. 结论：自顶向下需存根，自底向上需驱动。

### 候选 4：第五章单选3/4/5 · 软件维护四类活动辨析 `[PROPOSED — verify]`
- 概念 tag：软件维护的定义、改正性/适应性/完善性/预防性维护
- 题干（精简）：为提高系统性能而进行的修改、开发测试不彻底导致的修改、为使软件在改变环境下仍能使用分别属于什么维护？
- 解题骨架（3-8 步）：
  1. 识别四类维护的触发原因：改错（改正性）、环境变化（适应性）、扩充功能/提高性能（完善性）、为未来维护做准备（预防性）。
  2. "提高系统性能"属于用户提出的新要求或改进意见 → 完善性维护。
  3. "测试不彻底遗留错误" → 改正性维护。
  4. "环境改变" → 适应性维护。
  5. 补充考点：完善性维护在实际维护工作中占比最大。
