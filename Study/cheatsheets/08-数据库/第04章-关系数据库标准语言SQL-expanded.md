## 一、SQL基础概念与特点

**结构化查询语言**：是关系数据库的标准语言，也是一个通用的、功能极强的关系数据库语言。其功能不仅仅是查询，而是包括数据库模式创建、数据库数据的插入与修改、数据库安全性完整性定义与控制等一系列功能。*(讲义 p033)*

**SQL的基本概念**：支持SQL的关系数据库管理系统同样支持关系数据库三级模式结构。其中外模式包括若干视图和部分基本表，数据库模式包括若干基本表，内模式包括若干存储文件。基本表和视图一样，都是关系。*(讲义 p033)*
- **基本表**：本身独立存在的表，在关系数据库管理系统中一个关系就对应一个基本表。一个或多个基本表对应一个存储文件，索引也存放在存储文件中。*(讲义 p033)*
- **存储文件**：逻辑结构组成了关系数据库的内模式。物理结构对最终用户是隐蔽的。*(讲义 p033)*
- **视图**：从一个或几个基本表导出的表。它本身不独立存储在数据库中（数据库中只存放视图的定义而不存放数据），是一个虚表。*(讲义 p033)*

**SQL语言的特点** ⭐：*(讲义 p033)*
1. 综合统一：集数据定义语言(DDL)、数据操纵语言(DML)、数据控制语言(DCL)的功能于一体。*(讲义 p033)*
2. 高度非过程化：只要提出“做什么”，无须指明“怎么做”。*(讲义 p033)*
3. 面向集合的操作方式：操作对象和结果可以是元组的集合。*(讲义 p033)*
4. 一种语法，多种方式：既是独立的语言，又是嵌入式语言（SQL语句能够嵌入到高级语言程序中，此高级语言称为宿主语言）。*(讲义 p033)*
5. 语言简洁：完成核心功能只用了9个动词。*(讲义 p033)*

| SQL功能 | 动词 |*(讲义 p033)*
|---|---|
| 数据查询 | SELECT |
| 数据定义 | CREATE，DROP，ALTER |
| 数据操纵 | INSERT，UPDATE，DELETE |
| 数据控制 | GRANT，REVOKE |

> Caveat (题目1): 嵌入高级语言中使用的DML称为宿主型。
> Caveat (题目2): INSERT、DELETE、UPDATE 实现的是数据操纵功能，不是数据定义。
> Caveat (判断题1): SQL既是独立的语言，又是嵌入式语言，能够嵌入到高级语言程序中。

## 二、数据定义与基本表操作

**数据定义的对象**：包括模式定义、表定义、视图和索引的定义。*(讲义 p034)*
一个关系数据库管理系统的实例中可以建立多个数据库，一个数据库中可以建立多个模式，一个模式下通常包括多个表、视图和索引等数据库对象。*(讲义 p034)*

**SQL的数据定义语句** ⭐：*(讲义 p034)*

| 操作对象 | 创建 | 删除 | 修改 |*(讲义 p034)*
|---|---|---|---|
| 模式 | CREATE SCHEMA | DROP SCHEMA | [原文未述] |
| 表 | CREATE TABLE | DROP TABLE | ALTER TABLE |
| 视图 | CREATE VIEW | DROP VIEW | [原文未述] |
| 索引 | CREATE INDEX | DROP INDEX | ALTER INDEX |

> Caveat (多选题1): ROLLBACK 属于恢复操作，GRANT 属于数据控制语句，两者都不是数据定义语句。
> Caveat (判断题2): 数据定义功能包括定义表、视图和索引；对数据的增、删、改、查属于数据操纵功能。

**模式的定义与删除**：*(讲义 p034)*
- 定义模式：`CREATE SCHEMA <模式名> AUTHORIZATION <用户名>;`
- 删除模式：`DROP SCHEMA <模式名> <CASCADE | RESTRICT>;`

**基本表的相关操作** ⭐：*(讲义 p034)*

1. 定义基本表：*(讲义 p034)*
```sql
CREATE TABLE <表名>(
  <列名> <数据类型> [<列级完整性约束条件>]
  [, <列名> <数据类型> [<列级完整性约束条件>]] ...
  [, <表级完整性约束条件>]
);
```

2. 修改基本表：*(讲义 p034)*
```sql
ALTER TABLE <表名>
[ ADD [COLUMN] <新列名> <数据类型> [完整性约束] ]
[ ADD <表级完整性约束>]
[ DROP [ COLUMN ] <列名> [CASCADE| RESTRICT] ]
[ DROP CONSTRAINT <完整性约束名> [ RESTRICT | CASCADE ] ]
[ ALTER COLUMN <列名> <数据类型> ];
```

3. 删除基本表：*(讲义 p034)*
```sql
DROP TABLE <表名> [RESTRICT|CASCADE];
```

> Caveat (题目3): 为已存在的表添加新列应使用 `ALTER TABLE <表名> ADD <新列名> <数据类型>`。
> Caveat (题目5): 分配给列中的值的存储空间量是根据数据类型来定义的。
> Caveat (题目6): 学号由数字组成，但本质上是字符串，为避免溢出或丢失前导零，应使用 CHAR 而非 INT。
> Caveat (多选题2): 定义一个列至少需要数据类型和名称，可以没有索引和键。

## 三、索引机制与数据字典

**索引的概念** ⚠：常见索引有顺序文件上的索引、B+树索引、散列索引、位图索引等。*(讲义 p034)*
- 虽然能够加速数据库查询，但需要占用一定的存储空间。*(讲义 p034)*
- 当基本表更新时，索引要进行相应的维护，会增加数据库的负担，因此要有选择地创建。*(讲义 p034)*
- 关系数据库管理系统在执行查询时会自动选择合适的索引作为存取路径，用户不必也不能显式地选择索引。*(讲义 p034)*
- 索引是关系数据库管理系统的内部实现技术，属于内模式的范畴。*(讲义 p034)*

**索引的建立与删除**：*(讲义 p034)*
```sql
-- 建立索引
CREATE [UNIQUE] [CLUSTER] INDEX <索引名>
ON <表名>(<列名> [<次序>] [, <列名> [<次序>]...]);

-- 修改索引
ALTER INDEX <旧索引名> RENAME TO <新索引名>;

-- 删除索引
DROP INDEX <索引名>;
```

**数据字典**：是关系数据库管理系统内部的一组系统表，记录了数据库中所有的定义信息（包括关系模式定义、视图定义、索引定义、完整性约束定义、操作权限、统计信息等）。执行SQL数据定义语句时实际上就是在更新数据字典表中的相应信息。*(讲义 p035)*

> Caveat (判断题3): 索引虽能加速查询，但过多建立会增加维护负担和占用空间，并非越多越好。

## 四、数据查询机制与单表查询

**数据查询一般格式** ⭐：数据查询是数据库的核心操作。SQL提供了SELECT语句进行数据查询。*(讲义 p035)*
```sql
SELECT [ALL|DISTINCT] <目标列表表达式> [, <目标列表表达式>] ...
FROM <表名或视图名> [, <表名或视图名>] ... | (SELECT 语句) [AS] <别名>
[WHERE <条件表达式>]
[GROUP BY <列名1> [HAVING <条件表达式>]]
[ORDER BY <列名2> [ASC|DESC]];
```

> Caveat (题目7): SELECT 语句执行的结果是表（或元组的集合）。
> Caveat (题目8): SQL 语句 `SELECT A FROM R WHERE C='男'` 等价于关系代数表达式 $\Pi_A(\sigma_{C='男'}(R))$（WHERE对应选择，SELECT对应投影）。

**单表查询** ⭐：指仅涉及一个表的查询。分为选择表中的若干列（查询指定列、全部列）和选择若干元组（消除重复行、条件查询）。*(讲义 p035)*

**常用查询条件谓词** ⭐：*(讲义 p035)*

| 查询条件 | 谓词 |*(讲义 p035)*
|---|---|
| 比较 | =, >, <, >=, <=, !=, <>, |>, |<; NOT+上述比较运算符 |
| 确定范围 | BETWEEN AND, NOT BETWEEN AND |
| 确定集合 | IN, NOT IN |
| 字符匹配 | LIKE, NOT LIKE |
| 空值 | IS NULL, IS NOT NULL |
| 多重条件（逻辑运算） | AND, OR, NOT |

**字符匹配通配符** ⚠：谓词LIKE可用来进行字符串的匹配。*(讲义 p035)*
- `%`（百分号）：代表任意长度的字符串。*(讲义 p035)*
- `_`（下横线）：代表任意单个字符。*(讲义 p035)*

> Caveat (题目9): `BETWEEN...AND...` 是包含边界值的。
> Caveat (题目10): 消除取值重复的行使用 `DISTINCT` 保留字。
> Caveat (题目13): 字符串匹配的操作符是 `LIKE`。
> Caveat (题目14): `DB_` 代表长度为 3 且以“DB”开头的字符串。
> Caveat (题目15): 查询空值必须使用 `IS NULL`，不能用等号（`=`）。
> Caveat (多选题4): AND 是逻辑运算符，IN 是确定集合的谓词，两者不属于比较运算符。

**ORDER BY 子句**：对查询结果按照一个或多个属性列的升序（ASC）或降序（DESC）排列，默认值为升序。*(讲义 p035)*

**聚集函数** ⚠：当聚集函数遇到空值时，除 `COUNT(*)` 外，都跳过空值而只处理非空值。*(讲义 p035)*
- `WHERE` 子句中**不能**用聚集函数作为条件表达式。聚集函数只能用于 SELECT 子句和 GROUP BY 中的 HAVING 子句。*(讲义 p035)*

| 函数 | 功能 |*(讲义 p035)*
|---|---|
| COUNT(*) | 统计元组个数 |
| COUNT([DISTINCT\|ALL] <列名>) | 统计一列中值的个数 |
| SUM([DISTINCT\|ALL] <列名>) | 计算一列值的总和（必须是数值型） |
| AVG([DISTINCT\|ALL] <列名>) | 计算一列值的平均值（必须是数值型） |
| MAX([DISTINCT\|ALL] <列名>) | 求一列值中的最大值 |
| MIN([DISTINCT\|ALL] <列名>) | 求一列值中的最小值 |

**GROUP BY 子句与 HAVING 短语** ⚠：将查询结果按某一列或多列的值分组，值相等的为一组。目的是为了细化聚集函数的作用对象。*(讲义 p035)*
- 如果分组后还要求按一定的条件对这些组进行筛选，则可以使用 HAVING 短语指定筛选条件。*(讲义 p035)*

> Caveat (题目17): 分组检索时，为了去掉不满足条件的分组，应该在 GROUP BY 后面使用 HAVING 子句，而不是 WHERE 子句。
> Caveat (题目18): 查询选修了 3 门以上课程的学生的学号，应使用 `GROUP BY S# HAVING COUNT(*)>3`。

## 五、连接查询

**连接查询** ⭐：若一个查询同时涉及两个以上的表，则称之为连接查询。包括等值连接、自然连接、非等值连接、自身连接、外连接和复合条件连接等。*(讲义 p035)*

**连接条件的一般格式**：连接条件中的列名称称为连接字段。*(讲义 p036)*
- 格式一：`[<表名1>.]<列名1> <比较运算符> [<表名2>.]<列名2>`*(讲义 p036)*
- 格式二：`[<表名1>.]<列名1> BETWEEN [<表名2>.]<列名2> AND [<表名2>.]<列名3>`*(讲义 p036)*
- 当连接运算符为 `=` 时，称为等值连接。使用其他运算符称为非等值连接。*(讲义 p036)*

**自身连接**：一个表与其自己进行连接。需要给表起别名以示区别，因为所有属性名都是同名属性，必须使用“别名”。*(讲义 p036)*

**外连接** ⚠：*(讲义 p036)*
- 外连接：OUTER JOIN
- 左外连接：LEFT OUTER JOIN（保留左边表的全部记录）*(讲义 p036)*
- 右外连接：RIGHT OUTER JOIN（保留右边表的全部记录）*(讲义 p036)*

**多表连接**：两个以上的表进行连接。*(讲义 p036)*

> Caveat (题目19): 根据关系代数表达式转 SQL 时，需将对应字段用等值连接（如 `WHERE R.A=S.A`）。
> Caveat (题目20): 查找学生基本信息及选课平均成绩，需使用学生表 S 和选课表 SC。
> Caveat (题目21): 左外连接将保留左表（如 R）的全部记录，以及右表（如 S）匹配上的部分记录。
> Caveat (多选题7): LEFT JOIN 保留左表全部记录，因此右表字段可能出现空值，左表字段不会出现空值。

## 六、嵌套查询与集合查询

**嵌套查询** ⭐：将一个查询块（一个 SELECT-FROM-WHERE 语句）嵌套在另一个查询块的 WHERE 子句或 HAVING 短语的条件中。*(讲义 p036)*
- **父查询与子查询**：上层查询块称为父查询，下层查询块称为子查询。子查询中还可以嵌套其他子查询。*(讲义 p036)*
- 子查询的 SELECT 语句中**不能**使用 ORDER BY 子句，ORDER BY 子句只能对最终查询结果排序。*(讲义 p036)*

**子查询分类** ⚠：*(讲义 p036)*
- **不相关子查询**：子查询的查询条件不依赖于父查询。*(讲义 p036)*
- **相关子查询**：子查询的查询条件依赖于父查询。*(讲义 p036)*

**常用的子查询类型** ⭐：带有 IN 谓词、比较运算符、ANY/ALL 谓词、EXISTS 谓词的子查询。*(讲义 p036)*

**集合查询**：多个 SELECT 语句的结果可进行集合操作。*(讲义 p036)*
- 主要包括：并操作 UNION、交操作 INTERSECT、差操作 EXCEPT。*(讲义 p036)*

**基于派生表的查询**：子查询出现在 FROM 子句中，生成的临时派生表成为主查询的查询对象。*(讲义 p036)*

> Caveat (题目22): 子查询是嵌入到另一个查询语句之中的查询语句。
> Caveat (判断题6): 相关子查询的查询条件**依赖于**父查询。
> Caveat (判断题7): 选择、投影、连接不属于 SQL 的集合操作，集合操作主要指并、交、差。
> Caveat (多选题8): 使用 UNION 合并两个 SELECT 查询结果时，两个 SELECT 语句必须输出同样的列数，且对应项的数据类型必须相同。

## 七、数据更新（增删改）

**数据更新的三种操作** ⭐：插入数据 INSERT、修改数据 UPDATE、删除数据 DELETE。*(讲义 p036)*

**插入数据 INSERT** ⚠：*(讲义 p036)*
```sql
-- 插入元组
INSERT INTO <表名> [(<属性列1> [, <属性列2>]...)] VALUES (<常量1> [, <常量2>]...);

-- 插入子查询结果
INSERT INTO <表名> [(<属性列1> [, <属性列2>]...)] 子查询;
```

**修改数据 UPDATE** ⚠：*(讲义 p037)*
```sql
UPDATE <表名> SET <列名>=<表达式> [, <列名>=<表达式>]... [WHERE <条件>];
```
- 功能：修改指定表中满足 WHERE 子句条件的元组。如果省略 WHERE 子句，则表示要修改表中的所有元组。*(讲义 p037)*

**删除数据 DELETE** ⚠：*(讲义 p037)*
```sql
DELETE FROM <表名> [WHERE <条件>];
```
- 功能：从指定表中删除满足 WHERE 子句条件的所有元组。*(讲义 p037)*
- 如果省略 WHERE 子句则表示删除表中全部元组，但**表的定义仍在字典中**。*(讲义 p037)*

> Caveat (题目24): 更新表中数据的命令动词是 UPDATE。
> Caveat (题目25): 插入数据时，数值型变量不应该加单引号。
> Caveat (题目26): DELETE 权限允许删除数据，与 DROP（删除关系/表结构）有本质区别。
> Caveat (题目27): 修改数据的标准语法是 `UPDATE 表名 SET 列名=表达式 WHERE 条件`。
> Caveat (题目28): 删除数据库使用 `DROP DATABASE`，DELETE 仅用于操作数据。
> Caveat (多选题9): `BETWEEN 55 AND 60` 等价于 `>=55 AND <=60`，不能使用 OR。

## 八、空值的处理

**空值的处理** ⚠：所谓空值就是“不知道”或“不存在”或“无意义”的值。含有不确定性。*(讲义 p037)*
- 一般有以下几种情况：该属性应该有一个值，但目前不知道具体值；该属性不应该有值；由于某种原因不便于填写。*(讲义 p037)*
- 相关操作包括：空值的产生、空值的判断（IS NULL）、空值的约束条件、空值的算术运算。*(讲义 p037)*

## 九、视图机制

**视图概念** ⭐：视图是从一个或几个基本表（或视图）导出的表。它与基本表不同，是一个虚表。*(讲义 p037)*
- 数据库中只存放视图的定义，而不存放视图对应的数据，数据仍存放在原来的基本表中。*(讲义 p037)*
- 基本表中的数据发生变化，从视图中查询出的数据也就随之改变。*(讲义 p037)*

**建立视图** ⭐：*(讲义 p037)*
```sql
CREATE VIEW <视图名> [(<列名> [, <列名>]...)] AS <子查询> [WITH CHECK OPTION];
```
- WITH CHECK OPTION：表示对视图进行 UPDATE、INSERT 和 DELETE 操作时要保证更新、插入或删除的行满足视图定义中的谓词条件（即子查询中的条件表达式）。*(讲义 p037)*
- 组成视图的属性列名或者全部省略或者全部指定，没有第三种选择。*(讲义 p037)*
- 执行 CREATE VIEW 语句的结果只是把视图的定义存入数据字典，并不执行其中的 SELECT 语句。只是在对视图查询时，按视图的定义从基本表中将数据查出。*(讲义 p037)*

**行列子集视图**：若一个视图是从单个基本表导出的，并且只是去掉了基本表的某些行和某些列，但保留了主码，称为行列子集视图。*(讲义 p037)*

**删除视图**：*(讲义 p037)*
```sql
DROP VIEW <视图名> [CASCADE];
```
- 使用 CASCADE 级联删除语句可以把该视图和由它导出的所有视图一起删除。*(讲义 p037)*
- 基本表删除后，由该基本表导出的所有视图均无法使用了，但是视图的定义没有从字典中清除，需显式使用 DROP VIEW 语句。*(讲义 p037)*

> Caveat (判断题9): DBMS 对视图进行查询时首先进行有效性检查，检查被查询的表、视图等是否存在。
> Caveat (判断题10): 视图是虚拟表，并不表示任何物理数据，只是用来查看数据的窗口。
> Caveat (题目29): 要求视图的更新必须满足查询中的条件，需在建立视图时加上 `WITH CHECK OPTION`。
> Caveat (多选题10): 视图只是逻辑结构，没有独立存储；与并发无关；简化了用户数据查看方式，但不提高查询效率。

## 十、视图的查询、消解与更新限制

**视图消解机制** ⚠：关系数据库管理系统执行对视图的查询时，首先进行有效性检查。如果存在，则从数据字典中取出视图的定义，把定义中的子查询和用户的查询结合起来，转换成等价的对基本表的查询，然后再执行修正了的查询。这一转换过程称为视图消解。*(讲义 p038)*

**更新视图的注意事项** ⚠（由于视图是不实际存储数据的虚表，对视图的更新最终要转换为对基本表的更新，且有严格限制）：*(讲义 p038)*
1. 若视图是由两个以上基本表导出的，则此视图不允许更新。*(讲义 p038)*
2. 若视图的字段来自字段表达式或常数，则不允许对此视图执行 INSERT 和 UPDATE 操作，但允许执行 DELETE 操作。*(讲义 p038)*
3. 若视图的字段来自聚集函数，则此视图不允许更新。*(讲义 p038)*
4. 若视图定义中含有 GROUP BY 子句，则此视图不允许更新。*(讲义 p038)*
5. 若视图定义中含有 DISTINCT 短语，则此视图不允许更新。*(讲义 p038)*
6. 若视图定义中有嵌套查询，并且内层查询的 FROM 子句中涉及的表也是导出该视图的基本表，则此视图不允许更新。*(讲义 p038)*

**视图的作用** ⭐：*(讲义 p038)*
1. 视图能够简化用户的操作*(讲义 p038)*
2. 视图使用户能以多种角度看待同一数据*(讲义 p038)*
3. 视图对重构数据库提供了一定程度的逻辑独立性*(讲义 p038)*
4. 视图能够对机密数据提供安全保护*(讲义 p038)*
5. 适当利用视图可以更清晰地表达查询*(讲义 p038)*

> Caveat (题目30): 视图机制可以让用户只看到特定的行和列，限制对表中特定行或列数据的访问。

## Worked Examples

### 例 1：组合查询条件与空值判断 *(来自第四章-多选-6 / 单选-15)*
*Tags: 字符匹配, 空值判断, 逻辑多重条件*
**问题**：执行查询 `LIKE '_A%'`，可能查询到什么结果？若想查询所有姓张的且出生日期为空的学生，WHERE 条件该怎么写？*(多选题6)*
**解答**：
1. 解析 `_A%`：`_` 代表任意单个字符，`%` 代表任意长度的字符串。所以 `_A%` 匹配的是“第二个字符为 A”的任意长度字符串（如 BACDE, BAABB）。*(多选题6)*
2. 解析姓张且出生日期为空：匹配姓张使用 `姓名 LIKE '张%'`。*(题目15)*
3. 判断空值不能使用 `=NULL`，必须使用 `IS NULL`。*(题目15)*
4. 组合条件使用 AND：最终条件为 `姓名 LIKE '张%' AND 出生日期 IS NULL`。*(题目15)*
