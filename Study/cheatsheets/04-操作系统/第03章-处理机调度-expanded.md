## 一、调度概念与三级调度

**调度的概念**：在多道程序系统中，调度的实质是一种资源分配，处理机调度是对处理机资源进行分配。处理机调度算法是指根据处理机分配策略所规定的处理机分配算法。在多道批处理系统中，一个作业从提交到获得处理机执行，直至作业运行完毕，可能需要经历多级处理机调度。*(讲义 p46)*

**处理机调度的三个层次** ⭐：*(讲义 p46)*

| 层次 | 别称 | 主要任务 | 频率 |
|---|---|---|---|
| 作业调度 | 高级调度 | 按一定原则从外存后备状态作业中挑选作业，分配内存、I/O设备等必要资源，建立相应进程，使其获得竞争处理机的权利 | [原文未述] |
| 中级调度 | 内存调度 | 将暂时不能运行的进程调至外存（挂起态）；具备运行条件且内存空闲时，再调入内存并修改为就绪态，挂在就绪队列上 | [原文未述] |
| 进程调度 | 低级调度 | 按某种方法和策略从就绪队列中选取一个进程，将处理机分配给它 | 很高，一般几十毫秒一次 |

> Caveat (题目3): 中级调度的目的是节省内存，将内存中阻塞态或长期不运行的进程挂起到外存。
> Caveat (题目5): 进程从创建态转换到就绪态由高级调度（作业调度）完成。

## 二、作业管理

**作业**：作业是一个比程序更为广泛的概念，它不仅包含了通常的程序和数据，而且还应配有一份作业说明书，系统根据该说明书来对程序的运行进行控制。在批处理系统中，是以作业为基本单位从外存调入内存的。*(讲义 p47)*

**作业步**：在作业运行期间，每个作业都必须经过若干个相对独立、又相互关联的顺序加工步骤才能得到结果。*(讲义 p47)*

**JCB（作业控制块）** ⭐：为了管理和调度作业，在多道批处理系统中，为每个作业设置了一个作业控制块JCB，它是作业在系统中存在的唯一标志，其中保存了系统对作业进行管理和调度的全部信息。*(讲义 p47)*

JCB 包含的内容：作业标识、用户名称、用户账号、作业类型（CPU繁忙型、I/O繁忙型、批量型、终端型）、作业状态、调度信息（优先级、作业运行时间）、资源需求（预计运行时间、要求内存大小等）、资源使用情况等。*(讲义 p47)*

JCB 生命周期：作业进入系统时由"作业注册"程序建立 JCB → 根据作业类型放入相应后备队列 → 被调度到的作业装入内存 → 运行期间按 JCB 和作业说明书控制 → 作业执行结束进入完成状态时，系统回收资源、撤销 JCB。*(讲义 p47)*

**作业的三个阶段和三种状态** ⭐：*(讲义 p47)*

| 阶段 | 状态 | 说明 |
|---|---|---|
| 收容阶段 | 后备状态 | 操作员把用户提交的作业通过某种输入方式或 SPOOLing 系统输入到硬盘上，再为该作业建立 JCB，并放入作业后备队列中 |
| 运行阶段 | 运行状态 | 作业被作业调度选中后，分配必要资源、建立进程、放入就绪队列；从第一次进入就绪状态开始直到运行结束前 |
| 完成阶段 | 完成状态 | 作业运行完成或发生异常提前结束时，"终止作业"程序回收 JCB 和所有资源，将运行结果形成输出文件后输出 |

**作业调度的主要任务**：根据 JCB 中的信息，检查系统中的资源能否满足作业对资源的需求，以及按照一定的调度算法，从外存的后备队列中选取某些作业调入内存，并为它们创建进程、分配必要的资源。作业调度称为接纳调度。*(讲义 p47)*

每次执行作业调度时需做出两个决定：*(讲义 p47)*
1. 接纳多少个作业——多道程序度
2. 接纳哪些作业——调度算法

## 三、调度的相关问题

### 调度的时机

发生进程调度的情况：*(讲义 p47-48)*
1. 正在运行的进程运行完毕或发生某事件而不能再继续运行
2. 运行中的进程因提出输入/输出请求而暂停运行
3. 在进程通信或同步过程中运行了某种原语操作，如 P 操作等
4. 在可抢先式调度中，有一个比当前进程优先级更高的进程进入就绪队列
5. 在时间片轮转法中，时间片用完

### 进程调度方式 ⭐

| 方式 | 别称 | 定义 | 优点 | 适用场景 |
|---|---|---|---|---|
| 非剥夺调度方式 | 非抢占方式 | 进程一旦获得 CPU 就保持执行直到完成或进入阻塞态，即使有更紧迫的进程进入就绪队列也不抢占 | 实现简单、系统开销小 | 大多数批处理系统 |
| 剥夺调度方式 | 抢占方式 | 当有更为重要或紧迫的进程需要处理机时，立即暂停正在执行的进程，将处理机分配给新进程 | 提高系统吞吐率和响应效率 | 分时系统和大多数实时系统 |

*(讲义 p48)*

剥夺调度必须遵循的原则：优先权、短进程优先、时间片原则等。*(讲义 p48)*

### 进程调度的基本准则 ⭐

| 准则 | 定义 |
|---|---|
| CPU 利用率 | CPU 一直处于忙碌状态时利用率最高 |
| 系统吞吐量 | 单位时间内完成的作业数量，短作业多时吞吐量大 |
| 周转时间 | 作业从提交到完成的时间，包括作业等待、在就绪队列排队、运行、I/O 操作的时间总和 |
| 平均周转时间 | 多个作业的周转时间的平均值 |
| 带权周转时间 | 作业周转时间 / 作业实际运行时间 |
| 平均带权周转时间 | 多个带权周转时间的平均值 |
| 等待时间 | 进程等待处理器的时间之和 |
| 响应时间 | 从用户提交到首次响应所花费的时间 |

*(讲义 p48)*

$$\text{带权周转时间} = \frac{\text{作业周转时间}}{\text{作业实际运行时间}}$$*(讲义 p48)*

### 调度算法的共同目标

| 目标 | 说明 |
|---|---|
| 资源利用率 | 使处理机和其它所有资源尽可能保持忙碌状态 |
| 公平性 | 使诸进程获得合理的 CPU 时间，不会发生进程饥饿；公平是相对的，相同类型进程获得相同服务，不同类型进程提供不同服务 |
| 平衡性 | 使 CPU 和各种外部设备都能经常处于忙碌状态，保持系统资源使用的平衡性 |
| 策略强制执行 | 制订的策略（含安全策略）只要需要就必须准确执行，即使造成某些工作的延迟也要执行 |

*(讲义 p46, p48-49)*

$$\text{CPU的利用率} = \frac{\text{CPU有效工作时间}}{\text{CPU有效工作时间} + \text{CPU空闲时间}}$$*(讲义 p46)*

**批处理系统的目标**：①平均周转时间短；②系统吞吐量高；③处理机利用率高。*(讲义 p49)*

## 四、调度算法 ⭐

### 1. 先来先服务（FCFS）

该算法既可用于作业调度，也可用于进程调度。当在作业调度中采用该算法时，系统将按照作业到达的先后次序来进行调度，优先考虑在系统中等待时间最长的作业，而不管该作业所需执行时间的长短，从后备作业队列中选择几个最先进入该队列的作业，将它们调入内存，为它们分配资源和创建进程，然后放入就绪队列。*(讲义 p49)*

### 2. 短作业优先（SJF）

SJF 算法以作业的长短来计算优先级，作业越短，其优先级越高。作业的长短以作业所要求的运行时间来衡量。可用于作业调度和进程调度。*(讲义 p49)*

**SJF 的缺点** ⚠：*(讲义 p49)*
1. 必须预知作业的运行时间，难以准确估计
2. 对长作业非常不利，长作业周转时间明显增长；完全忽视作业等待时间，可能使作业等待时间过长，出现饥饿现象
3. 在采用 FCFS 算法时，人-机无法实现交互 [原文表述如此]
4. 完全未考虑作业的紧迫程度，不能保证紧迫性作业得到及时处理

**FCFS 与 SJF 比较示例** ⭐：*(讲义 p50)*

| 作业情况 \ 调度算法 | 进程名 | A | B | C | D | E | 平均 |
|---|---|---|---|---|---|---|---|
| | 到达时间 | 0 | 1 | 2 | 3 | 4 | |
| | 服务时间 | 4 | 3 | 5 | 2 | 4 | |
| FCFS | 完成时间 | 4 | 7 | 12 | 14 | 18 | |
| | 周转时间 | 4 | 6 | 10 | 11 | 14 | 9 |
| | 带权周转时间 | 1 | 2 | 2 | 5.5 | 3.5 | 2.8 |
| SJF | 完成时间 | 4 | 9 | 18 | 6 | 13 | |
| | 周转时间 | 4 | 8 | 16 | 3 | 9 | 8 |
| | 带权周转时间 | 1 | 2.67 | 3.1 | 1.5 [原文OCR为"1/5"，按计算应为 3/2=1.5] | 2.25 | 2.1 |

> Caveat: SJF 在同时到达时平均周转时间最小，但可能使长作业饥饿。*(讲义 p49)*

### 3. 优先级调度算法（PSA）

把处理机分配给就绪队列中优先级最高的进程。*(讲义 p50)*

**类型**：*(讲义 p50)*
- 非抢占式优先级调度算法
- 抢占式优先级调度算法

**优先级的类型** ⭐：*(讲义 p50)*

| 类型 | 定义 | 确定依据 |
|---|---|---|
| 静态优先级 | 创建进程时确定，整个运行期间保持不变。优先级利用某一范围内的整数表示（如 0～255），称为优先数 | ①进程类型 ②进程对资源的需求 ③用户要求 |
| 动态优先级 | 创建进程之初赋予初值，随进程推进或等待时间增加而改变，以获得更好的调度性能 | [原文未述] |

对于 FCFS，作业的等待时间就是优先级；对于 SJF，作业的长短就是优先级。但两者都不能反映作业的紧迫程度。*(讲义 p50)*

### 4. 高响应比优先调度算法（HRRN）⭐

每个作业引入一个动态优先级，随等待时间延长而增加，使长作业的优先级在等待期间不断增加，等到足够时间后必然有机会获得处理机。*(讲义 p51)*

$$\text{优先级} = \frac{\text{等待时间} + \text{要求服务时间}}{\text{要求服务时间}}$$*(讲义 p51)*

由于等待时间与服务时间之和就是系统对该作业的响应时间，故该优先级又相当于响应比 $R_p$：*(讲义 p51)*

$$R_p = \frac{\text{等待时间} + \text{要求服务时间}}{\text{要求服务时间}} = \frac{\text{响应时间}}{\text{要求服务时间}}$$*(讲义 p51)*

**HRRN 的特性** ⭐：*(讲义 p51)*
1. 等待时间相同 → 服务时间越短优先权越高 → 有利于短作业
2. 服务时间相同 → 等待时间越长优先权越高 → 实现先来先服务
3. 长作业优先级随等待时间增加而提高 → 等待足够长后可获得处理机 → 较好的折衷

> Caveat (题目36): 响应比高者优先调度算法解决了长作业死等的问题。

### 5. 时间片轮转算法（RR）⭐

系统将所有就绪进程按 FCFS 策略排成就绪队列，每隔一定时间产生中断激活进程调度程序进行调度，把 CPU 分配给队首进程执行一个时间片，运行完毕后再把处理机分配给就绪队列中新的队首进程执行一个时间片，保证就绪队列中所有进程在确定时间段内都能获得一个时间片的处理机时间。*(讲义 p51)*

> Caveat (题目34): 时间片轮转算法是绝对可抢占的，时间一到不管是否完成当前进程必须撤下。
> Caveat (题目20): RR 计算平均周转时间时，必须严格按照时间片大小交替执行，注意进程完成后从就绪队列移除，切勿按整体运行时间简单相加。

### 调度算法场景选择 ⭐

| 需求 | 适用算法 |
|---|---|
| 照顾短作业用户 | 短作业优先调度算法 |
| 照顾紧急作业 | 剥夺式优先级调度算法 |
| 实现人机交互 | 时间片轮转调度算法 |
| 短作业、长作业和交互作业用户都满意 | 多级反馈队列调度算法 |

## 五、同步与互斥

### 制约关系

| 制约关系 | 来源 | 例子 |
|---|---|---|
| 间接相互制约关系 | 共享系统资源（如 CPU、I/O 设备等） | 打印机、磁带机等临界资源必须互斥访问，由系统统一分配 |
| 直接相互制约关系 | 进程间为完成同一任务而相互合作 | 输入进程 A 和计算进程 B 共享缓冲区，缓冲空时 B 阻塞，缓冲满时 A 阻塞 |

*(讲义 p51)*

### 临界资源和临界区 ⭐⚠

**临界资源**：在一段时间内只允许一个进程访问的资源。如打印机、磁带机等硬件资源，以及栈、变量和表格等软件资源，要求被互斥地共享。*(讲义 p51)*

**临界区**：每个进程中访问临界资源的那段代码。*(讲义 p51)*

> Caveat (题目32): 临界区是指进程中访问共享资源的那段代码，不是用来实现互斥访问的那段代码。
> Caveat (题目14): 进程进入临界区后被 I/O 中断时，允许其他进程抢占处理器，但不得进入该进程的临界区。

### 同步机制应遵循的原则 ⭐

| 原则 | 说明 |
|---|---|
| 空闲让进 | 无进程处于临界区时，应允许请求进入的进程立即进入自己的临界区 |
| 忙则等待 | 已有进程进入临界区时，其它试图进入的进程必须等待 |
| 有限等待 | 应保证进程在有限时间内能进入自己的临界区，避免"死等" |
| 让权等待 | 进程不能进入临界区时应立即释放处理机，避免"忙等" |

*(讲义 p52)*

### 信号量机制 ⭐

**整型信号量**：由 Dijkstra 定义，表示资源数目的整型量 S，除初始化外仅能通过 wait(S) 和 signal(S) 两个原子操作访问（又称 P、V 操作）。*(讲义 p52)*

```c
wait(S)
{
    while (S <= 0); //资源不够，循环等待
    S--;
}

signal(S)
{
    S++;//使用完资源后，退出释放资源
}
```
*(讲义 p52)*

> Caveat: 整型信号量的 wait 操作在 S≤0 时会不断测试，未遵循"让权等待"准则，使进程处于"忙等"状态。*(讲义 p52)*

**记录型信号量**：不存在"忙等"现象的进程同步机制，增加了进程链表指针 list 用于链接等待进程。*(讲义 p52-53)*

```c
typedef struct
{
    int value;
    struct process_control_block *list;
} semaphore;  /* [原文OCR为"scmapnore"，疑识别错误] */
```
*(讲义 p53)*

```c
wait(semaphore *S)
{
    S->value--;
    if (S->value < 0)
        block(S->list);
}

signal(semaphore *S)
{
    S->value++;
    if (S->value <= 0)
        wakeup(S->list);
}
```
*(讲义 p53)*

**记录型信号量的关键语义** ⚠：*(讲义 p53)*
- S->value 初值表示系统中某类资源的数目（资源信号量）
- 每次 wait 操作：S->value--，表示请求一个单位的该类资源
- S->value < 0 时：进程调用 block 原语自我阻塞，插入 S->list 链表；S->value 的绝对值表示已阻塞进程的数目
- 每次 signal 操作：S->value++，表示释放一个单位资源
- S->value ≤ 0 时：调用 wakeup 原语唤醒 S->list 链表中的第一个等待进程
- S->value 初值为 1 时：转化为互斥信号量，用于进程互斥

> Caveat (题目15): P、V 操作是一种低级进程通信原语，执行时不可中断。
> Caveat (题目40): PV 操作不能破坏死锁条件，反而可能加强互斥和占有并等待条件，不能防止系统出现死锁。

### 管程机制

**定义**：一个管程定义了一个数据结构和能为并发进程所执行（在该数据结构上）的一组操作，这组操作能同步进程和改变管程中的数据。*(讲义 p53)*

**管程的组成**：*(讲义 p53)*
1. 管程的名称
2. 局部于管程的共享数据结构说明
3. 对该数据结构进行操作的一组过程
4. 对局部于管程的共享数据设置初始值的语句

> Caveat (题目16): 管程定义了共享数据结构和各种进程在该数据结构上的全部操作。

## 六、死锁 ⭐

### 死锁的定义

所谓死锁，是指多个进程因竞争资源而造成的一种僵局（互相等待），若无外力作用，这些进程都将无法向前推进。*(讲义 p54)*

### 死锁产生的原因

| 原因 | 说明 |
|---|---|
| 系统资源的竞争 | 不可剥夺资源数量不足以满足多个进程运行需要（如磁带机、打印机）；只有对不可剥夺资源的竞争才可能产生死锁，对可剥夺资源的竞争不会引起死锁 |
| 进程推进顺序非法 | 请求和释放资源的顺序不当；信号量使用不当也会造成死锁；进程间彼此等待对方发来的消息也会导致死锁 |

*(讲义 p54)*

### 死锁产生的四个必要条件 ⭐⚠

| 必要条件 | 说明 |
|---|---|
| 互斥条件 | 任一时刻只允许一个进程使用资源 |
| 请求和保持条件 | 进程占有部分资源，申请更多的资源，且不会释放已经占有的资源 |
| 不可抢占条件 | 进程已经占用的资源，不会被强制剥夺 |
| 循环等待条件 | 请求资源的进程形成了循环 |

*(讲义 p54)*

> Caveat (题目18): 死锁四个必要条件中，"互斥"条件通常由设备物理特性决定（如打印机），无法通过操作系统策略破坏，而其他三个条件均可破坏。
> Caveat (题目7): 竞争可能引起死锁，但不一定导致死锁；死锁一定由竞争引起。

### 死锁的处理策略

| 策略 | 说明 | 特点 |
|---|---|---|
| 预防死锁 | 设置限制条件，破坏死锁四个必要条件中的一个或几个 | 较简单、易实现，广泛使用 |
| 避免死锁 | 资源动态分配过程中，用某种方法防止系统进入不安全状态 | 限制条件较弱，系统性能较好 |
| 检测死锁 | 允许死锁发生，通过检测机构及时检测出，再采取措施 | 无须事先限制 |
| 解除死锁 | 检测到死锁后，撤销进程或回收资源，将进程从死锁中解脱 | 撤销进程代价较高 |

*(讲义 p54)*

### 死锁的预防 ⭐

| 破坏条件 | 方法 | 可行性 / 缺点 |
|---|---|---|
| 互斥条件 | 允许系统资源都能共享使用 | 不太可行，有些资源（如打印机）只能互斥使用，且有的场合应保护互斥性 |
| 不可剥夺条件 | 进程请求新资源得不到时，必须释放已保持的所有资源，以后需要时再重新申请 | [原文未述] |
| 请求并保持条件 | 预先静态分配方法：进程运行前一次申请完所需全部资源，未满足前不投入运行 | 实现简单，但资源严重浪费，可能导致"饥饿"现象 |
| 循环等待条件 | 顺序资源分配法：给资源编号，规定进程按编号递增顺序请求资源，同类资源一次申请完 | [原文未述] |

*(讲义 p54-55)*

> Caveat (题目4, 11): 资源的有序分配（对资源编号，按序号顺序申请）可以破坏循环等待条件。

**不发生死锁的最少资源数公式** ⭐：*(题目12)*

$$\text{不发生死锁的最少资源数} = n \times (m - 1) + 1$$

其中 $n$ 为进程数，$m$ 为每个进程对资源的最大需求量。*(题目12)*

> Caveat (题目8): 死锁最小资源数计算时，注意"每个进程最多需要 M 台"与"每个进程已占有 M-1 台"的等价转换，此时系统处于临界死锁状态，再多分配 1 个资源即可打破死锁。

### 死锁的避免——银行家算法 ⭐

**安全状态**：系统能按某种进程序列 $\langle P_1, P_2, \ldots, P_n \rangle$（安全序列）为每个进程 $P_i$ 分配其所需资源，直至满足每个进程对资源的最大需求，使每个进程都可顺利完成。*(讲义 p55)*

**银行家算法的数据结构** ⭐：*(讲义 p56)*

| 数据结构 | 说明 |
|---|---|
| Available | 可利用资源向量 |
| Max | 最大需求矩阵 |
| Allocation | 分配矩阵 |
| Need | 需求矩阵 |

**银行家算法步骤** ⭐：*(讲义 p56)*

设 Request_i 是进程 $P_i$ 的请求向量，Request_i[j]=K 表示进程 $P_i$ 需要 K 个 $R_j$ 类型的资源。

1. 若 Request_i[j] ≤ Need[i,j]，转向步骤②；否则出错（所需资源数超过所宣布的最大值）
2. 若 Request_i[j] ≤ Available[j]，转向步骤③；否则尚无足够资源，$P_i$ 须等待
3. 系统试探分配，修改数据结构：
   - Available[j] = Available[j] - Request_i[j]
   - Allocation[i,j] = Allocation[i,j] + Request_i[j]
   - Need[i,j] = Need[i,j] - Request_i[j]
4. 执行安全性算法，检查此次分配后系统是否处于安全状态。若安全，才正式分配；否则，作废试探分配，恢复原来资源分配状态，让 $P_i$ 等待

> Caveat (题目9): 银行家算法是避免死锁的算法（非预防），不破坏死锁必要条件。系统处于安全状态一定无死锁；不安全状态可能产生死锁（但不一定）。

### 死锁的检测

利用资源分配图进行检测。资源分配图 $G=(N, E)$，N 分为进程结点集 $P=\{P_1, P_2, \ldots, P_n\}$ 和资源结点集 $R=\{R_1, R_2, \ldots, R_n\}$，$N = P \cup R$。*(讲义 p56)*

- 请求边 $e=(P_i, R_j)$：由 $P_i$ 指向 $R_j$，表示进程 $P_i$ 请求一个单位的 $R_j$ 资源
- 分配边 $e=(R_j, P_i)$：由 $R_j$ 指向 $P_i$，表示把一个单位的 $R_j$ 分配给 $P_i$

**资源分配图简化方法** ⭐：*(讲义 p57)*
1. 找出一个既不阻塞又非独立的进程结点 $P_i$，使其获得所需资源运行完毕，释放全部资源，消去其所有请求边和分配边，使之成为孤立结点
2. $P_i$ 释放资源后可使其他进程获得资源继续运行，重复上述过程
3. 若能消去图中所有边，使所有进程结点都成为孤立结点 → **完全简化** → 无死锁
4. 若不能通过任何过程使该图完全简化 → **不可完全简化** → 存在死锁

### 死锁的解除 ⭐

| 方法 | 说明 |
|---|---|
| 资源剥夺法 | 挂起某些死锁进程，抢占其资源分配给其他死锁进程；应防止被挂起进程长时间得不到资源而处于资源匮乏状态 |
| 撤销进程法 | 强制撤销部分甚至全部死锁进程并剥夺资源；可按进程优先级和撤销进程代价的高低进行 |
| 进程回退法 | 让一个或多个进程回退到足以回避死锁的地步，进程回退时自愿释放资源而非被剥夺；要求系统保持进程的历史信息，设置还原点 |

*(讲义 p55, 57-58)*

> Caveat (题目1, 17): 解除死锁可采用资源剥夺法和撤销进程法，但不采用从非死锁进程处抢夺资源的方法。

## 八、一句话记忆要点

- **三级调度**：高级（作业）选作业入内存建进程，中级（内存）管挂起/调回，低级（进程）分配 CPU。
- **调度方式**：非剥夺简单开销小（批处理），剥夺提高响应（分时/实时）。
- **FCFS**：按到达顺序，公平但对短作业不利。
- **SJF**：按运行时间最短优先，平均周转时间最短，但长作业可能饥饿。
- **HRRN**：响应比=(等待+服务)/服务，兼顾短作业和长作业，解决饥饿。
- **RR**：时间片轮转，绝对可抢占，适合分时交互。
- **临界区**：访问临界资源的那段代码（不是实现互斥的代码）。
- **同步四原则**：空闲让进、忙则等待、有限等待、让权等待。
- **整型信号量**：忙等，不满足让权等待；记录型信号量：block/wakeup，满足让权等待。
- **死锁四条件**：互斥、请求和保持、不可抢占、循环等待——互斥不可破坏。
- **银行家算法**：避免死锁（非预防），安全状态一定无死锁。
- **死锁解除**：资源剥夺、撤销进程、进程回退——不从非死锁进程抢资源。

## Worked Examples

### 例 1：死锁临界资源计算 *(来自 exam-signals 候选 1)*

*Tags: 死锁预防, 资源分配*

**问题**：某计算机系统中有 8 台打印机，有 K 个进程竞争使用，每个进程最多需要 3 台打印机。该系统可能会发生死锁的 K 的最小值是多少？*(题目8)*

**解答**：`[PROPOSED — verify]`

利用不发生死锁的最少资源数公式 $R = n \times (m-1) + 1$，其中 $n$ 为进程数，$m$ 为每个进程最大需求。

1. 死锁发生的临界条件：每个进程都已获得 $m-1$ 台资源，但均未达到最大需求，且系统已无剩余资源。
2. 设系统总资源 $R = K \times (m-1)$ 时，系统无空闲资源，所有进程互相等待，发生死锁。
3. 代入：$8 = K \times (3-1)$，解得 $K = 4$。
4. 验证 $K=3$：最坏情况每个进程分得 2 台，共占 6 台，系统还剩 2 台，足够分给一个进程使其运行完毕并释放资源，不会死锁。
5. 结论：发生死锁的最小 $K$ 值为 **4**。

*(题目8)*

### 例 2：时间片轮转法平均周转时间计算 *(来自 exam-signals 候选 2)*

*Tags: 时间片轮转, 平均周转时间*

**问题**：有 4 个进程 A、B、C、D，依次进入就绪队列（可视为同时到达），按轮转法分别运行 11、7、2、4 个时间单位，时间片为 1，求平均周转时间。*(题目20)*

**解答**：`[PROPOSED — verify]`

1. 队列初始顺序为 A, B, C, D，每个进程每次执行 1 个时间单位。
2. 模拟执行过程（每轮 4 个进程各执行 1 个时间单位，完成的移出队列）：
   - 第 1 轮：A(1) B(1) C(1) D(1) → 剩余 A(10) B(6) C(1) D(3)
   - 第 2 轮：A(1) B(1) C(1) D(1) → C 在第 7 个时间单位完成；剩余 A(9) B(5) D(2)
   - 第 3 轮：A(1) B(1) D(1) → 剩余 A(8) B(4) D(1)
   - 第 4 轮：A(1) B(1) D(1) → D 在第 14 个时间单位完成；剩余 A(7) B(3)
   - 第 5 轮：A(1) B(1) → 剩余 A(6) B(2)
   - 第 6 轮：A(1) B(1) → 剩余 A(5) B(1)
   - 第 7 轮：A(1) B(1) → B 在第 20 个时间单位完成；剩余 A(4)
   - 第 8-11 轮：A(1)×4 → A 在第 24 个时间单位完成
3. 周转时间 = 完成时间 - 到达时间（到达时间均为 0）：C=7, D=14, B=20, A=24。
4. 平均周转时间 = $(7 + 14 + 20 + 24) / 4 = 65 / 4 = \mathbf{16.25}$。

*(题目20)*
