# 软件设计与开发 · 系统维护 — 复习资料

> 来源：6-3-软件设计与开发计算机类讲义.pdf 物理页 [113, 124]
> 说明：⭐=高频考点，⚠=易错点；> Caveat: 为陷阱/技巧；末尾 Worked Examples 为代表性题目（无书内答案的标 [PROPOSED — verify]）。

## 一、本章概览
软件维护是软件生命周期的最后一个阶段，也是持续时间最长、工作量最大的阶段。本章核心围绕「软件维护」展开，重点涵盖维护的四类活动定义、维护的特点（结构化与非结构化、代价高昂、问题众多）、维护的过程组织、软件的可维护性因素（5个决定因素与文档作用），以及面向老程序的预防性维护与软件再工程过程。*(讲义 p113)*

## 二、软件维护的定义与分类 ⭐
**软件维护的定义**：所谓软件维护就是在软件已经交付使用之后，为了改正错误或满足新的需要而修改软件的过程。*(讲义 p113)*

**四类维护活动**：可以通过描述软件交付使用后可能进行的4项活动，具体地定义软件维护。*(讲义 p113)*

| 维护类型 | 触发原因 / 具体定义 | 占比 |
| :--- | :--- | :--- |
| **完善性维护** ⭐ | 用户提出增加新功能、修改已有功能或一般性改进意见。 | $50\% \sim 68\%$ (占比最大) |
| **改正性维护** ⭐ | 诊断和改正遗留错误（测试未暴露的潜藏错误）。 | $17\% \sim 25\%$ |
| **适应性维护** ⭐ | 为了和变化了的环境适当地配合而进行的修改（如硬件/操作系统升级）。 | $18\% \sim 25\%$ |
| **预防性维护** | 为改进未来的可维护性或可靠性而修改软件。 | $4\%$左右 (相对较少) |

- *Caveat* (第五章单选4/判断3): "提高系统性能、扩充功能"属于**完善性维护**，极其容易被误认为适应性维护。*(题目第五章单选4/判断3)*
- *Caveat* (第五章单选5): 开发测试不彻底导致运行出错，属于**改正性维护**。*(题目第五章单选5)*

## 三、软件维护的特点 ⭐
### 1. 结构化维护 vs 非结构化维护
**非结构化维护**：如果软件配置的唯一成分是程序代码，维护活动从艰苦地评价程序代码开始，常因内部文档不足产生误解，且改动后果难于估量。*(讲义 p114)*
**结构化维护**：如果有一个完整的软件配置存在，维护工作从评价设计文档开始。步骤为：修改设计并复查 → 编写代码 → 回归测试 → 再次交付。*(讲义 p114)*
- *Caveat* (第五章判断4): 结构化维护是**有**完整软件配置的维护；非结构化维护才是没有完整配置。*(题目第五章判断4)*

### 2. 维护的代价高昂
维护的费用稳步上升，1970年占 $35\% \sim 40\%$，1980年占 $40\% \sim 60\%$，1990年上升为 $70\% \sim 80\%$。维护的无形代价包括：耽误开发良机、引起用户不满、引入潜伏错误降低质量、造成开发过程混乱、生产率大幅度下降。*(讲义 p114)*

**维护工作量模型**：
$$M = P + K \times exp(c - d)$$
*(讲义 p114)*
**记号约定**：
- $M$ = 维护用的总工作量
- $P$ = 生产性工作量（如分析评价、修改设计、编写程序代码等）
- $K$ = 经验常数
- $c$ = 复杂程度
- $d$ = 维护人员对软件的熟悉程度

### 3. 维护的问题很多 ⚠
绝大多数问题归因于软件定义和软件开发的方法有缺点。*(讲义 p115)*
- *Caveat* (第五章单选8): 软件维护困难的主要原因要追溯到**开发设计中的缺陷**（前期没有科学管理和规划）。*(题目第五章单选8)*

## 四、软件维护过程 ⭐
维护过程本质上是修改和压缩了的软件定义和开发过程。*(讲义 p115)*
**标准维护流程组织**：*(讲义 p115-117)*
1. **维护组织**：建立维护管理员、系统管理员、变化授权人等角色，明确责任。
2. **维护报告**：用户填写标准化"软件问题报告表"（维护要求表）；内部制定"软件修改报告"交变化授权人批准。
3. **维护的事件流**：确定维护类型 → 估量严重程度/优先级 → 分析设计 → 修改代码 → 回归测试 → 复审。
4. **保护维护记录**：Swanson 提出了18项记录数据（如源语句数、运行次数、失效次数等）。
5. **评价维护活动**：从7个方面进行定量度量（如每次运行平均失效次数、不同维护类型所占百分比等）。

## 五、软件的可维护性 ⭐
**软件的可维护性定义**：维护人员理解、改正、改动或改进这个软件的难易程度。提高可维护性是支配软件工程方法学所有步骤的关键目标。*(讲义 p117)*

**决定软件可维护性的5个因素**：*(讲义 p117-118)*
| 决定因素 | 具体内涵与关联设计原理 |
| :--- | :--- |
| **可理解性** | 依赖模块化（高内聚，松耦合）、详细设计文档、结构化设计。 |
| **可测试性** | 依赖文档与可用测试工具。可用程序复杂度（环形复杂度）度量。 |
| **可修改性** | 和耦合、内聚、信息隐藏、局部化、控制域与作用域关系直接有关。 |
| **可移植性** | 转移计算环境的难易程度。与硬件/OS相关代码集中放到特定模块。 |
| **可重用性** | 同一事物不修改或稍加改动在不同环境重复使用。构件重用能减少改正性/适应性/完善性维护需求。 |

- *Caveat* (第五章单选9): 决定可维护性的因素是以上5个，**不包括可用性、可靠性、可扩展性等**。*(题目第五章单选9)*
- *Caveat* (第五章单选10): 总体设计阶段提高可维护性的做法是**提高模块的独立性**。*(题目第五章单选10)*

**文档与复审**：文档是影响软件可维护性的决定因素。分为**用户文档**和**系统文档**。必须在开发的每个阶段进行可维护性复审，测试结束时进行最正式的配置复审。当修改代码时必须立即修改文档。*(讲义 p118)*

## 六、预防性维护与软件再工程 ⭐
**预防性维护**：由 Miller 提出，定义为："把今天的方法学应用到昨天的系统上，以支持明天的需求。"*(讲义 p119)*

**软件再工程过程**的6个典型活动：*(讲义 p119-120)*
1. **库存目录分析**：分析应用系统，选出再工程的候选者。
2. **文档重构**：采用"使用时建文档"方法。
3. **逆向工程** ⭐：分析程序以便在比源代码更高的抽象层次上创建出程序的某种表示的过程。
4. **代码重构**：最常见的再工程活动，重构可疑模块代码。
5. **数据重构**：全范围的再工程活动，始于逆向工程，重新定义数据模型。
6. **正向工程**：不仅恢复设计信息，且使用该信息改变或重构现有系统，以提高整体质量。
- *Caveat* (第五章判断5): 软件的逆向工程是一个**恢复设计结果**的过程。*(题目第五章判断5)*

## Worked Examples

### 例 1：软件维护四类活动辨析 *(来自 exam-signals 候选 4)*
*Tags: 软件维护的定义, 改正性维护, 适应性维护, 完善性维护, 预防性维护*
**问题**：为提高系统性能而进行的修改、开发测试不彻底导致的修改、为使软件在改变环境下仍能使用分别属于什么维护？*(题目第五章单选3/4/5)*
**解答**：`[PROPOSED — verify]`
1. 识别四类维护的触发原因：改错（改正性）、环境变化（适应性）、扩充功能/提高性能（完善性）、为未来维护做准备（预防性）。
2. "提高系统性能"属于用户提出的新要求或改进意见 → **完善性维护**。
3. "测试不彻底遗留错误" → **改正性维护**。
4. "环境改变" → **适应性维护**。
5. 补充考点：在实际维护工作中，**完善性维护占比最大**。
