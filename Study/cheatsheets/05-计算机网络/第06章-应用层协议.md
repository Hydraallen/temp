## 一、本章概览

本章介绍 OSI 模型中最靠近用户的一层——应用层的核心协议。涵盖六大应用层协议/服务：**域名系统 DNS**（主机名→IP 地址转换）、**文件传输协议 FTP/TFTP**（文件共享与传输）、**远程终端协议 Telnet**（远程登录）、**电子邮件 SMTP/POP3**（邮件发送与接收）、**万维网 WWW**（超文本信息检索，含 URL 与 HTTP）、**动态主机配置协议 DHCP**（自动分配 IP 地址）。这些协议是 Internet 应用层的基础设施，与传输层 TCP/UDP 紧密关联，是考试高频出题区域。

## 二、知识点详解

### 1. 域名系统 DNS ⭐

#### 1.1 域名系统的概念

- **Internet 上的主机名**：用英文字母和数字组成。如：`mail.tsinghua.edu.cn`。
- **域名**是一个逻辑概念，用于标识主机，并不反映主机所在的物理地点。
- **域名系统 DNS（Domain Name System）**：负责将主机名转换为 IP 地址。Internet 的域名系统被设计成一个联机分布式数据库系统，并采用客户／服务器模式。
- **域名服务器程序**：在专设的结点上运行。运行该程序的计算机称为域名服务器。

#### 1.2 域名结构 ⭐

- 域名采用**层次结构**：`三级域名．二级域名．顶级域名`
- 每一级域名不超过 **63** 个字符（不分大小写），完整的域名不超过 **255** 个字符。
- **顶级域名**由 Internet 的有关机构管理，有三大类：

| 类别 | 说明 | 示例 |
|---|---|---|
| 国家顶级域名 | 以英文缩写表示 | cn、us、uk 等 |
| 国际顶级域名 | 用于国际性组织 | int |
| 通用顶级域名 | 各类组织 | com（公司企业）、net（网络服务机构）、org（非赢利性组织）、edu（教育机构）、gov（政府部门，美国专用）、mil（军事部门，美）、firm（公司企业）、shop（销售公司和企业）、web（突出万维网活动的单位）、arts（突出文化娱乐活动的单位）、rec（突出消遣娱乐活动的单位）、info（提供信息服务的单位）、nom（个人） |

- **下级域名**：各级域名由其上一级域名管理机构管理。
- 在国家顶级域名下注册的二级域名均由该国家自行确定。如：荷兰不再设二级域名，所有机构均在顶级域名 nl 之下；日本将教育机构二级域名定为 ac，企业机构定为 co。
- **中国**将二级域名划分为"类别域名"和"行政区域域名"：
  - **类别域名 6 个**：ac（科研机构）、com（企业）、edu（教育）、gov（政府）、net（网络信息中心）、org（非赢利性组织）。
  - **行政区域名 34 个**：省、自治区、直辖市，如 bj（北京）、sh（上海）。
- **三级域名**：在中国，在二级域名 edu 下申请注册三级域名由中国教育和科研网网络中心负责；在其他二级域名下申请注册三级域名，应向中国互联网网络信息中心（CNNIC）申请。

> ⚠ 域名层次排列方式：从右到左越来越小，分多层排列（练习9，答案D）。

#### 1.3 域名与 IP 的转换

- **转换过程**：源主机的某个应用进程，将待转换的域名放在 DNS 请求报文中，以 **UDP 数据报**方式发给本地域名服务器。本地域名服务器在查找域名后，将对应的 IP 地址放在回答报文中，返回给请求进程。
- 若本地域名服务器不能回答该请求，则该域名服务器就暂时成为 DNS 中的另一个客户，去寻找能够回答该请求的域名服务器，直到找到为止。
- **优化域名查询**：每个域名服务器都维护一个**高速缓存**，存放着最近使用过的域名以及从何处获得域名的 IP 地址映射信息的记录。

#### 1.4 解析方法 ⭐

| 方法 | 说明 |
|---|---|
| **递归查询** | 逐级查询，逐级回答 |
| **递归与迭代结合** | 根域名服务器在收到查询后，直接将下属的授权域名服务器的 IP 地址返回给发出查询的本地域名服务器，由本地域名服务器直接向授权域名服务器查询，从而可减轻根域名服务器的负担 |

### 2. 文件传输协议 FTP、TFTP ⭐

#### 2.1 文件传输协议 FTP ⭐

- **基于 TCP**。使用端口号 **20**（数据连接）和 **21**（控制连接）。
- 传送文件类型分为两种：**ASCII**（文本）文件和二进制（非文本）文件。[OCR 原文为"ASOI"，应为 ASCII]
- FTP 服务器进程由两部分组成：一个**主进程**，负责接受请求；若干个**从属进程**，负责处理单个请求。
- 工作时需要建立两个连接：**控制连接**和**数据连接**。
- **匿名 FTP**：使用默认的帐号（anonymous）、口令即可登录访问的 FTP 服务。
- **特点**：必须先获得文件的副本，才能修改文件，然后将该副本传回到服务器。

> ⚠ FTP 默认 TCP 端口号为 21（练习11，答案A）。FTP 依赖 TCP 协议（练习5，答案A）。

#### 2.2 简单文件传送协议 TFTP ⭐

- TFTP 是一个很小且易于实现的文件传送协议。
- TFTP 使用客户服务器方式和使用 **UDP 数据报**，因此 TFTP 需要有自己的差错改正措施。
- TFTP 只支持文件传输而不支持交互。
- TFTP 没有一个庞大的命令集，没有列目录的功能，也不能对用户进行身份鉴别。

**TFTP 主要特点：**
1. 每次传送的数据 PDU 中有 **512 字节**的数据，但最后一次可不足 512 字节。
2. 数据 PDU 也称为文件块（block），每个块按序编号，从 **1** 开始。
3. 支持 ASCII 码或二进制传送。
4. 可对文件进行读或写。
5. 使用很简单的首部。

### 3. 远程终端协议 Telnet ⭐

- **远程登录**是为用户提供的以终端方式与 Internet 上的主机建立在线连接的一种服务。
- 连接建立后，用户的计算机就可以作为远程主机的一台终端来使用该主机提供的各种资源。
- 实现远程登录的工具程序很多，最常见的就是 **Telnet 程序**（与 Telnet 协议同名），在 Unix 系统和 Windows、DOS 系统中都可以找到。
- 用户在本地计算机上运行 telnet 程序，通过 Internet 把用户的计算机与指定的远程主机连接起来。一旦连接成功，telnet 就成为用户主机与另外一个计算机的中介物。
- 用户通过键盘输入的任何信息都被发送到远程主机；在远程主机屏幕上显示的信息也被传回到用户的屏幕上。
- **术语**：用户所在的计算机称为**本地（Local）计算机**或本地主机（Host）；用户所连接的另一台计算机称为**远程（Remote）计算机**或远程主机。

> ⚠ Telnet 是远程登录协议（练习8，答案D）。Telnet 工作在应用层（练习16/17，答案D）。Telnet 依赖 TCP 协议（练习5，答案A）。

### 4. 电子邮件 SMTP、POP3 和 IMAP ⭐

#### 4.1 电子邮件概况

- 电子邮件是 Internet 最早开发的而且是使用最广泛的应用之一。
- 1982 年制订了 ARPANET 的"简单电子邮件协议"**SMTP** 和"电子邮件报文格式"**MAIL**。
- CCITT 制订了"报文处理系统"**MHS** 的标准 **X.400**。
- 1993 年制订了"通用因特网扩充"**MIME** 标准（为传送多媒体数据）。
- 目前因特网的电子邮件系统已成为事实上的标准，X.400 由于过于复杂而基本上被淘汰。

#### 4.2 电子邮件系统组成与服务

一般由两个子系统组成：

| 子系统 | 说明 |
|---|---|
| **用户代理 UA** | 用户与系统的接口，本地程序，提供收发邮件的界面 |
| **邮件服务器** | 运行在后台的服务器软件，负责消息的管理、传送与提交 |

#### 4.3 电子邮件系统的基本功能

1. **撰写**：给用户提供创建、编辑消息的环境。
2. **传输**：自动地将消息传送给接收者，一般为基于连接方式。
3. **报告**：向发送者反馈消息的传送情况。
4. **呈现**：显示来信的内容。
5. **处理**：使收信人可根据情况按不同方式对来信进行处理。

#### 4.4 SMTP 协议与 POP3 协议 ⭐

在电子邮件系统中，最重要的协议就是 SMTP 协议与 POP3 协议。**发送邮件使用 SMTP 协议，接收邮件使用 POP3 协议。**

| 特性 | SMTP | POP3 |
|---|---|---|
| 全称 | Simple Mail Transfer Protocol，简单邮件传输协议 | 邮局协议版本 3（Post Office Protocol version 3） |
| 功能 | 负责电子邮件的**发送** | 用于**接收** Internet 上的电子邮件 |
| 传输层 | TCP | TCP |
| 端口号 | **25** | **110** |
| 工作方式 | 客户/服务器模式，在 TCP 连接上传送 ASCII 码邮件 | 客户 POP3 软件安装在收信人计算机上；服务器 POP3 软件安装在邮件服务器上 |
| 连接特点 | 发送前要先建立 TCP 连接，可在此连接上发送多个电子邮件，发送完后释放 TCP 连接 | POP 客户与 POP 服务器建立 TCP 连接后，发送用户名和口令访问邮箱，可列出清单并逐个读取邮件报文 |

**SMTP 补充说明：**
- SMTP 属于 TCP/IP 协议族，帮助每台计算机在发送或中转信件时找到下一个目的地。
- SMTP 的重要特点是可以在交互的通信系统中转发邮件，它是一组用于由源地址到目的地址传送邮件的规则，由它来控制信件的中转方式。
- SMTP 服务器是遵循 SMTP 协议的发送邮件服务器，用来发送或中转发出的电子邮件。

**电子邮件地址格式：**

```
loginname@full host name.domain name
即：登录名@主机名．域名
```

中间用一个表示"在"（at）的符号"@"分开，符号的左边是对方的登录名，右边是完整的主机名，它由主机名与域名组成。

> ⚠ 发送邮件用 SMTP（端口25），接收/读取邮件用 POP3（端口110）（练习6，答案B）。SMTP、POP3 依赖 TCP 协议（练习5，答案A）。

### 5. 万维网 WWW ⭐

#### 5.1 什么是 WWW

- **WWW** 是 World Wide Web 的简称。类似的简称有 Web、W3、3W 等，译为万维网。
- WWW 使用**超文本（Hypertext）**组织、查找和表示信息，利用了从一个站点到另一个站点的链接。彻底摆脱了以前查询工具只能按特定路径一步步地查找信息的限制。
- 具有连接已有信息系统（Gopher、FTP、News）的能力。
- 超文本文件使用**超文本标记语言（HTML）**格式写成，由欧洲粒子物理实验室（CERN）提出。
- WWW 文本含有作为**热链**的词、词组、句子、图象、图标，通过颜色和字体的改变与普通文本区别开来，含有指向其他 Internet 信息 **URL 地址**（Uniform Resource Locator）[OCR 原文为"Unifoiy"，应为 Uniform]。
- 链接同样可以指向声音、视频等多媒体超文本，与多媒体一起形成**超媒体（Hypermedia）**。
- WWW 实现了文本与声音、图象的同时传输。

#### 5.2 WWW 的历史

- 1989 年，位于瑞士的欧洲粒子物理实验室（CERN）实施超媒体项目，最初方案由 **Tim Berners Lee** 负责实施。
- 1990 年 11 月，NeXT 发布了最初的 World Wide Web 模型。
- 1993 年 NCSA 开发出了 WWW 的图形化浏览器 **Mosaic**。
- 1994 年，Netscape Communicator 公司推出 **Netscape Navigator**。
- 1996 年和 1997 年，微软发布 **Internet Explorer 3.0 和 4.0**。
- 目前流行的 Web 浏览器：Mozilla Firefox、Google Chrome、Internet Explorer、傲游、Safari 等。

#### 5.3 WWW 的工作原理 ⭐

- WWW 由三部分组成：**客户机（Client）**、**服务器（Server）**和 **HTTP 协议**。
- 以 **Client/Server** 方式工作[OCR 原文为"Client/Scerver"，应为 Client/Server]。
- 客户机向服务器发送一个请求，并从服务器上得到一个响应。服务器负责管理信息和对来自客户机的请求作出回答。
- 客户机与服务器使用 **HTTP 协议**（超文本传送协议）传送文本信息。
- 服务器的基本信息单位叫**网页**（Web 页），由 **HTML** 写成。
- 客户机请求采用 **URL** 形式，Web 页中有许多超链，每个超链指向含有其他信息的 URL 地址。
- 用户用来查看 HTML 超文本的客户机软件称为 **WWW 浏览器**。浏览器分为文本浏览器（如 Lynx）和图形浏览器（如 Netscape）等，常用的是图形浏览器。

> ⚠ WWW 工作的三部分是：客户机、服务器、HTTP 协议（练习18，答案D，FTP 不是 Web 工作的部分）。浏览器与 Web 服务器之间使用 HTTP 协议（练习10，答案C）。

#### 5.4 URL 地址 ⭐

- **URL** 是 Uniform Resource Locator 的缩写，译为统一资源定位器或普遍资源定位器。
- URL 的思想：使所有的信息资源都能得到有效利用，将分散的孤立的信息点连接起来，实现资源的统一寻址。

**URL 组成（三部分）：**

```
协议://主机名/目录或文件名
```

示例：`http://daytona.raceway.com/home/www/files/race.html`

- **协议**：访问页的方式，即浏览器用于取得超文本文件的协议或程序。如 http、ftp、gopher 等。协议必须与系统安装的信息服务器匹配。
- **主机名**：Internet 上用于标识信息保存的地方，如 `www.netcom.com`、`ftp.apple.com`。同一个主机名因使用不同的协议而有不同的 URL。
- URL 的主机名中可包含一个**端口号**，写在主机名和目录之间，如：`http://my-public-access-unix.com:150/pub/file`。当服务器只提供对应于该特定端口的信息服务时才有必要写端口号。
- **目录或文件名**：信息在主机上的文件所在，可以是实际的目录名或文件名，或者是协议用于指定信息所在的另一种指示器。

**URL 允许使用的字符：** 字母 A-Z、a-z，数字 0-9，美元符号（$），加号（+），减号（-），下划线（_）和句点（.）。其他字符必须用 URL 换码字符代替。

**URL 换码序列：** 以百分号（%）加上两个字符形成的 ISO-Latin-1 字符集（标准 ASCII 码的超集）。

| 换码 | 字符 |
|---|---|
| `%20` | 空格符 |
| `%3f` | 问号 |
| `%2f` | 斜杠 |

**常见 URL 类型：**

| 类型 | 格式示例 | 说明 |
|---|---|---|
| **HTTP** | `http://www.foo.com/home/foo/` | Internet 上最流行的 URL 类型，超文本传输协议，WWW 服务器用于传输 HTML 页的协议。若 URL 以斜杠结尾，最后一部分被认为是目录名，HTTP 服务器通常会自动打开 `index.html` 文件 |
| **匿名 FTP** | `ftp://ftp.com/home/foo` | 用于访问 FTP 服务器上的文件，通常匿名，用 anonymous 作为登录名，以电子邮件地址作为口令 |
| **非匿名 FTP** | `ftp://username:password@ftp.foo.com/home/foo/homepage.html` | 加入账号、密码的 FTP URL。URL 中无法对口令进行隐藏 |
| **文件（file）** | `file://dirl/dir2/file` 或 `file://localhost/dirl/dir2/file` | 索引本地盘上的文件，引用浏览器所在系统上的文件。第一部分是空主机名（3 个而非 2 个斜杠），或用 localhost 作主机名 |

### 6. 动态主机配置协议 DHCP ⭐

#### 6.1 DHCP 概念

| 角色 | 说明 |
|---|---|
| **DHCP Client** | DHCP 客户端，通过 DHCP 协议请求 IP 地址的客户端。是接口级概念，一个主机多个以太接口可各配一个 DHCP 客户端；交换机上每个 Vlan 接口也可配置 |
| **DHCP Server** | DHCP 服务端，负责为 DHCP 客户端提供 IP 地址，并负责管理分配的 IP 地址 |
| **DHCP Relay** | DHCP 中继器，DHCP 客户端跨网段申请 IP 地址时，实现 DHCP 报文的转发功能 |
| **DHCP Security** | DHCP 安全特性，实现合法用户 IP 地址表的管理功能 |
| **DHCP Snooping** | DHCP 监听，记录通过二层设备申请到 IP 地址的用户信息 |

#### 6.2 DHCP 工作原理 ⭐

DHCP 工作过程分为以下阶段：

**① 发现阶段（DHCP Discovery）**
- DHCP 客户端寻找 DHCP 服务端的过程。
- 客户端发送 **DHCP Discovery** 报文，是**广播包**，源地址为 `0.0.0.0`，目的地址为 `255.255.255.255`。
- 网络上所有支持 TCP/IP 的主机都会收到该报文，但只有 DHCP Server 会响应。
- 如果网络中存在多个 DHCP Server，则多个均会回复。
- **DHCP Relay 场景**：如果同一个 vlan 内没有 DHCP Server，而该 VlanIf 配置了 DHCP Relay 功能，则该 VlanIf 即为 DHCP 中继。DHCP 中继会将该 DHCP 报文的源 IP 地址修改为该 VlanIf 的 IP 地址，同时修改 DHCP 报文中 giaddress 为 VlanIf 的 IP 地址，并以**单播**将 DHCP Discovery 发送到 DHCP Server 端。

**② DHCP Server 提供阶段（DHCP Offer）**
- DHCP Server 响应 DHCP Discovery 所发的 DHCP Offer。
- DHCP Server 收到 DHCP Discovery 报文后，解析该报文请求 IP 地址所属的 Subnet，从 `dhcpd.conf` 文件中匹配的 subnet 中取出一个可用的 IP 地址。
- **IP 地址冲突检测机制**：从可用地址段选择一个 IP 地址后，首先发送 ICMP 报文来 ping 该 IP 地址，如果收到该 IP 地址的 ICMP 回复，则抛弃该 IP 地址，重新选择并继续测试，直到找到网络中没有人使用的 IP 地址。此机制可配置。
- 设置在 DHCP Discovery 报文中 **yiaddress** 字段中，表示为该客户端分配的 IP 地址，并为该 Lease 设置该 Subnet 配置 Option（如默认租期、最大租期、router 等信息）。

**DHCP 从地址池中选择 IP 地址的优先级：**
1. 当前已经存在的 IP-MAC 的对应关系
2. Client 以前的 IP 地址
3. 读取 Discovery 报文中的 Requested IP Address Option 的值（如果存在并且 IP 地址可用）
4. 从配置的 Subnet 中选择 IP 地址

> DHCP Server 解析 DHCP Discovery 请求的 IP 所属 Subnet 时：首先看报文中 **giaddress** 是否有 DHCP Relay，如果有，则从 giaddress 所述的 subnet 中可用 IP 地址段中获取并分配 IP；如果 giaddress 没有 IP 地址，则从该 DHCP Server 绑定的接口的 IP 地址所属的网段分配 IP 地址。

**③ DHCP Client 选择阶段（DHCP Request）**
- DHCP Client 收到若干个 DHCP Server 响应的 DHCP Offer 报文后，选择其中一个 DHCP Server 作为目标。
- 选择策略通常为**选择第一个响应的 DHCP Offer 报文**所属的 DHCP Server。
- 以**广播**方式回答一个 **DHCP Request** 报文，该报文中包含向目标 DHCP 请求的 IP 地址等信息。
- 之所以以广播方式发出，是为了通知其他 DHCP Server 自己将选择该 DHCP Server 所提供的 IP 地址。

**④ DHCP Server 确认阶段（DHCP ACK）**
- DHCP Server 收到 DHCP Client 发送的 DHCP Request 后，确认要为该 Client 提供的 IP 地址，向该 Client 响应一个包含该 IP 地址以及其他 Option 的 **ACK** 报文。
- DHCP Client 即可以将该 IP 地址与网卡绑定。
- 其他 DHCP Server 都将收回自己之前为 DHCP Client 提供的 IP 地址。

**⑤ DHCP Client 重新登录网络**
- DHCP Client 重新登录后，发送一个包含之前 DHCP Server 分配的 IP 地址信息的 **DHCP Request** 报文。
- DHCP Server 收到该请求后，会尝试让 DHCP 客户端继续使用该 IP 地址，并回答一个 **ACK** 报文。
- 如果该 IP 地址无法再次分配给该 DHCP Client，DHCP 回复一个 **NAK** 报文。DHCP Client 收到 NAK 后，会重新发送 **DHCP Discovery** 报文来重新获取 IP 地址。

**⑥ DHCP Client 更新租约**
- DHCP 获取到的 IP 地址都有一个**租约**，租约过期后，DHCP Server 将回收该 IP 地址。
- 更新方式：当前租约期限过了**一半**后，DHCP Client 都会发送 **DHCP Renew** 报文来续约租期。

## 五、高频考点与易错提醒

1. ⚠ **应用层是 OSI 最靠近用户的一层**，负责为用户的应用程序提供网络服务（例题1，练习）。
2. ⚠ **DNS 将域名转换为 IP 地址**，工作在应用层，使用 UDP 数据报传输（例题2，练习2/7/10）。
3. ⚠ **FTP 默认 TCP 端口号 21**（控制连接），数据连接用 20；FTP 依赖 TCP 协议（练习5/11，答案A/A）。
4. ⚠ **FTP 与 TFTP 的区别**：FTP 基于 TCP，TFTP 基于 UDP；TFTP 每次传送 512 字节数据块，编号从 1 开始（练习5）。
5. ⚠ **SMTP 发送邮件（端口25），POP3 接收/读取邮件（端口110）**，两者均依赖 TCP（练习5/6，答案A/B）。
6. ⚠ **电子邮件地址格式**：`登录名@主机名.域名`（例题5，练习3）。
7. ⚠ **Telnet 是远程登录协议**，工作在应用层，依赖 TCP（练习5/8/16/17，答案A/D/D/D）。
8. ⚠ **WWW 三部分：客户机、服务器、HTTP 协议**；FTP 不是 Web 工作的部分（练习18，答案D）。
9. ⚠ **HTTP 是超文本传输协议**，端口号 80，用于浏览器与 Web 服务器之间通信（练习1/10/13/19，答案C/C/C/A）。
10. ⚠ **域名层次结构：从右到左越来越小，分多层排列**（练习9，答案D）。
11. ⚠ **URL 组成：协议://主机名/目录或文件名**（练习12，答案D——用户访问 Web 服务器需要知道 URL 地址）。
12. ⚠ **DHCP Discovery 是广播包**，源 `0.0.0.0`，目的 `255.255.255.255`；租约过半发送 DHCP Renew 续约（练习）。
13. ⚠ **应用层协议判断**：FTP、Telnet、SMTP、POP3、HTTP、DNS、DHCP 均为应用层协议；IP、TCP、UDP、ARP 不是应用层协议（练习2/16，答案B/D）。

## 六、一句话记忆要点

1. **DNS**：域名→IP，UDP 传输，层次结构从右到左越来越小，递归+迭代减轻根服务器负担。
2. **FTP**：TCP，端口 21 控制 / 20 数据，匿名 anonymous，先获副本再修改。
3. **TFTP**：UDP，512 字节/块，编号从 1 开始，无交互无身份鉴别，简单首部。
4. **Telnet**：远程登录协议，TCP，应用层，本地键盘→远程主机。
5. **SMTP/POP3**：SMTP 发邮件端口 25，POP3 收邮件端口 110，均 TCP，邮件地址 `登录名@主机名.域名`。
6. **WWW**：客户机+服务器+HTTP，URL=`协议://主机名/目录或文件名`，HTTP 端口 80，HTML 写网页。
7. **DHCP**：Discovery 广播→Offer→Request 广播→ACK；租约过半 Renew 续约；NAK 后重新 Discovery。
