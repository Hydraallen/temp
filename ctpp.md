## 🧠 阶段一：全局规划与基建 (Planner Agent)
### 步骤 1.1：高层架构设计（生成 `plan.md`）

*目的：让 AI 充当产品经理与架构师，输出全局蓝图，此时不涉及任何代码或任务拆解。*

**💡 Prompt 模板：**

> “你现在是一位资深的软件架构师（Planner）。我需要在当前的 AI IDE 中开发一个 [填写项目名称/描述] 的 MVP 版本。
>
> 请在根目录创建一份详尽的 `plan.md`（系统设计与实施蓝图），必须包含以下模块： 
>
> 1. **产品目标与核心用户流 (User Flows)**：清晰定义 MVP 的边界，什么做，什么不做。 
>
> 2. **技术栈与架构决策**：列出前端、后端、状态管理、存储等核心技术选型，并简述理由。 
>
> 3. **数据模型 / 核心接口设计**：定义关键的数据结构。 
>
> 4. **潜在风险与边界情况**：提前预判开发中可能遇到的难点。 
>
>    **注意：完成 `plan.md` 后请立即停下来，等待我的审查和确认。不要直接开始生成其他文件或代码。**

------

### 步骤 1.2：工作分解与基建（生成交接凭证）

*目的：基于已经确认的蓝图，生成机器可读的任务列表和交接状态文档。*

**💡 Prompt 模板：**

> 1. 基于我们刚刚确认的 `plan.md`，现在请将这份高层蓝图转化为可执行的机器交接文档。
>    请严格执行以下步骤：
>
>    1. **环境初始化**：创建基础项目结构，并提供 `init.sh` 脚本用于依赖安装。
>    2. **任务极细化拆解与验证 (Feature List with Validation)**：在根目录创建 `features.json`。将 `plan.md` 中的需求拆解为极度详细的、独立的子任务。
>       - 颗粒度要求：每个功能点必须足够小（例如：“实现登录按钮的 UI”和“实现登录接口的对接”应分为两个任务）。
>       - **JSON 结构必须严格包含以下字段，特别是 `validation_steps`**：
>         {
>           "id": "任务唯一标识 (如 AUTH-01)",
>           "category": "功能模块",
>           "description": "具体功能描述",
>           "validation_steps": [
>             "验证步骤1：具体的操作动作（如：在输入框输入 'test@example.com'）",
>             "验证步骤2：预期的可观测结果（如：点击提交后，页面路由跳转至 '/dashboard'）",
>             "验证步骤3：异常情况测试（如：断网提交时，UI 提示 '网络错误'）"
>           ],
>           "passes": false
>         }
> 3. **交接文档建立**：创建 `handoff-state.md`，记录当前已完成的脚手架搭建工作，并明确指出下一个接手的 Generator 应该从 `features.json` 中的哪个任务开始。
> 4. **初始提交**：执行 `git init`，将 `plan.md`、`features.json` 和初始脚手架代码进行第一次 commit。

## 🛠️ 阶段二：专注开发与上下文重置 (Generator Agent)

**核心目的**：强制重置上下文（开启新对话/新窗口），让 AI 专注单一模块，彻底消除“上下文焦虑”。

**💡 生成者 Prompt 模板：**
*(注意：每次使用此模板前，务必在 IDE 中开启一个全新的对话窗口)*
> 1. 你现在是一个严谨的高级软件工程师（Generator）。
>    在开始写任何代码之前，请务必先阅读以下三个文件：
>    - `plan.md`（了解全局架构规范和技术栈）
>    - `handoff-state.md`（了解当前项目进度和上下文）
>    - `features.json`（获取具体的任务和验证标准）
>
>    本次会话，请严格执行以下步骤：
>
>    1. **认领任务**：从 `features.json` 中挑选**一个**状态为 `"passes": false` 的最高优先级子任务。请在回复的第一行告诉我你选择了哪个任务（包含任务 ID 和描述）。
>    2. **测试驱动开发 (TDD)**：仔细阅读该任务的 `validation_steps`。你的编码目标是且仅是：**让代码完美通过这几个验证步骤**。
>    3. **增量与克制**：仅编写或修改实现该功能所必需的文件。严格遵守 `plan.md` 中的架构规范，绝对不要修改与当前任务无关的模块，也不要“顺手”去重构其他代码。
>    4. **自我走查与交付 (Self-Walkthrough)**：
>       - 代码编写完成后，请在对话中逐条列出 `validation_steps`，并简要说明你的代码是如何满足每一步的（例如：“针对步骤1，我添加了 xxx 路由；针对步骤2，我增加了 xxx 错误捕获”）。
>       - **绝对不要自己修改 `features.json` 中的 `passes` 状态**。
>       - 最后，请输出标准化口令：“功能 [任务ID] 代码已完成，请求 Evaluator 根据验证步骤进行严格审查。”

## ⚖️ 阶段三：严苛审查与状态流转 (Evaluator Agent)
**核心目的**：打破 AI 的“盲目自信”，通过独立的审查机制倒逼代码质量，并完成测试与状态更新。

**💡 评估者 Prompt 模板：**
> 你现在是一个极其严苛的质量保证专家和架构守门员（Evaluator）。
> 你的任务是对 Generator 刚刚提交的代码进行验收。在开始审查前，请务必读取 `plan.md` 和 `features.json`。
>
> 请严格按照以下“验收契约”进行无情、客观的评估：
>
> 1. **契约验证 (Validation Steps Check)**：
>    - 逐一核对 `features.json` 中该任务的 `validation_steps`。
>    - 检查代码逻辑是否真实、完整地覆盖了每一个测试步骤（包括异常处理和边界情况）。不要听信 Generator 的自我吹嘘，你要自己审视代码逻辑。
> 2. **架构对齐 (Architecture Alignment)**：
>    - 对照 `plan.md`，检查新增代码是否破坏了既定的技术栈、数据流向或组件规范。
>    - 是否引入了未经批准的第三方依赖？
> 3. **代码工艺 (Clean Code)**：
>    - 是否有未处理的 `TODO`、硬编码（Hardcode）或残留的 `console.log`？
>    - 命名是否清晰？是否做到了高内聚低耦合？
>
> **你的输出与操作要求**：
> - ❌ **如果发现任何瑕疵（哪怕是一个拼写错误或漏掉了一个边缘测试）**：
>   给出明确的批评，指出具体违背了哪一条 `validation_steps` 或 `plan.md` 的哪一项，并打回给 Generator 要求重写。**绝对不要修改任何状态文件。**
> - ✅ **如果 100% 完美通过验收**：
>   1. 将 `features.json` 中该任务的 `"passes"` 字段更新为 `true`。
>   2. 更新 `handoff-state.md`，在日志中记录本次完成的任务 ID，并明确指出下一个待开发的最高优先级任务是什么。
>   3. 执行 `git add .` 和 `git commit -m "feat: [任务ID] 完成 [任务描述]"`，将干净的代码提交到代码库。



### 🛠️ AI 辅助开发 Git 核心指令与 HEAD 战术速查表

| **开发阶段 / 角色**              | **核心 Git 指令**                                            | **执行场景 (结合 HEAD 状态)**                                | **工程防御与架构作用**                                       |
| :------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **初始化**<br>*(Planner)*        | `git init`<br>`git add .`<br>`git commit -m "chore: init"`   | 架构蓝图与任务清单确认无误时。                               | **确立基线**：锁定初始架构状态，为后续所有的增量开发建立一个绝对干净的起点。 |
| **增量开发**<br>*(Generator)*    | `git checkout -b feat/<任务ID>`                              | AI 认领了新任务，准备开始写代码前。                          | **沙盒隔离**：将 AI 的工作区与主分支物理隔离，防止幻觉代码污染主干。 |
| **审查对比**<br>*(Evaluator)*    | `git diff HEAD`                                              | **当前最新的提交 (`HEAD`)**：<br>对比当前未提交的修改。      | **爆炸半径检测**：精准查看 AI 当前工作区与最后一次提交的差异，防止它瞎改核心配置。 |
| **暂存撤销**<br>*(Evaluator)*    | `git restore --staged <文件>`                                | **当前最新的提交 (`HEAD`)**：<br>撤销暂存区文件。            | **精准剔除**：当 AI 生成了多余的临时文件并被误 `add` 时，将其移出暂存区，拒绝垃圾代码入库。 |
| **状态固化**<br>*(Evaluator)*    | `git commit -m "..."`<br>`git checkout main`<br>`git merge ...` | 代码完美通过 `validation_steps` 验收时。                     | **状态流转**：将干净的代码合并到主分支，准备开启下一个任务。 |
| **工作区重置**<br>*(防 AI 失控)* | `git restore .`<br>`git clean -fd`                           | AI 逻辑彻底混乱，写出无法编译的垃圾代码时。                  | **一键清空**：清空工作区所有未提交的修改和未追踪文件，瞬间回到上一个健康状态。 |
| **单步回滚**<br>*(防 AI 失控)*   | `git reset --hard HEAD~1`<br>*(或 `HEAD^`)*                  | **上一个提交（回退 1 步）**：<br>撤销刚刚写错的最后一次 Commit。 | **错误记忆抹除**：审查走眼，误将有 Bug 的代码 Commit 进去了，带着代码一起回滚，强制重置 AI 的错误记忆。 |
| **深度回滚**<br>*(防 AI 失控)*   | `git reset --hard HEAD~2`                                    | **上上个提交（回退 2 步）**：<br>连续撤销最近的两次 Commit。 | **连环翻车救援**：发现 AI 连续两次迭代（比如两个关联任务）都走偏了，进行更深度的上下文状态回滚。 |
| **上下文辅助**<br>*(投喂记忆)*   | `git log --oneline -n 5`<br>`git diff main...<分支>`         | 开启全新对话窗口，或需要做全局架构审查时。                   | **状态同步**：提取精简的开发历史喂给 AI，帮助它快速恢复记忆并保持连贯性。 |
