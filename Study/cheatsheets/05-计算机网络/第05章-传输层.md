## 一、传输层概述

**传输层定位**：传输层是 OSI 中最重要、最关键的一层，是唯一负责总体的数据传输和数据控制，提供端到端的交换数据机制，对会话层等高三层提供可靠的传输服务，对网络层提供可靠的目的地站点信息。

**传输层功能**：
- 网络层协议只提供点到点的连接，而传输层协议提供端到端的服务，即应用进程之间的通信。
- 网络层协议提供不可靠、无连接和尽力投递的服务，因此对于可靠性要求很高的上层协议，需要在传输层实现可靠性的保障。

**传输层两大协议**：
- **TCP**（Transmission Control Protocol）：可靠的、面向连接的协议，允许网络间两台主机之间无差错的信息传输；TCP 协议还进行流量控制，以避免发送过快而发生拥塞；这一切对用户都是透明的。
- **UDP**（User Datagram Protocol）：采用无连接的方式传送数据，发送端不关心发送的数据是否到达目标主机、数据是否出错等；收到数据的主机也不会告诉发送方是否收到了数据，其可靠性由上层协议来保障。

**TCP vs UDP 适用场景**：
- TCP：效率较低但可靠性高，适合网络链路不好或可靠性要求高的环境。
- UDP：面向非连接、不可靠，但不用传送许多与数据本身无关的信息，效率较高，常用于实时业务及对差错不敏感的应用。

> Caveat (课后习题5): UDP 数据传输的可靠性由应用层负责，不是传输层、数据链路层或网络层。

> Caveat (课后习题8): OSI 参考模型中，提供端到端的透明数据传输服务、差错控制和流量控制的层是传输层。

## 二、TCP 协议

### 1. TCP 协议总述 ⭐

TCP 是为了在主机间实现高可靠性数据交换的传输协议，是面向连接的端到端的可靠协议。TCP 对下层服务没有多少要求，假定下层只能提供不可靠的数据包，可在多种硬件构成的网络上运行。

TCP 为实现可靠传输提供的方法和手段：
- **数据段编号**：保证数据正确顺序（IP 数据包可能失序到达，TCP 对数据段编号并重新排序后交给应用层）。
- **丢弃重复数据**：IP 数据包可能重复，TCP 接收端必须具有丢弃重复数据的功能。
- **流量控制**：TCP 连接的每一方都有固定大小的缓冲空间，发送端只发送接收端缓冲区能接收的数据，防止较快主机导致较慢主机缓冲区溢出。

TCP 被认为是一种**流式传输层服务**：发送端从应用程序接收字符流，提取适当长度创建数据段发送到网络上；接收端接收数据段，提取数据，若没有按序号到达还要排序，并作为字符流交付给接收端应用程序。

为了进行流式交付，发送 TCP 和接收 TCP 都要利用缓冲。发送 TCP 使用发送缓冲来存储从发送应用程序截取的数据。

### 2. TCP 封装格式（首部结构） ⭐⚠

TCP 首部通常 20 字节（不计任选字段），各字段如下：

| 字段 | 位数 | 说明 |
|---|---|---|
| 源端口地址 | 16 位 | 发送方进程对应的端口号，由发送方进程产生的随机数，唯一标识发送端的一个进程 |
| 目的端口地址 | 16 位 | 接收端的进程端口号，接收端据此确定把数据递送给哪个应用程序 |
| 序列号 | 32 位 | 提供 $0 \sim 2^{32}-1$ 范围内的数字，给数据段打标记，接收端按序列号重新排列保证数据正确性 |
| 确认号 | 32 位 | 提供 $0 \sim 2^{32}-1$ 范围内的数字，告诉发送端该序号之前的数据段都收到了（如确认号为 $X$，表示前 $X-1$ 个数据段都收到了） |
| 首部长度 | 4 位 | 确定首部字节长度；4 位最大值 1111=15，表示 15 行，每行 32 位=4 字节，故首部最大 $15 \times 4 = 60$ 字节 |
| 保留 | 6 位 | 保留作今后扩展功能用 |
| URG | 1 位 | 紧急指针有效位，URG=1 时与紧急指针配合使用，确定紧急数据最后一个字节的位置 |
| ACK | 1 位 | ACK=1 时确认序列号字段才有效；ACK=0 时确认号无效 |
| PSH | 1 位 | PSH=1 时要求接收方尽快将数据段送达应用层，加快特殊数据的处理速度 |
| RST | 1 位 | RST=1 时通知重新建立 TCP 连接 |
| SYN | 1 位 | 同步序号位，TCP 建立连接时置为 1 |
| FIN | 1 位 | 发端完成发送任务位，需要断开连接时提出断开的一方置为 1 |
| 窗口大小 | 16 位 | 说明本地可接收数据段的数目，值可变；网络通畅时变大加快传输，不稳定时减小保证可靠传输；TCP 流量控制依靠变化窗口大小实现 |
| 校验和 | 16 位 | 差错控制；发送端计算所有字节的校验和，到达目的地再计算一次，一致则数据基本正确，否则认为被破坏并抛弃 |
| 紧急指针 | 16 位 | 与 URG 配合使用，URG=1 时有效，说明紧急数据的末尾字节位置 |
| 选项与填充 | 可变 | 只有当首部长度大于 20 时才有效 |

**TCP 连接的唯一定位**：一个 TCP 连接不仅需要端口，还需要 IP 地址来确定通信的主机。发送端 IP 地址 + 发送端端口号 = 连接的发送端；目标端 IP 地址 + 接收端端口号 = 连接的接收端。这样就唯一地确定了一个 TCP 连接。

> Caveat (例题1): TCP 套接字通过四元组（源 IP 地址，源端口号，目的 IP 地址，目的端口号）来标识。当 TCP 报文段从网络到达一台主机时，主机使用全部四个值来将报文段定向（多路分解）到相应的套接字。

### 3. TCP 的连接与断开 ⭐⚠

TCP 是面向连接的服务，在数据通信之前，发送端与接收端要先建立连接；数据发送结束后，双方再断开连接。

**TCP 建立连接——三次握手**：

1. **第 1 次握手**：主机 A 发送含有 SYN 标志位的数据段给主机 B，告知 B 两点：A 希望建立连接请求，B 应答；A 告诉 B 使用哪个序列号作为数据传输时数据段的起始号。
   - 例：`seq=100, ctl=SYN`
2. **第 2 次握手**：主机 B 发送含有 ACK 和 SYN 标志位的数据段响应 A，两个目的：发送 ACK 通知 A 收到了数据段；通知 A 从哪个序列号开始给数据段做标记。
   - 例：`seq=300, ack=101, ctl=SYN, ACK`
3. **第 3 次握手**：主机 A 再次发送数据段，确认收到了 B 的数据段，并可以开始传送实际数据。
   - 例：`seq=101, ack=301, ctl=ACK`

三次握手特点：没有应用层数据；SYN 标志位只有 TCP 建立连接时才被置为 1，握手完成后 SYN 标志位被置为 0。

**TCP 断开连接——四次挥手**：

TCP 终止连接要经过 4 次，由 TCP 的**半关闭**（half-close）造成。TCP 连接是全双工的（数据可在两个方向上同时传递），关闭时每个方向必须单独关闭，单方向的关闭称为半关闭。关闭的方法是一方完成数据发送任务后，发送一个 FIN 来通告将要终止这个方向连接。

1. **第 1 步**：A 将控制位 FIN 置为 1，提出停止 TCP 连接请求。
   - 例：`seq=101, ack=301, ctl=FIN, ACK`
2. **第 2 步**：B 收到 FIN 后做出响应，确认这一方向上的 TCP 连接将关闭。
   - 例：`seq=301, ack=102, ctl=ACK`
3. **第 3 步**：B 提出反方向的关闭要求，将 FIN 置为 1。
   - 例：`seq=301, ack=102, ctl=FIN, ACK`
4. **第 4 步**：A 对 B 提出的关闭做出应答，双方方向的关闭结束。
   - 例：`seq=102, ack=302, ctl=ACK`

> Caveat (课后习题3): TCP 传输连接的释放过程中，客户端和服务器端都可以主动提出释放连接的请求，并非只有客户端可以。

> Caveat (Ch5-判10): TCP 释放连接使用四次挥手，不是三次握手。三次握手是连接建立。

### 4. TCP 的流控机制——滑动窗口 ⭐

为解决发送与接收双方数据通信能力差异带来的数据丢失问题，TCP 引入流量控制机制，使用**滑动窗口**来实现。

滑动窗口工作原理：
- TCP 建立连接时进行窗口大小协商，主机 A 在发送连接建立请求时通告自己发送窗口的大小，单位是字节。
- 接收方根据缓冲区大小宣告窗口大小。若缓冲区大小为 0，B 向 A 宣告窗口大小为 0，A 接收到后停止发送，等待 B 宣告一个非 0 窗口时再发送数据。
- 应用程序读取数据后，缓冲区清空空间，接收方据此调整宣告的窗口大小。

### 5. TCP 的拥塞控制 ⭐

接收方可根据缓冲区大小指定窗口大小，但由于中间传输网络带宽较小可能导致拥塞。TCP 引入**拥塞窗口**（cwnd）的概念。

**实际发送窗口**：

$$\text{实际发送窗口} = \min(\text{接收方允许的窗口},\ \text{拥塞窗口 cwnd})$$

发送方维护两个窗口：接收方允许的窗口、拥塞窗口（cwnd），两者中的最小值即为发送方可以发送的窗口大小。

拥塞控制过程示例：
- 主机 A 与 B 协商窗口大小为 3，A 发送 3 个数据段。
- B 只接收到一个数据段，发送确认号为 102 的确认。
- 在重传计时器超时前，A 没有收到来自 B 的确认，A 认为数据段 102 和 103 丢失。
- A 根据接收到的确认将拥塞窗口 cwnd 减小。

> Caveat (课后习题2): TCP 中发送方的窗口大小取决于接收方允许的窗口和拥塞窗口。

### 6. TCP 的差错控制 ⭐

TCP 差错控制包括：检测受损数据段、丢失的数据段、失序的数据段和重复的数据段，以及检测出差错后的纠错机制。

差错检验通过 3 种方式完成：

| 方式 | 说明 |
|---|---|
| 校验和 | 每个数据段包含校验和字段，检测受损数据段；若受损伤则由目的 TCP 将其丢弃 |
| 确认 | TCP 使用确认方式证实收到了某些无损伤到达的数据段 |
| 超时 | 若一个报文在超时前未被确认，则被认为受到损伤或已丢失 |

**5 种差错情况处理**：

| 差错类型 | 处理方式 |
|---|---|
| 受损数据段 | 目标站点丢弃，不认为已收到，请求重新发送 |
| 丢失的数据段 | 与受损数据段情况一样（被中间节点丢弃），TCP 不认为收到过 |
| 重复的数据段 | 目的 TCP 期望连续字节流，含同样序号的分组到达时丢弃重复数据段 |
| 失序的数据段 | 对失序的数据段不确认，直到收到所有它以前的数据段为止；若确认晚了，源 TCP 计时器到期会重发该数据段 |
| 丢失的确认 | TCP 使用累计确认系统，每一个确认证实由确认号指明的字节之前的所有字节都已收到；后一个确认到达后前一个丢失的确认变得没有意义 |

> Caveat (例题2): TCP 快速重传——当接收方检测到数据流中出现缺失，对收到的最后一个有序字节给出重复确认。如果 TCP 发送方收到同一数据的**三个重复 ACK**，就表示该数据段后面的数据已丢失，此时 TCP 执行快速重传。

### 7. TCP 的计时器 ⭐

TCP 协议设计了 4 种计时器：

| 计时器 | 功能 | 关键参数 |
|---|---|---|
| 重传计时器 | 控制丢失或丢弃的数据段；发送数据段时创建，收到确认则撤销，超时则重传并复位 | 重传时间 $= 2 \times \text{RTT}$ |
| 坚持计时器 | 解开窗口大小为零时的死锁；发送端收到零窗口确认时启动，到期发送探测数据段（1 字节） | 初始值为重传时间值，未收到响应则加倍，极限值 60 秒，之后每隔 60 秒发送探测数据段 |
| 保活计时器 | 防止两个 TCP 之间连接长时间空闲；服务器收到客户信息时复位 | 超时通常设为 2 小时；2 小时后发送探测数据段，每 75 秒一个，发 10 个无响应则中止连接 |
| 时间等待计时器 | 断开连接时，A 发送最后一个 ACK 后不立即关闭，经过时间等待计时器的时间再关闭 | 保证 A 能收到重复的 FIN 数据段 |

**坚持计时器详解**：在 TCP 中对确认是不需要确认的。若接收端宣布窗口大小为零，发送端停止传送；若接收端的确认丢失，双方进入死锁等待。坚持计时器解开此死锁：到期后发送端发送一个探测数据段（只有一个字节的数据，序号永远不需要确认），提醒接收端重传确认。

**时间等待计时器详解**：TCP 断开连接时，如果 A 发送完最后一个 ACK 后立即关闭连接，而此时 ACK 丢失，B 无法判断是 FIN 丢失还是 ACK 丢失，会重传 FIN。A 设置时间等待计时器，在发送最后一个 ACK 后不立即关闭，经过该计时器时间再关闭，保证能收到重复的 FIN。

### 8. TCP 的应用与常用端口 ⭐

TCP 主要用在数据传输可靠性要求高的环境，如远程登录程序 TELNET（客户端发出的每个命令都要在远程主机上执行，需要可靠传输）、网页浏览 HTTP。

| 端口 | 协议 | 说明 |
|---|---|---|
| 21 | FTP | 文件传输协议，用于上传、下载 |
| 23 | Telnet | 远程登录，通过连接目标计算机的这一端口，得到验证后可以远程控制管理目标计算机 |
| 25 | SMTP | 简单邮件传输协议，用于发送邮件 |
| 53 | DNS | 域名服务，将网站名称解析成 IP 地址 |
| 80 | HTTP | 超文本传输协议，实现网络上超文本的传输 |


> Caveat (课后习题12): FTP 端口是 21，不是 24。

## 三、UDP 协议

### 1. UDP 协议总述 ⭐

UDP 是一个简单的面向数据包的传输层协议，进程的每个输出操作都恰好产生一个 UDP 数据段，并组装成一份待发送的 IP 数据包。

UDP 不提供可靠性：把应用程序传给 IP 层的数据发送出去，但不保证能到达目的地。

UDP 的优势：在小数据文件传输中发挥重要作用。例如发送 10 字节短消息时，TCP 至少加 20 字节头部且需要确认，效率低；UDP 首部仅 8 字节且无确认，数据传输率大幅提高。

> Caveat (例题5): UDP 协议是一种不可靠的无连接协议，不提供可靠服务。

> Caveat (课后习题7): 接收端收到有差错的 UDP 用户数据时的处理方式是丢弃。

### 2. UDP 的封装格式（首部结构） ⭐⚠

UDP 首部仅 8 字节，各字段如下：

| 字段 | 位数 | 说明 |
|---|---|---|
| 源端口号 | 16 位 | 标识数据发送端的进程 |
| 目标端口号 | 16 位 | 标识数据接收端的进程 |
| UDP 长度 | 16 位 | 指出 UDP 数据段中所包含数据的大小，可准确计算数据的结束位置 |
| UDP 校验和 | 16 位 | UDP 提供的唯一可靠机制，通过计算校验和判断到达数据的正确性；校验和是可选的 |

> Caveat (课后习题4): UDP 长度字段是 UDP 数据报的长度，**不包括**伪首部的长度。但 UDP 校验和对伪首部、UDP 报文头以及应用层数据进行校验。

> Caveat (课后习题6): UDP 数据报首部不包含"首部长度"字段（因为 UDP 首部固定 8 字节，不需要该字段）。

> Caveat (例题3): UDP 套接字由一个包含目的 IP 地址和目的端口号的二元组来标识。目标主机通过检查报文段中的目的端口号，将报文段定向到相应套接字。

### 3. UDP 的应用与常用端口 ⭐

UDP 协议能高速传输数据，在实时业务和对差错不敏感的应用中有优势（如 QQ 发送短消息使用 UDP 方式）。

| 端口 | 协议 | 说明 |
|---|---|---|
| 69 | TFTP | 简单文件传输协议 |
| 53 | DNS | 域名服务 |
| 123 | NTP | 网络时间协议 |
| 111 | RPC | 远程过程调用 |


**DNS 同时支持 TCP 和 UDP 查询（端口都是 53）**，大多数查询使用 UDP，一般需要 TCP 查询的两种情况：
1. 当查询数据较大以至于产生了数据分段，需要利用 TCP 的分片能力来进行数据传输。
2. 当主服务器和辅服务器之间进行数据同步通信的时候。

**UDP 的局限性**：
- 没有流控机制，来到的报文太多时接收端可能溢出。
- 除校验和外没有差错控制机制，发送端不知道数据是丢失还是重复交付。接收端用校验和检测出差错时，悄悄地将此用户数据丢掉。
- 缺少流控制和差错控制，使用 UDP 的进程必须提供这些机制（如 TFTP 协议提供分块传输、分块确认的机制来保证可靠性）。

> Caveat (课后习题18): UDP 使用应用层协议提供可靠性。

## 四、重点对比与易混辨析

### TCP vs UDP 对比 ⭐

| 对比项 | TCP | UDP |
|---|---|---|
| 连接方式 | 面向连接 | 无连接 |
| 可靠性 | 可靠 | 不可靠 |
| 首部长度 | 通常 20 字节（最大 60 字节） | 固定 8 字节 |
| 流量控制 | 有（滑动窗口） | 无 |
| 差错控制 | 校验和 + 确认 + 超时 | 仅校验和（可选） |
| 拥塞控制 | 有（cwnd） | 无 |
| 传输方式 | 字节流 | 数据报 |
| 效率 | 较低 | 较高 |
| 适用场景 | 可靠性要求高（TELNET、HTTP、FTP、SMTP） | 实时业务、短消息、对差错不敏感（QQ、DNS、TFTP、NTP） |
| 套接字标识 | 四元组（源IP, 源端口, 目的IP, 目的端口） | 二元组（目的IP, 目的端口） |


### TCP 4 种计时器对比

| 计时器 | 触发条件 | 作用 | 关键参数 |
|---|---|---|---|
| 重传计时器 | 发送数据段后启动 | 超时未确认则重传 | 重传时间 $= 2 \times \text{RTT}$ |
| 坚持计时器 | 收到窗口大小为零的确认 | 发送探测数据段解开死锁 | 值加倍至极限 60 秒 |
| 保活计时器 | 服务器收到客户信息时复位 | 防止连接长时间空闲 | 超时 2 小时；10 个探测（每 75 秒）无响应则中止 |
| 时间等待计时器 | 发送最后一个 ACK 后 | 等待一段时间再关闭连接 | 保证收到重复 FIN |


### 端口号分类

| 类别 | 范围 | 说明 |
|---|---|---|
| 熟知端口号 | $0 \sim 1023$ | 为常用应用层协议保留（FTP、HTTP 等），限制使用 |


> Caveat (课后习题19): TCP 和 UDP 分别拥有自己的端口号，二者互不干扰，可以共存于同一台主机。

## 五、高频考点与易错提醒

1. **TCP vs UDP 特性对比**：面向连接/无连接、可靠/不可靠、首部大小（20 vs 8 字节）、是否有流控/差错控制/拥塞控制。 ⭐
2. **TCP 首部各控制位含义**（URG/ACK/PSH/RST/SYN/FIN）及首部长度计算（4 位，最大 $15 \times 4 = 60$ 字节）。 ⭐⚠
3. **三次握手与四次挥手过程**：SYN/ACK/FIN 标志位变化、seq/ack 数值推算、半关闭概念。 ⭐⚠
4. **TCP 滑动窗口与流量控制**：窗口大小可变、零窗口停止发送。 ⭐
5. **TCP 拥塞控制**：实际发送窗口 $= \min(\text{rwnd}, \text{cwnd})$。 ⭐
6. **TCP 差错控制**：5 种差错情况处理、累计确认、三个重复 ACK 触发快速重传。 ⭐
7. **TCP 4 种计时器**：重传/坚持/保活/时间等待，各自触发条件与参数。 ⭐
8. **UDP 首部仅 8 字节**：源端口、目的端口、UDP 长度、UDP 校验和；无首部长度字段。 ⭐⚠
9. **套接字标识**：TCP 四元组 vs UDP 二元组。 ⭐⚠
10. **常用端口号**：TCP（FTP 21、Telnet 23、SMTP 25、DNS 53、HTTP 80）；UDP（TFTP 69、DNS 53、NTP 123、RPC 111）。 ⭐
11. **UDP 校验和包含伪首部，但 UDP 长度字段不包含伪首部**。 ⚠
12. **熟知端口号范围 $0 \sim 1023$**。 ⭐

## 六、一句话记忆要点

- **传输层**：OSI 中唯一端到端、负责总体数据传输和数据控制的一层。
- **TCP**：面向连接、可靠、字节流，首部 20 字节，三次握手建连、四次挥手断连。
- **UDP**：无连接、不可靠、数据报，首部 8 字节，无流控无差错控制，可靠性靠应用层。
- **滑动窗口**：接收方根据缓冲区宣告窗口大小，发送方据此控制发送速率。
- **拥塞控制**：实际发送窗口 $= \min(\text{rwnd}, \text{cwnd})$。
- **差错控制**：校验和 + 确认 + 超时；三个重复 ACK 触发快速重传。
- **4 种计时器**：重传（$2 \times \text{RTT}$）、坚持（解死锁，极限 60 秒）、保活（2 小时）、时间等待（等重复 FIN）。
- **套接字**：TCP 四元组，UDP 二元组。
- **端口号**：熟知端口 $0 \sim 1023$；TCP 和 UDP 端口独立共存。

## Worked Examples

### 例 1：TCP 序号与确认号推算 *(来自 exam-signals 候选 4)*
*Tags: TCP seq/ack, 确认机制*
**问题**：A 向 B 发送 TCP 报文段，seq=300，ack=101，数据 7 字节，求 B 确认报文段的 seq 和 ack。
**解答**：`[PROPOSED — verify]`
1. B 的确认号 ack = A 的 seq + 数据长度 = $300 + 7 = 307$
2. B 的序号 seq = A 期望收到的下一个字节 = A 的 ack 值 = $101$
3. 注意：确认号表示"期望收到的下一个字节序号"
4. 选 D：seq=101, ack=307

### 例 2：TCP 套接字定向 *(来自本章例题1)*
*Tags: 套接字, TCP*
**问题**：当一个 TCP 报文段到达目的主机时，通过什么来定向套接字？
**解答**：
1. TCP 套接字通过四元组标识：(源 IP 地址, 源端口号, 目的 IP 地址, 目的端口号)
2. 当 TCP 报文段从网络到达一台主机时，主机使用全部四个值来将报文段定向（多路分解）到相应的套接字
3. 选 D：以上所有

### 例 3：UDP 套接字定向 *(来自本章例题3)*
*Tags: 套接字, UDP*
**问题**：当一个 UDP 报文段到达某个主机时，为了将报文段提交给合适的套接字，操作系统使用什么？
**解答**：
1. UDP 套接字由一个包含目的 IP 地址和目的端口号的二元组来标识
2. 在多路分解时，目标主机通过检查报文段中的目的端口号，将报文段定向到相应套接字
3. 选 C：目的端口号

### 例 4：TCP 快速重传 *(来自本章例题2)*
*Tags: 快速重传, 差错控制*
**问题**：TCP 什么时候对报文段采用快速重传？
**解答**：
1. 当接收方检测到数据流中出现缺失，对收到的最后一个有序字节给出重复确认
2. 如果 TCP 发送方收到同一数据的三个重复确认，表示该数据段后面的数据已丢失
3. 此时 TCP 执行快速重传
4. 选 C：收到之前发出的一个报文段的三个重复 ACK
