## 一、本章概览

本章解决的核心问题是：关系数据库管理系统（RDBMS）如何把用户提交的 SQL 查询语句转换为高效的查询执行计划。整章围绕"查询处理的四个阶段"展开——查询分析 → 查询检查 → 查询优化 → 查询执行，其中查询优化是重点，分为代数优化（改变操作次序与组合）和物理优化（选择存取路径与底层算法）。启发式规则与代价估算是两种优化思路，查询计划的执行方式（自顶向下 / 自底向上）收束全章。本章与第二章关系代数、第四章 SQL 紧密关联，是数据库内部实现的核心内容。

> **注**：OCR 中知识点 9–15 的正文内容缺失（物理页 p47 在"知识点9：选择操作的实现"标题后即截断，p48 直接从知识点16 开始）。以下仅覆盖 OCR 中可读的知识点 1–8、16–20；缺失部分在"高频考点与易错提醒"中依据练习题答案解析给出考点方向提示，但不作为正文来源。

## 二、查询处理概述

**查询处理** ⭐：关系数据库管理系统执行查询语句的过程，其任务是把用户提交给关系数据库管理系统的查询语句转换为高效的查询执行计划。*(讲义 p47)*

**查询处理的步骤** ⭐：可分为 4 个阶段——查询分析、查询检查、查询优化和查询执行。*(讲义 p47)*

**查询处理流程**（全流程图）：*(讲义 p47)*

```
查询语句
  │
  ▼
┌─────────── 查询分析 ───────────┐
│  词法分析                        │
│  语法分析                        │
└─────────────────────────────────┘
  │
  ▼
┌─────────── 查询检查 ───────────┐  ← 数据字典 / 数据库
│  语义分析                        │
│  符号名转换                      │
│  安全性检查                      │
│  完整性检查                      │
└─────────────────────────────────┘
  │
  ▼
┌─────────── 查询优化 ───────────┐
│  查询树                          │
│    ├─ 代数优化                   │
│    └─ 物理优化                   │
└─────────────────────────────────┘
  │
  ▼
┌─────────── 查询执行 ───────────┐
│  查询执行计划                    │
│  代码生成                        │
│  查询计划的执行代码              │
└─────────────────────────────────┘
  │
  ▼
查询结果
```

## 三、查询处理四阶段详解

### 1. 查询分析 ⭐

**查询分析**：扫描查询语句识别出语言符号，进行语法检查和语法分析。如果没有语法错误就转入下一步处理。*(讲义 p47)*

> Caveat: 查询分析不仅要完成语法检查，还要进行语法分析，两者缺一不可才能进入下一步。*(题目第五章-判断-1)*

### 2. 查询检查 ⭐

**查询检查**：对合法的查询语句进行语义检查，即根据数据字典中有关的模式定义检查语句中的数据库对象。*(讲义 p47)*

查询检查包含以下工作：*(讲义 p47)*

- **视图操作转换**：如果是对视图的操作，则要用视图消解方法把对视图的操作转换成对基本表的操作。
- **权限与完整性检查**：根据数据字典中的用户权限和完整性约束定义对用户的存取权限进行检查。
- **内部表示转换**：把 SQL 查询语句转换成内部表示，即等价的关系代数表达式。
- **名称转换**：把数据库对象的外部名称转换为内部表示。

> Caveat: 视图操作转换使用的是"视图消解"方法，不是"视图简化"或"逻辑消解"。*(题目第五章-单选-4)*
>
> Caveat: SQL 查询语句转换成的内部表示是等价的"关系代数表达式"，不是关系演算。*(题目第五章-单选-5)*

### 3. 查询树 ⭐

**查询树**：关系数据库管理系统一般用查询树（也称为语法分析树）来表示扩展的关系代数表达式。*(讲义 p47)*

> Caveat: 查询树也称为语法分析树，二者是同一概念。*(题目第五章-判断-2)*

### 4. 查询优化 ⭐

**查询优化**：选择一个高效执行的查询处理策略。*(讲义 p47)*

**查询优化方法** ⭐：按照优化的层次一般分为代数优化和物理优化。*(讲义 p47)*

| 维度 | 代数优化 | 物理优化 |
|---|---|---|
| **优化对象** | 关系代数表达式 | 存取路径和底层操作算法 |
| **优化手段** | 按照一定规则对关系代数表达式进行等价变换，改变操作的次序和组合 | 选择存取路径和底层操作算法 |
| **目标** | 使查询执行更高效 | 求得优化的查询计划 |
| **选择依据** | 等价变换规则 | 基于规则 / 基于代价 / 基于语义 |

*(讲义 p47)*

> Caveat: 等价变换属于代数优化方法，不是物理优化方法。*(题目第五章-判断-3)*
>
> Caveat: 物理优化是指存取路径和底层操作算法的选择，不是"磁盘优化"或"文件结构"。*(题目第五章-单选-7)*

### 5. 查询执行 ⭐

**查询执行**：依据优化器得到的执行策略生成查询执行计划，由代码生成器生成执行这个查询计划的代码，然后加以执行，回送查询结果。*(讲义 p47)*

> Caveat: 查询执行时由"代码生成器"生成执行代码，不是由用户程序生成。*(题目第五章-判断-4)*

## 四、代数优化与启发式规则

### 查询树的启发式优化 ⭐⚠

**查询树的启发式优化**：应用启发式规则的代数优化，是对关系代数表达式的查询树进行优化的方法。*(讲义 p48)*

**典型的启发式规则**（5 条）：*(讲义 p48)*

| 序号 | 启发式规则 | 要点说明 |
|---|---|---|
| 1 | 选择运算应尽可能先做 | 选择尽早执行可大幅减小中间结果 |
| 2 | 把投影运算和选择运算同时进行 | 避免重复扫描关系 |
| 3 | 把投影同其前或其后的双目运算结合起来 | 没有必要为了去掉某些字段而扫描一遍关系 |
| 4 | 把某些选择同在它前面要执行的笛卡儿积结合起来成为一个连接运算 | 连接（特别是等值连接）要比同样关系上的笛卡儿积省很多时间 |
| 5 | 找出公共子表达式 | 避免重复计算 |

> Caveat: 规则 1 是"选择运算"尽可能先做，不是"投影运算"先做——投影先做不在启发式规则中。*(题目第五章-多选-8)*
>
> Caveat: 规则 3 是把投影同其前或其后的"双目运算"结合起来，不是单目运算。*(题目第五章-单选-15)*
>
> Caveat: 投影与选择同时做的目的是"避免重复扫描关系"，不是改变中间结果大小或节约磁盘空间。*(题目第五章-单选-19)*

## 五、物理优化

### 1. 物理优化概述 ⭐

**物理优化**：选择高效合理的操作算法或存取路径，求得优化的查询计划，达到查询优化的目标。*(讲义 p49)*

选择方法可以是：*(讲义 p49)*

- 基于规则的启发式优化
- 基于代价估算的优化
- 两者结合的优化方法

### 2. 基于启发式规则 ⭐

#### 2.1 选择操作的启发式规则

对于**小关系**，使用全表顺序扫描，即使选择列上有索引。*(讲义 p49)*

对于**大关系**，启发式规则如下：*(讲义 p49)*

| 序号 | 选择条件类型 | 启发式规则 |
|---|---|---|
| 1 | 主码 = 值 | 查询结果最多一个元组，选择主码索引（DBMS 一般自动建立主码索引） |
| 2 | 非主属性 = 值，选择列上有索引 | 估算查询结果元组数目，比例较小（<10%）用索引扫描，否则全表顺序扫描 |
| 3 | 属性上的非等值查询或范围查询，选择列上有索引 | 同上，选择率 <10% 用索引扫描，否则全表顺序扫描 |
| 4 | AND 连接的合取选择条件 | 有组合索引则优先采用组合索引扫描；某些属性有索引可用索引扫描，否则全表顺序扫描 |
| 5 | OR 连接的析取选择条件 | 一般使用全表顺序扫描 |

> Caveat: 对于大关系，选择条件是"非主属性=值"且选择列有索引时，需要估算查询结果元组数目，比例 <10% 才用索引扫描。*(题目第五章-单选-16)*

#### 2.2 连接操作的启发式规则 ⭐

| 序号 | 适用条件 | 选用算法 |
|---|---|---|
| 1 | 2 个表都已按照连接属性排序 | 排序-合并算法 |
| 2 | 一个表在连接属性上有索引 | 索引连接算法 |
| 3 | 上面 2 条都不适用，其中一个表较小 | Hash Join 算法 |
| 4 | 最后兜底 | 嵌套循环算法，选较小的表（占用的块数 B 较少的表）作为外表 |

*(讲义 p49)*

> Caveat: 如果两个表都已经按照连接属性排序，则选用排序-合并算法，不是嵌套循环或 Hash Join。*(题目第五章-单选-17)*
>
> Caveat: 嵌套循环算法应选择占用的块数（B）较少的表作为外表（外循环的表），不是元组数较少的表。*(讲义 p49)*

### 3. 基于代价估算的优化 ⭐

**基于代价估算的优化**：基于代价的优化方法要计算各种操作算法的执行代价，它与数据库的状态密切相关。为此在数据字典中存储了优化器需要的统计信息。*(讲义 p49)*

**统计信息主要包括三个方面**：*(讲义 p49)*

| 统计对象 | 统计信息项 | 符号 |
|---|---|---|
| **每个基本表** | 元组总数 | $N$ |
| | 元组长度 | $L$ |
| | 占用的块数 | $B$ |
| | 占用的溢出块数 | $BO$ |
| **基本表的每个列** | 不同值的个数 | $m$ |
| | 最大值、最小值 | — |
| | 是否已建立索引及索引类型（B+树索引、hash 索引、聚集索引） | — |
| **索引（如 B+树索引）** | 索引的层数 | $Z$ |
| | 不同索引值的个数 | — |
| | 索引的选择基数（有 $S$ 个元组具有某个索引值） | $S$ |
| | 索引的叶结点数 | $Y$ |

*(讲义 p49)*

> Caveat: 优化器可以从数据字典中获取统计信息做出正确估算，而用户程序难以获得数据字典信息。*(题目第五章-判断-7)*
>
> Caveat: 数据库的物理统计信息改变时，系统可以自动对查询进行重新优化以选择相适应的执行计划；优化器可考虑数百种执行计划，程序员一般只能考虑有限的几种。*(题目第五章-多选-6)*

## 六、查询计划的执行

**查询计划的执行** ⭐：查询优化完成后，关系管理系统为用户查询生成了一个查询计划。该查询计划的执行可以分为自顶向下和自底向上两种执行方法。*(讲义 p49)*

| 执行方法 | 特性 |
|---|---|
| 自顶向下 | 被动的、需求驱动的执行方式 |
| 自底向上 | 主动的执行方式 |

*(讲义 p49)*

> Caveat: 自顶向下 = 被动 / 需求驱动；自底向上 = 主动。二者特性不可混淆。*(题目第五章-单选-18)*

## 八、重点对比与易混辨析

### 1. 代数优化 vs 物理优化

| 维度 | 代数优化 | 物理优化 |
|---|---|---|
| 优化层次 | 关系代数表达式层 | 存取路径与底层算法层 |
| 优化手段 | 等价变换改变操作次序和组合 | 选择存取路径和操作算法 |
| 典型方法 | 启发式规则（选择先做等） | 基于规则 / 基于代价 / 基于语义 |
| 等价变换归属 | 属于代数优化 | 不属于物理优化 |

*(讲义 p47, 题目第五章-判断-3)*

### 2. 查询计划执行方式对比

| 维度 | 自顶向下 | 自底向上 |
|---|---|---|
| 驱动方式 | 需求驱动 | 主动 |
| 特性 | 被动的 | 主动的 |

*(讲义 p49)*

### 3. 优化器 vs 用户程序

| 维度 | 优化器 | 用户程序 |
|---|---|---|
| 统计信息获取 | 可从数据字典获取 | 难以获得数据字典信息 |
| 执行计划考虑数量 | 可考虑数百种 | 一般只能考虑有限的几种 |
| 优化技术 | 包含很多复杂的优化技术 | 只有最好的程序员才能掌握 |
| 自动重优化 | 物理统计信息改变时可自动重新优化 | 无法自动重优化 |

*(讲义 p49, 题目第五章-多选-6, 题目第五章-判断-7)*

## 九、高频考点与易错提醒

> 以下考点信号来自本章专项练习与参考答案解析，仅列考点方向与易错点，不收录题目。

### 高频考点

1. ⭐ **查询处理的任务**：把查询语句转换为高效的查询执行计划（不是编译程序、高级语言程序或 SQL 语句）。*(题目第五章-单选-1)*
2. ⭐ **查询处理四阶段**：查询分析、查询检查、查询优化、查询执行（顺序不可乱）。*(题目第五章-单选-2, 多选-1)*
3. ⭐ **查询分析**：识别语言符号 → 语法检查 + 语法分析（两者都要做）。*(题目第五章-单选-3, 判断-1)*
4. ⭐ **视图消解**：把对视图的操作转换为对基本表的操作。*(题目第五章-单选-4)*
5. ⭐ **SQL → 关系代数**：查询检查时把 SQL 转换为等价的关系代数表达式。*(题目第五章-单选-5)*
6. ⭐ **查询优化总目标**：选择有效策略，使得查询代价较小。*(题目第五章-单选-12)*
7. ⭐ **等价记号**：$E_1 \equiv E_2$，不是 $E_1 = E_2$。*(题目第五章-单选-13, 判断-9)*
8. ⭐ **启发式规则**：选择先做、投影与选择同时做、投影与双目运算结合、选择与笛卡儿积结合成连接、找公共子表达式。*(题目第五章-多选-8)*
9. ⭐ **物理优化—连接算法选择**：两表已排序→排序-合并；有索引→索引连接；表小→Hash Join；兜底→嵌套循环。*(题目第五章-单选-17, 多选-5)*
10. ⭐ **查询计划执行**：自顶向下=被动/需求驱动，自底向上=主动。*(题目第五章-单选-18, 多选-9)*
11. ⭐ **全表扫描块数计算**：块数 = 元组总数 ÷ 每块元组数。*(题目第五章-单选-20)*

### 易错点

1. ⚠ **等价变换属于代数优化**，不是物理优化。*(题目第五章-判断-3)*
2. ⚠ **查询执行由代码生成器生成代码**，不是由用户程序生成。*(题目第五章-判断-4)*
3. ⚠ **等价记号是 $\equiv$**，不是 $=$。*(题目第五章-单选-13, 判断-9)*
4. ⚠ **优化器能获取数据字典统计信息**，用户程序不能。*(题目第五章-判断-7)*
5. ⚠ **用户只需提出"干什么"**（声明性），不必指出"怎么干"（过程性）。*(题目第五章-单选-9)*
6. ⚠ **选择与笛卡儿积的交换律**：$\sigma_F(E_1 \times E_2) \equiv \sigma_F(E_1) \times E_2$，注意不是 $\sigma_F(E_1 \bowtie E_2) \equiv \sigma_F(E_1) \bowtie \sigma_F(E_2)$。*(题目第五章-多选-10)*

### 缺失知识点考点提示

> 以下概念在专项练习中被反复考查，但其知识点正文在 OCR 中缺失（知识点 9–15），仅列出考点方向供复习参考：

- ⭐ **全表扫描算法**：对基本表进行顺序扫描，逐一检查每个元组是否满足选择条件。*(题目第五章-单选-8)*
- ⭐ **集中式数据库查询执行开销**：I/O 代价 + CPU 代价 + 内存代价；最主要的是 I/O 代价。*(题目第五章-单选-10, 单选-11)*
- ⭐ **分布式数据库查询执行开销**：I/O 代价 + CPU 代价 + 内存代价 + 通信代价。*(题目第五章-多选-7)*
- ⭐ **查询代价衡量单位**：一般用查询处理读写的块数作为衡量单位。*(题目第五章-判断-8)*
- ⭐ **选择操作算法**：一般采用全表扫描或索引扫描。*(题目第五章-多选-4)*
- ⭐ **连接操作实现算法**：嵌套循环算法、排序-合并算法、索引连接算法、Hash Join 算法。*(题目第五章-多选-5)*
- ⭐ **Hash Join**：是处理等值连接的算法。*(题目第五章-判断-6)*

## 十、一句话记忆要点

1. **查询处理** = 把 SQL 转换为高效查询执行计划的过程。
2. **四阶段** = 分析 → 检查 → 优化 → 执行（口诀："析检优执"）。
3. **查询检查** = 语义检查 + 视图消解 + 权限/完整性检查 + 转关系代数。
4. **查询树** = 语法分析树 = 扩展关系代数表达式的表示。
5. **代数优化** = 等价变换改操作次序和组合；**物理优化** = 选存取路径和算法。
6. **启发式五条** = 选择先做、投影选择同时做、投影搭双目、选择搭笛卡儿积变连接、找公共子表达式。
7. **连接算法选择** = 已排序→排序合并、有索引→索引连接、表小→Hash Join、兜底→嵌套循环（小表做外表）。
8. **等价记号** = $E_1 \equiv E_2$（三横线，不是等号）。
9. **执行方式** = 自顶向下被动需求驱动、自底向上主动。
10. **代价估算** = 统计 $N/L/B/BO$（表）+ $m$/max/min/索引类型（列）+ $Z/S/Y$（索引）。

## Worked Examples

### 例 1：全表扫描块数计算 *(来自第五章-单选-20)*

*Tags: 全表扫描, 查询代价, I/O 代价*

**问题**：关系 R 有 20000 个元组，关系 S 有 1200 个元组，一个块能装 40 个 R 的元组，能装 30 个 S 的元组，R 上没有索引，`SELECT * FROM R` 需要（ ）次磁盘块读写。*(题目第五章-单选-20)*

**解答**：

1. R 上没有索引，需要对 R 进行全表扫描。
2. 全表扫描需要读取 R 的所有磁盘块。
3. 块数 = 元组总数 ÷ 每块元组数 = $20000 \div 40 = 500$。
4. 答案：**500 次**（选项 C）。
