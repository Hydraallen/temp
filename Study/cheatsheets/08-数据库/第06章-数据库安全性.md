# 数据库 · 数据库安全性 — 复习资料

> 来源：8-3-数据库计算机类讲义.pdf 物理页 [56, 67]
> 说明：⭐=高频考点，⚠=易错点；> Caveat: 为陷阱/技巧；末尾 Worked Examples 为代表性题目（无书内答案的标 [PROPOSED — verify]）。

## 一、安全性基本概念

**数据库安全性**：保护数据库以防止不合法使用所造成的数据泄露、更改或破坏。数据库系统中大量数据集中存放，为众多最终用户直接共享，安全性问题突出。系统安全保护措施是否有效是数据库系统的主要技术指标之一。

**不安全因素**（三方面）：
1. 非授权用户对数据库的恶意存取和破坏
2. 数据库中重要或敏感的数据被泄露
3. 安全环境的脆弱性

> Caveat (第六章-单选-19): 封锁是并发控制的措施，不是安全性控制措施。安全性控制的常用方法和技术为：用户标识与鉴别、存取控制、视图机制、审计、数据加密。

> Caveat (第六章-多选-2): 封锁用于并发控制而不是安全性控制；数据转储是数据库恢复技术，不是安全性技术。

## 二、安全标准

**安全标准概述**：计算机及信息安全技术方面最有影响的是 TCSEC 和 CC 两个标准。目前 CC 已基本取代 TCSEC，成为评估信息产品安全性的主要标准。

**信息安全标准发展历程**：

| 年份 | 标准 |
|---|---|
| 1985 | 美国国防部可信计算机系统评估准则（TCSEC） |
| 1991 | 欧洲信息技术安全评估准则（ITSEC） |
| 1993 | 加拿大可信计算机产品评估准则（CICPEC） |
| 1993 | 美国信息技术安全联邦标准（FC）草案 |
| 1996 | 通用准则（CC）V1.0 |
| 1998 | CC V2.0 |
| 1999 | CC V2.1 成为国际标准（ISO 15408） |

### TCSEC/TDI 安全级别划分 ⭐

将系统划分为 **4 组 7 个等级**，依次是 D、C（C1, C2）、B（B1, B2, B3）、A（A1），按系统可靠或可信程度逐渐增高。

| 安全级别 | 定义 |
|---|---|
| A1 | 验证设计 |
| B3 | 安全域 |
| B2 | 结构化保护 |
| B1 | 标记安全保护 |
| C2 | 受控的存取保护 |
| C1 | 自主安全保护 |
| D | 最小保护 |

### 通用准则 CC ⭐

CC 在各评估准则及实践基础上发展而来，特点是**结构开放、表达方式通用**。CC 把对信息产品的安全要求分为：

- **安全功能要求**：规范产品和系统的安全行为
- **安全保证要求**：解决如何正确有效地实施这些功能

两者都以"**类-子类-组件**"结构表述，**组件是安全要求的最小构件块**。

> Caveat (第六章-多选-3): CC 把安全要求分为"安全功能要求"和"安全保证要求"两类，不是"加密/解密功能要求"。

### CC 文本组成 ⭐

由三部分组成，相互依存，缺一不可：

| 部分 | 内容 |
|---|---|
| 第一部分 | 简介和一般模型 |
| 第二部分 | 安全功能要求（列出类、子类和组件） |
| 第三部分 | 安全保证要求（列出保证类、子类和组件），提出评估保证级 EAL1–EAL7 |

**CC 评估保证级（EAL）划分**：从 EAL1 至 EAL7 共 7 级，按保证程度逐渐增高。

| 评估保证级 | 定义 | TCSEC 安全级别（近似相当） |
|---|---|---|
| EAL1 | 功能测试 | — |
| EAL2 | 结构测试 | C1 |
| EAL3 | 系统地测试和检查 | C2 |
| EAL4 | 系统地设计、测试和复查 | B1 |
| EAL5 | 半形式化设计和测试 | B2 |
| EAL6 | 半形式化验证的设计和测试 | B3 |
| EAL7 | 形式化验证的设计和测试 | A1 |

## 三、数据库安全性控制体系

**层层设防**：在一般计算机系统中，安全措施是一级一级层层设置的。

```
用户标识和鉴别 → 数据库安全保护（存取控制）→ 操作系统安全保护 → 数据密码存储
                                                    ↓
                                              简单入侵检测、审计
```

具体层次：
- **用户标识鉴定**用户身份，合法用户准许进入系统
- **DBMS 进行存取控制**，只允许用户执行合法操作
- **操作系统**有自己的保护措施
- **数据以密文形式**存储到数据库中

数据库安全性控制的多层架构包括：身份鉴别 → 多层访问控制策略（自主访问控制 → 强制访问控制 → 推理控制）→ SQL 层 → 数据层（用户数据信息、安全数据信息含审计）。

## 四、用户身份鉴别 ⭐

用户身份鉴别是 DBMS 提供的**最外层安全保护措施**。每个用户标识由**用户名**和**用户标识号**两部分组成。

| 鉴别方式 | 特点 | 安全性 |
|---|---|---|
| 静态口令鉴别 | 用户自己设定，输入正确口令即可登录；方式简单但容易被攻击 | 较低 |
| 动态口令鉴别 | 口令动态变化，一次一密；如短信密码方式 | 较安全 |
| 生物特征鉴别 | 通过指纹、虹膜、掌纹等生物特征认证；采用图像处理和模式识别技术 | 安全性较高 |
| 智能卡鉴别 | 不可复制的硬件，内置集成电路芯片，具有硬件加密功能；实际应用中一般采用 PIN 和智能卡相结合 | 较高 |

> Caveat: 智能卡鉴别中，PIN 与智能卡结合使用，即使其中一种被窃取，用户身份仍不会被冒充。

## 五、存取控制机制

### 存取控制的两部分 ⭐

存取控制机制主要包括**定义用户权限**和**合法权限检查**两部分：

1. **定义用户权限**，并将用户权限登记到数据字典中。用户对某一数据对象的操作权力称为权限。DBMS 定义的用户权限经编译后存储在数据字典中，称作**安全规则**或**授权规则**。
2. **合法权限检查**：每当用户发出操作请求，DBMS 查找数据字典进行合法权限检查。

两者一起组成了 DBMS 的**存取控制子系统**。

### 自主存取控制（DAC）与强制存取控制（MAC）⭐⚠

| 对比项 | 自主存取控制（DAC） | 强制存取控制（MAC） |
|---|---|---|
| 支持级别 | C2 级 DBMS | B1 级 DBMS |
| 机制 | 用户对不同对象有不同权限，不同用户对同一对象有不同权限，用户可将权限转授他人 | 每个对象标以密级，每个用户授予许可证级别，只有合法许可证才可存取 |
| 特点 | 非常灵活 | 相对严格 |
| 安全性 | 较低 | 较高 |
| 缺点 | 可能存在数据"无意泄露"，数据本身无安全性标记 | [原文未述] |

> Caveat (第六章-判断-7): 自主存取控制的安全性**低于**强制存取控制。

### 授权概念 ⭐

用户权限由两个要素组成：**数据库对象**和**操作类型**。定义一个用户的存取权限就是要定义这个用户可以在哪些数据库对象上进行哪些类型的操作。在数据库系统中，定义存取权限称为**授权**。

> Caveat (第六章-判断-6): 用户权限由**数据对象**和**操作类型**组成，不是"数据类型"。

### 关系数据库系统中的存取权限 ⭐

| 对象类型 | 对象 | 操作类型 |
|---|---|---|
| 数据库模式 | 模式 | CREATE SCHEMA |
| 基本表 | CREATE TABLE, ALTER TABLE |
| 视图 | CREATE VIEW |
| 索引 | CREATE INDEX |
| 数据 | 基本表和视图 | SELECT, INSERT, UPDATE, DELETE, REFERENCES, ALL PRIVILEGES |
| 属性列 | SELECT, INSERT, UPDATE, REFERENCES, ALL PRIVILEGES |

> Caveat (第六章-单选-11): "删除基本表中的元组"应授予 **DELETE** 权限，不是 DROP（DROP 是删除表本身）。

## 六、SQL 授权与收回

### GRANT 语句 ⭐

```sql
GRANT <权限>[, <权限>]...
ON <对象类型> <对象名>[, <对象类型> <对象名>]...
TO <用户>[, <用户>]...
[WITH GRANT OPTION];
```


语义：将对指定操作对象的指定操作权限授予指定的用户。如果指定了 `WITH GRANT OPTION` 子句，则获得某种权限的用户还可以把这种权限再授予其他用户。

### REVOKE 语句 ⭐

```sql
REVOKE <权限>[, <权限>]...
ON <对象类型> <对象名>[, <对象类型> <对象名>]...
FROM <用户>[, <用户>]...
[CASCADE | RESTRICT];
```


注意：默认值为 CASCADE，有的 DBMS 默认值为 RESTRICT，将自动执行级联操作。

> Caveat (第六章-单选-8): 收回查询权限的语句是 `REVOKE SELECT ON T FROM USER`，注意权限关键字是 SELECT 不是 UPDATE。

> Caveat (第六章-多选-10): GRANT（授权）和 REVOKE（回收）均为**数据控制语言 DCL**。

### 创建数据库模式的权限 ⭐

GRANT 和 REVOKE 语句向用户授予或收回对**数据**的操作权限。对**创建数据库模式**一类的数据库对象的授权则由 DBA 在创建用户时实现。

```sql
CREATE USER <username>
[WITH] [DBA|RESOURCE|CONNECT];
```


要点：
1. 只有系统的**超级用户**才有权创建一个新的数据库用户
2. 新创建的数据库用户有三种权限：CONNECT、RESOURCE 和 DBA
3. 未指定权限时，默认拥有 **CONNECT** 权限

| 权限 | 能否创建新用户 | 能否创建模式 | 能否创建基本表/视图 | 其他说明 |
|---|---|---|---|---|
| CONNECT（默认） | 否 | 否 | 否 | 只能登录数据库，由 DBA 或其他用户授予应有权限后操作 |
| RESOURCE | 否 | 否 | 是 | 成为所创建对象的属主，属主可用 GRANT 授予权限 |
| DBA | 是 | 是 | 是 | 系统超级用户，拥有对所有数据库对象的存取权限，可授权给一般用户 |

> Caveat (第六章-多选-8): CONNECT 权限的用户不能创建新用户、不能创建模式、也不能创建基本表，只能登录数据库。

### 数据库角色 ⭐

数据库角色是被命名的一组与数据库操作相关的权限，角色是**权限的集合**。优点：**简化授权过程**。

```sql
-- 1. 创建角色
CREATE ROLE <角色名>

-- 2. 给角色授权
GRANT <权限>[, <权限>]...
ON <对象类型> <对象名>
TO <角色>[, <角色>]...

-- 3. 将角色授予其他角色或用户
GRANT <角色1>[, <角色2>]...
TO <角色3>[, <用户1>]...
[WITH ADMIN OPTION]

-- 4. 收回角色权限
REVOKE <权限>[, <权限>]...
ON <对象类型> <对象名>
FROM <角色>[, <角色>]...
```


- 指定了 `WITH ADMIN OPTION`，则获得某种权限的角色或用户还可以把这种权限授予其他角色。
- 一个角色的权限 = 直接授予这个角色的全部权限 + 其他角色授予这个角色的全部权限。
- REVOKE 执行者是角色的创建者或拥有在这个角色上的 ADMIN OPTION。

> Caveat (第六章-判断-4): 角色授权使用 `WITH ADMIN OPTION`，**不是** `WITH GRANT OPTION`。`WITH GRANT OPTION` 用于普通权限的转授。

> Caveat (第六章-判断-8): 数据库中的角色是权限的集合，不是"用户扮演的虚拟人物"。

## 七、强制存取控制（MAC）详解 ⭐⚠

### 自主存取控制的缺点

自主存取控制可能存在数据的"**无意泄露**"。这种机制仅仅通过对数据的存取权限来进行安全控制，而**数据本身并无安全性标记**。

### 主体和客体

在强制存取控制中，DBMS 所管理的全部实体被分为**主体**和**客体**两大类：

| 类别 | 定义 | 举例 |
|---|---|---|
| 主体 | 系统中的**活动实体**，包括 DBMS 所管理的实际用户，也包括代表用户的各进程 | 实际用户、进程 |
| 客体 | 系统中的**被动实体**，受主体操纵 | 文件、基本表、索引、视图等 |

### 敏感度标记

敏感度标记被分成若干级别：

| 标记级别 | 缩写 |
|---|---|
| 绝密 | TS |
| 机密 | S |
| 可信 | C |
| 公开 | P |

密级次序：$TS \geq S \geq C \geq P$

- **主体的敏感度标记**称为**许可证级别**
- **客体的敏感度标记**称为**密级**

### MAC 读写规则 ⭐⚠

通过对比主体的敏感度标记和客体的敏感度标记，最终确定主体是否能够存取客体：

1. **读规则（向下读）**：仅当主体的许可证级别 $\geq$ 客体的密级时，该主体才能**读**相应的客体
2. **写规则（向上写）**：仅当主体的许可证级别 $\leq$ 客体的密级时，该主体才能**写**相应的客体

> Caveat (第六章-多选-9): 强制存取控制（MAC）中，读规则是"**向下读**"（主体级别 $\geq$ 客体密级），写规则是"**向上写**"（主体级别 $\leq$ 客体密级）。注意写规则方向与读规则相反，这是高频易错点。

## 八、视图机制

视图机制间接地实现支持存取谓词的用户权限定义。把要保密的数据对无权存取这些数据的用户隐藏起来，对数据提供一定程度的安全保护。

> Caveat (第六章-单选-16): SQL 中的视图提高了数据库的**安全性**（针对不同用户创建不同视图，限制其浏览和编辑的数据内容）。

## 九、审计 ⭐

### 审计概念

审计功能将用户对数据库的所有操作自动记录下来，写入**审计日志**。审计员利用审计日志监控数据库中的各种行为，找出非法存取数据的人、时间和内容。

> Caveat: 注意区别**审计日志**和**事务日志**。审计日志记录用户对数据库的**所有操作**；事务日志只记录事务对数据库的**更新操作**。

### 审计功能的可选性 ⚠

审计很费时间和空间。DBA 可以打开或关闭审计功能；审计功能主要用于安全性要求较高的部门。

> Caveat (第六章-判断-9): 审计比较**耗时耗资源**，不是"节约资源"，因此作为可选的安全性手段。

### 审计事件分类 ⭐

| 事件类别 | 说明 |
|---|---|
| 服务器事件 | 审计数据库服务器发生的事件 |
| 系统权限 | 对系统拥有的结构或模式对象进行操作的审计 |
| 语句事件 | 对 SQL 语句（DDL、DML、DQL 及 DCL）的审计 |
| 模式对象事件 | 对特定模式对象上的 SELECT 或 DML 操作的审计 |

> Caveat (第六章-单选-17): 模式对象包括表、视图、**存储过程**、函数等；模式对象**不包括**依附于表的索引、约束、触发器、分区表等。

### 审计功能内容

审计功能主要包括：

1. **基本功能**：提供多种审计查阅方式（基本的、可选的、有限的等）
2. **多套审计规则**：审计规则一般在数据库初始化时设定
3. **审计分析和报表功能**
4. **审计日志管理**：为防止审计员误删审计记录，审计日志必须**先转储后删除**；对转储的审计记录文件提供完整性和保密性保护；只允许审计员查阅和转储审计记录，不允许任何用户新增和修改审计记录
5. **专门视图**：系统提供查询审计设置及审计记录信息的专门视图；对于系统权限级别、语句级别及模式对象级别的审计记录也可通过相关的系统表直接查看

### 审计功能的设置与取消

```sql
-- 设置审计功能
AUDIT ...

-- 取消审计功能
NOAUDIT ...
```


### 用户级审计与系统级审计 ⭐

| 对比项 | 用户级审计 | 系统级审计 |
|---|---|---|
| 设置者 | 任何用户 | 仅 DBA |
| 审计范围 | 用户针对自己创建的数据库表或视图，记录所有用户对这些表或视图的一切成功和不成功的访问要求及各种类型的 SQL 操作 | 监测成功或失败的登录要求、监测授权和收回操作及其他数据库级权限下的操作 |

审计设置以及审计日志一般都存储在**数据字典**中。

> Caveat (第六章-单选-18): 系统级审计只能由**数据库管理员**设置。

## 十、数据加密

### 加密基本思想 ⭐

数据加密是防止数据库数据在存储和传输中失密的有效手段。

加密的基本思想：根据一定的算法将原始数据"**明文**"变换为不可直接识别的格式"**密文**"，使得不知道解密算法的人无法获知数据的内容。

数据加密主要包括**存储加密**和**传输加密**。

### 存储加密

提供**非透明**和**透明**两种存储加密方式：

| 方式 | 实现机制 | 特点 |
|---|---|---|
| 非透明存储加密 | 通过多个加密函数实现 | [原文未述] |
| 透明存储加密 | 内核级加密保护方式，数据写到磁盘时加密，授权用户读取时解密 | 对用户完全透明，应用程序不需要修改，只需在创建表语句中说明需加密的字段；DBMS 自动完成加解密；性能较好，安全完备性较高 |

### 传输加密

在客户/服务器结构中，数据库用户与服务器之间若采用明文方式传输数据，容易被网络恶意用户截获或篡改。

| 加密方式 | 加密层 | 加密范围 | 特点 |
|---|---|---|---|
| 链路加密 | 链路层 | 报文和报头均加密 | [原文未述] |
| 端到端加密 | 发送端加密，接收端解密 | 只加密报文，不加密报头 | 所需密码设备数量相对较少，容易被非法监听者发现并从中获取敏感信息 |

## 十一、其他安全性保护措施

**推理控制**以及数据库应用中**隐蔽信道**和**数据隐私保护**等技术。

**数据隐私**：控制不愿被他人知道或他人不便知道的个人数据的能力。

## 十二、公式 / 算法 / 定理速查表

| 项目 | 内容 |
|---|---|
| TCSEC/TDI 等级 | D < C(C1,C2) < B(B1,B2,B3) < A(A1)，4 组 7 级 |
| CC EAL 等级 | EAL1 < EAL2 < EAL3 < EAL4 < EAL5 < EAL6 < EAL7，7 级 |
| CC 安全要求分类 | 安全功能要求 + 安全保证要求 |
| CC 结构 | 类-子类-组件（组件 = 最小构件块） |
| 用户权限要素 | 数据库对象 + 操作类型 |
| 密级次序 | $TS \geq S \geq C \geq P$ |
| MAC 读规则 | 主体许可证级别 $\geq$ 客体密级 → 可读 |
| MAC 写规则 | 主体许可证级别 $\leq$ 客体密级 → 可写 |
| CREATE USER 默认权限 | CONNECT |
| 角色转授权限子句 | WITH ADMIN OPTION（非 WITH GRANT OPTION） |
| 普通权限转授子句 | WITH GRANT OPTION |
| 审计设置/取消 | AUDIT / NOAUDIT |
| 加密变换 | 明文 → 密文 |

## 十三、高频考点与易错提醒

**高频考点**（来自专项练习信号）：

1. **安全性 vs 完整性 vs 并发控制 vs 恢复的概念区分**：安全性 = 防止不合法使用造成泄露/更改/破坏；封锁属于并发控制；数据转储属于恢复技术
2. **GRANT/REVOKE 语法**：`GRANT <权限> ON <对象类型> <对象名> TO <用户> [WITH GRANT OPTION]`；`REVOKE <权限> ON <对象类型> <对象名> FROM <用户> [CASCADE|RESTRICT]`
3. **TCSEC/TDI 4 组 7 级**：D、C(C1,C2)、B(B1,B2,B3)、A(A1)
4. **DAC vs MAC**：DAC 灵活但安全性低，MAC 严格且安全性高；C2 级支持 DAC，B1 级支持 MAC
5. **MAC 读写规则**：向下读（主体 $\geq$ 客体）、向上写（主体 $\leq$ 客体）
6. **角色 = 权限集合**，简化授权过程
7. **CREATE USER 三种权限**：CONNECT（默认）、RESOURCE、DBA
8. **审计**：AUDIT 设置、NOAUDIT 取消；系统级审计由 DBA 设置；审计耗时耗资源
9. **数据加密**：明文 → 密文；存储加密（透明/非透明）、传输加密（链路/端到端）
10. **DCL**：GRANT 和 REVOKE 均属于数据控制语言 DCL

**易错点**：

- 封锁 ≠ 安全性措施（是并发控制措施）
- 数据转储 ≠ 安全性技术（是恢复技术）
- 授权粒度越细，检查权限的开销越**大**（非越小）
- 角色转授用 `WITH ADMIN OPTION`，普通权限转授用 `WITH GRANT OPTION`，两者不可混用
- 用户权限 = 数据对象 + 操作类型（非"数据类型"）
- DAC 安全性**低于** MAC
- 审计**耗时耗资源**（非节约资源）
- DELETE 删除元组，DROP 删除表本身
- 模式对象不包括索引、约束、触发器、分区表

## 十四、一句话记忆要点

- **安全性**：防泄露、防更改、防破坏（非防错误数据进入——那是完整性）
- **安全标准**：TCSEC 4 组 7 级（D<C<B<A），CC 取代 TCSEC，EAL 1→7 递增
- **层层设防**：用户标识鉴别 → 存取控制 → OS 保护 → 数据加密
- **身份鉴别**：静态口令（低）< 动态口令 < 生物特征（高）< 智能卡+PIN
- **存取控制**：定义用户权限 + 合法权限检查 = 存取控制子系统
- **DAC**：灵活可转授，但数据无安全标记，可能无意泄露
- **MAC**：主体许可证 vs 客体密级，向下读、向上写，严格安全
- **GRANT/REVOKE**：DCL 语言，GRANT…TO…[WITH GRANT OPTION]，REVOKE…FROM…
- **角色**：权限集合，WITH ADMIN OPTION 转授
- **CREATE USER**：CONNECT（默认，只能登录）/ RESOURCE（可建表视图）/ DBA（超级用户）
- **视图**：隐藏保密数据，提高安全性
- **审计**：AUDIT/NOAUDIT，记录所有操作，耗时耗资源，系统级由 DBA 设置
- **加密**：明文→密文，存储加密（透明/非透明）+ 传输加密（链路/端到端）

## Worked Examples

### 例 1：MAC 读写规则判定 *(来自本章 expanded 强制存取控制读写规则)*
*Tags: 安全性控制, MAC, 主体许可证, 客体密级*
**问题**：在强制存取控制（MAC）中，主体许可证级别为 S（机密），现有客体 X 密级为 C（可信）、客体 Y 密级为 TS（绝密）。判断该主体能否读 X、能否写 Y。
**解答**：`[PROPOSED — verify]`
1. 密级次序：$TS \geq S \geq C \geq P$。
2. 读规则（向下读）：主体许可证 $\geq$ 客体密级时才可读。$S \geq C$ 成立 → **可读 X**。
3. 写规则（向上写）：主体许可证 $\leq$ 客体密级时才可写。$S \leq TS$ 成立 → **可写 Y**。
4. 易错点：写规则方向与读规则相反，这是高频易错点。

### 例 2：GRANT/REVOKE 语法判定 *(来自本章 expanded SQL 授权与收回语法)*
*Tags: GRANT, REVOKE, DCL, WITH GRANT OPTION*
**问题**：判断以下两条 SQL 语句的正误，并说明原因：
（1）`GRANT SELECT ON T TO USER1 WITH ADMIN OPTION;`
（2）`REVOKE SELECT ON T FROM USER1;`
**解答**：`[PROPOSED — verify]`
1. 语句（1）错误：普通权限转授应使用 `WITH GRANT OPTION`；`WITH ADMIN OPTION` 用于角色转授，两者不可混用。
2. 正确写法：`GRANT SELECT ON T TO USER1 WITH GRANT OPTION;`
3. 语句（2）正确：`REVOKE <权限> ON <对象类型> <对象名> FROM <用户>` 是标准收回权限语法。
4. 归类：GRANT 与 REVOKE 同属数据控制语言 DCL。
