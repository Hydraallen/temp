# 数据库 · 第六章 数据库安全性 — 复习资料

> 说明：仅摘录「知识点」正文；⭐=高频考点，⚠=易错点（依据本章专项练习与答案标注）。

## 一、本章概览

本章解决数据库系统中**如何保护数据不被不合法使用**的问题，是数据库安全防护的核心内容。全章围绕"安全性概念→安全标准→安全性控制技术"三大块展开：首先定义数据库安全性并介绍 TCSEC/TDI 与 CC 两类安全标准；然后重点讲解数据库安全性控制的多种技术手段，包括用户身份鉴别、存取控制（自主存取控制 DAC 与强制存取控制 MAC）、视图机制、审计、数据加密等；最后补充推理控制与数据隐私等其他保护措施。本章与后续"数据库完整性"（防止非法数据进入）形成对照，安全性关注的是**防止不合法使用**，完整性关注的是**防止非法数据**。

---

## 二、知识点详解

### 1. 安全性 ⭐(练习1,多选1,判断1)

- **定义**：数据库的安全性是指保护数据库以防止不合法使用所造成的数据泄露、更改或破坏。
- **要点**：
  - 数据库系统中大量数据集中存放，为众多最终用户直接共享，安全性问题突出。
  - 系统安全保护措施是否有效是数据库系统的主要技术指标之一。

### 2. 不安全因素

对数据库安全性产生威胁的因素主要有以下几方面：

1. 非授权用户对数据库的恶意存取和破坏
2. 数据库中重要或敏感的数据被泄露
3. 安全环境的脆弱性

### 3. 安全标准

- 计算机以及信息安全技术方面有一系列的安全标准，最有影响的当推 **TCSEC** 和 **CC** 这两个标准。
- 目前 CC 已经基本取代了 TCSEC，成为评估信息产品安全性的主要标准。
- **信息安全标准发展历史**：
  - 1985年 美国国防部可信计算机系统评估准则（TCSEC）
  - 1993年 加拿大可信计算机产品评估准则（CICPEC）
  - 1991年 欧洲信息技术安全评估准则（ITSEC）
  - 1993年 美国信息技术安全联邦标准（FC）草案
  - 通用准则（CC）：V1.0 1996年，V2.0 1998年，V2.1 1999年
  - 1999年 CC V2.1 成为国际标准（ISO15408）

### 4. TCSEC/TDI ⭐(练习4)

- 将系统划分为 **4 组 7 个等级**，依次是 D、C（C1, C2）、B（B1, B2, B3）、A（A1），按系统可靠或可信程度逐渐增高。

| 安全级别 | 定义 |
| --- | --- |
| A1 | 验证设计 |
| B3 | 安全域 |
| B2 | 结构化保护 |
| B1 | 标记安全保护 |
| C2 | 受控的存取保护 |
| C1 | 自主安全保护 |
| D | 最小保护 |

### 5. 通用准则 CC (多选3)

- CC 是在上述各评估准则及具体实践的基础上通过相互总结和互补发展而来的。
- 和早期的评估准则相比，CC 具有**结构开放、表达方式通用**等特点。
- CC 提出了目前国际上公认的表述信息技术安全性的结构，即把对信息产品的安全要求分为**安全功能要求**和**安全保证要求**：
  - **安全功能要求**：用以规范产品和系统的安全行为。
  - **安全保证要求**：解决如何正确有效地实施这些功能。
  - 两者都以"**类-子类-组件**"的结构表述，**组件是安全要求的最小构件块**。

### 6. CC 的文本组成

由三部分组成，三个部分相互依存，缺一不可：

| 部分 | 内容 |
| --- | --- |
| 第一部分 | 简介和一般模型 |
| 第二部分 | 安全功能要求，列出了一系列类、子类和组件 |
| 第三部分 | 安全保证要求，列出了一系列保证类、子类和组件；根据系统对安全保证要求的支持情况提出评估保证级，从 EAL1 至 EAL7 共 7 级，按保证程度逐渐增高 |

**CC 评估保证级（EAL）划分表：**

| 评估保证级 | 定义 | TCSEC 安全级别（近似相当） |
| --- | --- | --- |
| EAL1 | 功能测试 | |
| EAL2 | 结构测试 | C1 |
| EAL3 | 系统地测试和检查 | C2 |
| EAL4 | 系统地设计、测试和复查 | B1 |
| EAL5 | 半形式化设计和测试 | B2 |
| EAL6 | 半形式化验证的设计和测试 | B3 |
| EAL7 | 形式化验证的设计和测试 | A1 |

### 7. 数据库安全性控制 ⭐(练习2)

在一般计算机系统中，安全措施是一级一级层层设置的：

- **用户标识和鉴别**：标识鉴定用户身份，合法用户准许进入系统
- **数据库管理系统存取控制**：只允许用户执行合法操作
- **操作系统安全保护**：操作系统有自己的保护措施
- **数据密码存储**：数据以密文形式存储到数据库中

此外还有简单入侵检测、审计等手段。整体安全架构涵盖：数据库用户→身份鉴别→多层访问控制策略（SQL 层：推理控制、强制访问控制、自主访问控制）→数据层（用户数据信息、用户数据备份信息、安全数据信息（包括审计）、安全数据备份信息）。

### 8. 用户身份鉴别

- 用户身份鉴别是数据库管理系统提供的**最外层安全保护措施**。
- 每个用户标识由**用户名**和**用户标识号**两部分组成。每次用户要求进入系统时，由系统进行核对，通过鉴定后才提供使用数据库管理系统的权限。

**四种鉴别方式：**

| 方式 | 要点 | 安全性 |
| --- | --- | --- |
| **（1）静态口令鉴别** | 一般由用户自己设定，鉴别时只要按要求输入正确的口令，系统将允许用户使用数据库管理系统。DBMS 从口令的复杂度、口令的管理、存储及传输等多方面来保障口令的安全可靠 | 方式简单，但容易被攻击，安全性较低 |
| **（2）动态口令鉴别** | 口令是动态变化的，每次鉴别时均需使用动态产生的新口令登录，即采用**一次一密**的方法。常用方式如短信密码方式 | 目前较为安全的鉴别方式 |
| **（3）生物特征鉴别** | 通过生物特征进行认证的技术。生物特征是指生物体唯一具有的、可测量、识别和验证的稳定生物特征，如指纹、虹膜和掌纹等。采用图像处理和模式识别等技术实现 | 与传统口令鉴别相比产生了质的飞跃，安全性较高 |
| **（4）智能卡鉴别** | 智能卡是不可复制的硬件，内置集成电路的芯片，具有硬件加密功能。登录时将智能插入专用读卡器验证。实际应用中一般采用**个人身份识别码（PIN）和智能卡相结合**的方式 | 即使 PIN 或智能卡中有一种被窃取，用户身份仍不会被冒充 |

### 9. 存取控制机制 (多选4)

主要包括**定义用户权限**和**合法权限检查**两部分：

1. **定义用户权限**，并将用户权限登记到数据字典中。用户对某一数据对象的操作权力称为权限。DBMS 定义用户权限，这些定义经过编译后存储在数据字典中，被称作**安全规则**或**授权规则**。
2. **合法权限检查**。

> 定义用户权限和合法权限检查机制一起组成了数据库管理系统的**存取控制子系统**。

### 10. 自主存取控制、强制存取控制 ⭐⚠(练习14,判断7,多选9)

- **C2 级**的 DBMS 支持**自主存取控制（DAC）**，**B1 级**的 DBMS 支持**强制存取控制（MAC）**。

| | 自主存取控制（DAC） | 强制存取控制（MAC） |
| --- | --- | --- |
| **方法** | 用户对于不同的数据库对象有不同的存取权限，不同的用户对同一对象也有不同的权限，而且用户还可将其拥有的存取权限转授给其他用户 | 每一个数据库对象被标以一定的密级，每一个用户也被授予某一个级别的许可证。对于任意一个对象，只有具有合法许可证的用户才可以存取 |
| **特点** | 非常灵活 | 相对比较严格 |
| **安全性** | 较低（可能存在数据的"无意泄露"，数据本身无安全性标记） | 较高 |

### 11. 授权 ⭐(练习5,6,7)

- 用户权限由两个要素组成：**数据库对象**和**操作类型**。
- 定义一个用户的存取权限就是要定义这个用户可以在哪些数据库对象上进行哪些类型的操作。
- 在数据库系统中，定义存取权限称为**授权**。

### 12. 关系数据库系统中的存取权限 ⭐(练习11,多选6,7)

| 对象类型 | 对象 | 操作类型 |
| --- | --- | --- |
| 数据库模式 | 模式 | CREATE SCHEMA |
| | 基本表 | CREATE TABLE, ALTER TABLE |
| | 视图 | CREATE VIEW |
| | 索引 | CREATE INDEX |
| 数据 | 基本表和视图 | SELECT, INSERT, UPDATE, DELETE, REFERENCES, ALL PRIVILEGES |
| | 属性列 | SELECT, INSERT, UPDATE, REFERENCES, ALL PRIVILEGES |

### 13. 授权与收回的实现 ⭐⚠(练习8,9,判断4)

SQL 中使用 **GRANT** 语句向用户授予权限，**REVOKE** 语句收回已经授予用户的权限。

**GRANT 语句：**

```sql
GRANT <权限>[, <权限>]...
ON <对象类型> <对象名>[, <对象类型> <对象名>]...
TO <用户>[, <用户>]...
[WITH GRANT OPTION];
```

- 语义：将对指定操作对象的指定操作权限授予指定的用户。
- 如果指定了 **WITH GRANT OPTION** 子句，则获得某种权限的用户还可以把这种权限再授予其他的用户。

**REVOKE 语句：**

```sql
REVOKE <权限>[, <权限>]...
ON <对象类型> <对象名>[, <对象类型> <对象名>]...
FROM <用户>[, <用户>]...
[CASCADE | RESTRICT];
```

- 注意：这里默认值为 **CASCADE**，有的 DBMS 默认值为 RESTRICT，将自动执行级联操作。

### 14. 创建数据库模式的权限 (多选8)

GRANT 和 REVOKE 语句向用户授予或收回对数据的操作权限。对创建数据库模式一类的数据库对象的授权则由**数据库管理员在创建用户时**实现。

```sql
CREATE USER <username> [WITH] [DBA|RESOURCE|CONNECT];
```

**要点：**

1. 只有系统的**超级用户**才有权创建一个新的数据库用户。
2. 新创建的数据库用户有三种权限：**CONNECT、RESOURCE 和 DBA**。
3. CREATE USER 命令中如果没有指定创建的新用户的权限，默认该用户拥有 **CONNECT** 权限。

| 权限 | 能力 |
| --- | --- |
| **CONNECT** | 不能创建新用户，不能创建模式，也不能创建基本表，只能登录数据库。由 DBA 或其他用户授予他应有的权限，根据获得的授权情况进行权限范围内的操作 |
| **RESOURCE** | 能创建基本表和视图，成为所创建对象的属主，但不能创建模式，不能创建新的用户。属主可以使用 GRANT 语句把该对象上的存取权限授予其他用户 |
| **DBA** | 系统中的超级用户，可以创建新的用户、创建模式、创建基本表和视图等。DBA 拥有对所有数据库对象的存取权限，还可以把这些权限授予一般用户 |

### 15. 数据库角色 ⭐(练习12,13,判断8)

- 数据库角色是被命名的**一组与数据库操作相关的权限**，角色是**权限的集合**。
- **优点**：简化授权的过程。

**1. 角色的创建：**

```sql
CREATE ROLE <角色名>
```

**2. 给角色授权：**

```sql
GRANT <权限>[, <权限>]...
ON <对象类型> <对象名>
TO <角色>[, <角色>]...
```

**3. 将一个角色授予其他的角色或用户：**

```sql
GRANT <角色 1>[, <角色 2>]...
TO <角色 3>[, <用户 1>]...
[WITH ADMIN OPTION]
```

- 该语句把角色授予某用户，或授予另一个角色。
- 授予者是角色的创建者或拥有在这个角色上的 ADMIN OPTION。
- 指定了 **WITH ADMIN OPTION** 则获得某种权限的角色或用户还可以把这种权限授予其他角色。
- 一个角色的权限 = 直接授予这个角色的全部权限 + 其他角色授予这个角色的全部权限。

**4. 角色权限的收回：**

```sql
REVOKE <权限>[, <权限>]...
ON <对象类型> <对象名>
FROM <角色>[, <角色>]...
```

- 用户可以回收角色的权限，从而修改角色拥有的权限。
- REVOKE 执行者是角色的创建者或拥有在这个（些）角色上的 ADMIN OPTION。

> ⚠ **易错**：角色间授权使用 `WITH ADMIN OPTION`，而非 `WITH GRANT OPTION`（判断4）。

### 16. 自主存取控制的缺点

- 自主存取控制可能存在数据的"**无意泄露**"。
- 这种机制仅仅通过对数据的存取权限来进行安全控制，而**数据本身并无安全性标记**。

### 17. 主体和客体 (练习14)

在强制存取控制中，DBMS 所管理的全部实体被分为**主体**和**客体**两大类：

| | 定义 | 举例 |
| --- | --- | --- |
| **主体** | 系统中的**活动实体**，既包括 DBMS 所管理的实际用户，也包括代表用户的各进程 | 实际用户、进程 |
| **客体** | 系统中的**被动实体**，是受主体操纵的 | 文件、基本表、索引、视图等 |

### 18. 敏感度标记 (练习15)

- 敏感度标记被分成若干级别，例如**绝密（TS）、机密（S）、可信（C）、公开（P）**等。
- 密级的次序是 $TS \geq S \geq C \geq P$。
- **主体的敏感度标记**称为**许可证级别**。
- **客体的敏感度标记**称为**密级**。

### 19. 强制存取控制机制 ⭐⚠(多选9)

通过对比主体的敏感度标记和客体的敏感度标记，最终确定主体是否能够存取客体：

1. **读规则**：仅当主体的许可证级别 **大于或等于** 客体的密级时，该主体才能读相应的客体。
2. **写规则**：仅当主体的许可证级别 **小于或等于** 客体的密级时，该主体才能写相应的客体。

### 20. 视图机制 (练习16)

- 视图机制间接地实现支持存取谓词的用户权限定义。
- 把要保密的数据对无权存取这些数据的用户隐藏起来，对数据提供一定程度的安全保护。

### 21. 审计 (练习17,18)

- 审计功能将用户对数据库的所有操作自动记录下来，写到**审计日志**。
- 审计员利用审计日志监控数据库中的各种行为，找出非法存取数据的人、时间和内容。

> ⚠ **注意区别审计日志和事务日志**：
> - **审计日志**：记录用户对数据库的**所有操作**。
> - **事务日志**：只记录事务对数据库的**更新操作**。（事务日志：第八章 数据库恢复技术 知识点22）

### 22. 审计功能的可选性 ⚠(判断9)

- 审计很费**时间和空间**。
- DBA 可以打开或关闭审计功能。
- 审计功能主要用于安全性要求较高的部门。

### 23. 审计事件 (练习17)

审计事件一般有多个类别：

| 类别 | 说明 |
| --- | --- |
| **（1）服务器事件** | 审计数据库服务器发生的事件 |
| **（2）系统权限** | 对系统拥有的结构或模式对象进行操作的审计 |
| **（3）语句事件** | 对 SQL 语句，如 DDL、DML、DQL 及 DCL 语句的审计 |
| **（4）模式对象事件** | 对特定模式对象上进行的 SELECT 或 DML 操作的审计。模式对象包括表、视图、存储过程、函数等。**模式对象不包括**依附于表的索引、约束、触发器、分区表等 |

### 24. 审计功能

主要包括以下几方面内容：

1. **基本功能**，提供多种审计查阅方式：基本的、可选的、有限的等。
2. 提供**多套审计规则**，审计规则一般在数据库初始化时设定，以方便审计员管理。
3. 提供**审计分析和报表功能**。
4. **审计日志管理功能**，包括：为防止审计员误删审计记录，审计日志必须先转储后删除；对转储的审计记录文件提供完整性和保密性保护；只允许审计员查阅和转储审计记录，不允许任何用户新增和修改审计记录等。
5. 系统提供查询审计设置及审计记录信息的**专门视图**。对于系统权限级别、语句级别及模式对象级别的审计记录也可通过相关的系统表直接查看。

### 25. 审计功能的设置与取消 (判断10)

- **AUDIT** 语句用来设置审计功能。
- **NOAUDIT** 语句则取消审计功能。

### 26. 用户级审计和系统级审计 (练习18)

| | 用户级审计 | 系统级审计 |
| --- | --- | --- |
| **设置者** | 任何用户可设置 | 只能由**数据库管理员**设置 |
| **内容** | 主要针对自己创建的数据库表或视图进行审计，记录所有用户对这些表或视图的一切成功和不成功的访问要求以及各种类型的 SQL 操作 | 监测成功或失败的登录要求、监测授权和收回操作以及其他数据库级权限下的操作 |

- 审计设置以及审计日志一般都存储在**数据字典**中。

### 27. 数据加密 (练习19,20)

- 数据加密是防止数据库数据在**存储和传输**中失密的有效手段。
- 加密的基本思想：根据一定的算法将原始数据"**明文**"变换为不可直接识别的格式——"**密文**"，从而使得不知道解密算法的人无法获知数据的内容。
- 数据加密主要包括**存储加密**和**传输加密**。

### 28. 存储加密

提供**非透明**和**透明**两种存储加密方式：

| 方式 | 说明 |
| --- | --- |
| **非透明存储加密** | 通过多个加密函数实现 |
| **透明存储加密** | **内核级**加密保护方式，对用户完全透明。数据在写到磁盘时对数据进行加密，授权用户读取数据时再对其进行解密。由于数据加密对用户透明，数据库的应用程序不需要做任何修改，只需在创建表语句中说明需加密的字段即可。当对加密数据进行增、删、改、查询操作时，DBMS 将自动对数据进行加、解密工作。基于数据库内核的数据存储加密、解密方法**性能较好，安全完备性较高** |

### 29. 传输加密

- 在客户/服务器结构中，数据库用户与服务器之间若采用明文方式传输数据，容易被网络恶意用户截获或篡改，存在安全隐患。
- DBMS 提供传输加密功能。
- 常用的传输加密方式如**链路加密**和**端到端加密**。

### 30. 链路加密

- 在**链路层**进行加密。
- 传输信息由**报头**和**报文**两部分组成：**报文和报头均加密**。

### 31. 端到端加密

- 在**发送端加密，接收端解密**。
- **只加密报文不加密报头**。
- 所需密码设备数量相对较少，容易被非法监听者发现并从中获取敏感信息。

### 32. 其他安全性保护措施

- 推理控制以及数据库应用中隐蔽信道和数据隐私保护等技术。

### 33. 数据隐私

- 控制不愿被他人知道或他人不便知道的个人数据的能力。

## 三、公式 / 算法 / 定理速查表

### SQL 授权语句

```sql
GRANT <权限>[, <权限>]...
ON <对象类型> <对象名>[, <对象类型> <对象名>]...
TO <用户>[, <用户>]...
[WITH GRANT OPTION];
```

### SQL 收回权限语句

```sql
REVOKE <权限>[, <权限>]...
ON <对象类型> <对象名>[, <对象类型> <对象名>]...
FROM <用户>[, <用户>]...
[CASCADE | RESTRICT];
```

### 创建用户语句

```sql
CREATE USER <username> [WITH] [DBA|RESOURCE|CONNECT];
```

### 角色相关语句

```sql
-- 创建角色
CREATE ROLE <角色名>

-- 给角色授权
GRANT <权限>[, <权限>]... ON <对象类型> <对象名> TO <角色>[, <角色>]...

-- 将角色授予其他角色或用户
GRANT <角色 1>[, <角色 2>]... TO <角色 3>[, <用户 1>]... [WITH ADMIN OPTION]

-- 收回角色权限
REVOKE <权限>[, <权限>]... ON <对象类型> <对象名> FROM <角色>[, <角色>]...
```

### 审计设置与取消

```sql
AUDIT ...    -- 设置审计功能
NOAUDIT ...  -- 取消审计功能
```

### 强制存取控制（MAC）读写规则

$$
\text{读：主体许可证级别} \geq \text{客体密级}
$$

$$
\text{写：主体许可证级别} \leq \text{客体密级}
$$

### 敏感度标记密级次序

$$
TS \geq S \geq C \geq P
$$

## 四、重点对比与易混辨析

### 1. 自主存取控制（DAC）vs 强制存取控制（MAC）

| 对比项 | DAC（自主存取控制） | MAC（强制存取控制） |
| --- | --- | --- |
| **支持级别** | C2 级 DBMS | B1 级 DBMS |
| **核心机制** | 用户对对象有不同权限，可转授 | 对象标密级，用户授许可证 |
| **灵活性** | 非常灵活 | 相对严格 |
| **安全性** | 较低 | 较高 |
| **数据本身标记** | 无安全性标记 | 有敏感度标记 |
| **缺点** | 可能存在"无意泄露" | [原文未述] |

### 2. WITH GRANT OPTION vs WITH ADMIN OPTION

| 对比项 | WITH GRANT OPTION | WITH ADMIN OPTION |
| --- | --- | --- |
| **适用场景** | GRANT 语句中授予权限给用户 | GRANT 语句中授予角色给用户/角色 |
| **效果** | 获得权限的用户可再将权限授予其他用户 | 获得权限的角色或用户可再将权限授予其他角色 |

### 3. 审计日志 vs 事务日志

| 对比项 | 审计日志 | 事务日志 |
| --- | --- | --- |
| **记录内容** | 用户对数据库的**所有操作** | 事务对数据库的**更新操作** |
| **用途** | 安全审计，找出非法存取 | 数据库恢复 |

### 4. 链路加密 vs 端到端加密

| 对比项 | 链路加密 | 端到端加密 |
| --- | --- | --- |
| **加密层** | 链路层 | 发送端加密，接收端解密 |
| **加密范围** | 报文和报头**均加密** | **只加密报文**，不加密报头 |
| **密码设备** | [原文未述] | 所需密码设备数量相对较少 |
| **缺点** | [原文未述] | 容易被非法监听者发现并从中获取敏感信息 |

### 5. CONNECT / RESOURCE / DBA 三种用户权限

| 权限 | 创建新用户 | 创建模式 | 创建基本表/视图 | 登录数据库 |
| --- | --- | --- | --- | --- |
| **CONNECT** | ✗ | ✗ | ✗ | ✓ |
| **RESOURCE** | ✗ | ✗ | ✓（成为属主） | [原文未述] |
| **DBA** | ✓ | ✓ | ✓ | [原文未述] |

### 6. 用户级审计 vs 系统级审计

| 对比项 | 用户级审计 | 系统级审计 |

| **设置者** | 任何用户 | 仅数据库管理员 |

## 五、高频考点与易错提醒

1. ⚠ **安全性 vs 完整性概念区分**是高频易错点：安全性 = 防止不合法使用造成的数据泄露、更改或破坏；完整性 = 防止错误/非法数据进入数据库（练习1,判断1）。

2. ⚠ **数据库安全性控制的常用方法和技术**：用户标识与鉴别、存取控制、视图机制、审计、数据加密。注意**封锁**是并发控制措施，**数据转储**是恢复技术，**防火墙**是网络安全技术，均不属于数据库安全性控制方法（练习2,19,多选2）。

3. ⚠ **TCSEC/TDI 安全级别**：4 组 7 个等级 D→C1→C2→B1→B2→B3→A1，按可信程度逐渐增高（练习4）。

4. ⚠ **CC 把安全要求分为安全功能要求和安全保证要求**，不是"加密/解密功能要求"（多选3）。

5. ⚠ **GRANT/REVOKE 语法**是高频考点：GRANT...ON...TO...[WITH GRANT OPTION]；REVOKE...ON...FROM...[CASCADE|RESTRICT]（练习7,8,9）。

6. ⚠ **角色间授权用 WITH ADMIN OPTION，不是 WITH GRANT OPTION**——这是高频易错点（判断4）。

7. ⚠ **用户权限由数据库对象和操作类型两个要素组成**，不是"数据对象和数据类型"（判断6）。

8. ⚠ **自主存取控制安全性低于强制存取控制**（判断7）。

9. ⚠ **授权粒度越细，系统检查权限的开销越大**，不是越小（判断3）。

10. ⚠ **审计日志记录所有操作，事务日志只记录更新操作**——二者区别是易错点（知识点21）。

11. ⚠ **审计很费时间和空间**，不是"节约资源"（判断9）。

12. ⚠ **强制存取控制读写规则**：读——主体许可证级别 ≥ 客体密级；写——主体许可证级别 ≤ 客体密级（多选9）。

13. ⚠ **模式对象不包括索引、约束、触发器、分区表**（练习17）。

14. ⚠ **GRANT 和 REVOKE 均属于数据控制语言 DCL**（多选10，反推）。

15. ⚠ **数据库角色是权限的集合**，不是"用户扮演的虚拟人物"（判断8）。

16. ⚠ **DCL（数据控制语言）用于数据访问权限的控制**（练习5，反推）。

## 六、一句话记忆要点

1. **安全性** = 防止不合法使用造成的泄露、更改或破坏（与完整性区分）。
2. **TCSEC/TDI** = 4 组 7 级（D<C1<C2<B1<B2<B3<A1），CC 取代 TCSEC 成为主流。
3. **CC** = 安全功能要求 + 安全保证要求，结构为"类-子类-组件"，EAL1~EAL7。
4. **安全措施层层设防**：用户标识鉴别→DBMS 存取控制→OS 保护→数据加密。
5. **身份鉴别四方式**：静态口令（低安全）→动态口令（一次一密）→生物特征（高安全）→智能卡（PIN+卡）。
6. **存取控制子系统** = 定义用户权限 + 合法权限检查。
7. **DAC（C2级）灵活但可能无意泄露；MAC（B1级）严格，靠密级标记控制。**
8. **GRANT 授权、REVOKE 收回**；WITH GRANT OPTION 允许转授权限。
9. **角色 = 权限集合**；角色间授权用 WITH ADMIN OPTION（不是 GRANT OPTION）。
10. **MAC 读写规则**：读 ≥，写 ≤（主体许可证 vs 客体密级）。
11. **视图机制** = 隐藏保密数据，间接实现存取谓词。
12. **审计日志 ≠ 事务日志**：前者记所有操作，后者只记更新操作；审计费时费空间，可由 DBA 开关。
13. **AUDIT 设置 / NOAUDIT 取消**；用户级审计任何用户可设，系统级审计仅 DBA 可设。
14. **数据加密** = 明文→密文；存储加密（透明/非透明）+ 传输加密（链路加密/端到端加密）。
15. **链路加密**报文报头都加密；**端到端加密**只加密报文。
