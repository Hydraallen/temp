# 软件设计与开发 · 可行性及需求分析 — 复习资料

> 来源：6-3-软件设计与开发计算机类讲义.pdf 物理页 [28, 52]
> 说明：⭐=高频考点，⚠=易错点；> Caveat: 为陷阱/技巧；末尾 Worked Examples 为代表性题目（无书内答案的标 [PROPOSED — verify]）。

## 一、本章概览

本章对应软件生命周期"定义时期"的前两个关键阶段——**可行性研究**与**需求分析**，并延伸到**面向对象分析（OOA）**。可行性研究解决"问题是否值得解、能否解"，需求分析解决"系统必须做什么"，OOA 则用对象/动态/功能三大模型从面向对象视角建模。全章涉及大量图形工具（系统流程图、数据流图、ER 图、状态图、用例图、类图等），是考试图形密集区。*(讲义 p28)*

## 二、可行性研究

### 1. 可行性研究的任务 ⭐

**定义**：可行性研究的目的，就是用最小的代价在尽可能短的时间内确定问题是否能够解决。*(讲义 p28)*

至少从以下 3 个方面研究每种解法的可行性：*(讲义 p28)*

| 方面 | 核心问题 | 说明 |
|------|----------|------|
| 技术可行性 | 使用现有的技术能实现这个系统吗？ | 度量特定技术信息系统解决方案的实用性及技术资源的可用性 |
| 经济可行性 | 这个系统的经济效益能超过它的开发成本吗？ | — |
| 操作可行性 | 系统的操作方式在这个用户组织内行得通吗？ | — |

- 必要时还应从**法律、社会效益**等更广泛的方面研究。*(讲义 p28)*
- 可行性研究成本通常只占预期工程总成本的 **5%～10%**。*(讲义 p28)*

> Caveat (判断1): 可行性研究的目的是"确定问题是否能够解决"，**不是**"规划如何开发项目"。*(题目判断1)*

### 2. 可行性研究过程

典型的可行性研究过程有 **8 个步骤**：*(讲义 p28-29)*

| 步骤 | 内容要点 |
|------|----------|
| (1) 复查系统规模和目标 | 确保分析员正在解决的问题确实是要求他解决的问题 |
| (2) 研究目前正在使用的系统 | 现有系统是信息的重要来源；新系统必须能完成旧系统的基本功能，且解决旧系统的缺点 |
| (3) 导出新系统的高层逻辑模型 | 从现有物理系统→现有逻辑模型→目标逻辑模型→新物理系统 |
| (4) 进一步定义问题 | 前 4 步构成循环：定义问题→分析→导出试探性解→再定义→再分析→修改解，直到逻辑模型完全符合系统目标 |
| (5) 导出和评价供选择的解法 | 从逻辑模型导出若干物理解法；考虑操作可行性→经济可行性→为可行方案制定实现进度表 |
| (6) 推荐行动方针 | 关键问题：是否继续开发？选择最好解法并说明理由；需进行成本/效益分析 |
| (7) 草拟开发计划 | 制定工程进度表、估计人员和资源需要、估计各阶段成本、给出需求分析阶段的详细进度表和成本估计 |
| (8) 书写文档提交审查 | 写成清晰文档，请用户/客户负责人/评审组审查 |

> Caveat (多选1): 步骤(7)草拟的开发计划**不需要制定得很详细**，通常只需估计生命周期每个阶段的工作量；选项"制定详细的进度计划"不属于可行性分析阶段。*(题目多选1)*

### 3. 成本/效益分析 ⭐

软件开发成本主要表现为人力消耗。成本估计不是精确的科学，应使用几种不同的估计技术相互检验。*(讲义 p34)*

**3 种估算技术**：*(讲义 p34)*

| 技术 | 方法 | 适用条件 |
|------|------|----------|
| 代码行技术 | 把开发每个功能的成本和源代码行数联系起来；每行代码平均成本 × 行数 | 有以往类似工程历史数据时非常有效 |
| 任务分解技术 | 把工程分解为若干相对独立的任务，分别估计成本后累加 | 最常用按开发阶段划分任务 |
| 自动估计成本技术 | 采用自动估计成本的软件工具 | 需长期搜集的大量历史数据 + 良好数据库系统支持 |

**成本/效益分析 4 个方面**：*(讲义 p34-35)*

**(1) 货币的时间价值**——用利率表示：*(讲义 p35)*

$$F = P(1 + i)^n$$

其中 $P$ 为现在存入金额，$i$ 为年利率，$n$ 为年数，$F$ 为 $n$ 年后可得金额。

**(2) 投资回收期**——使累计经济效益等于最初投资所需要的时间。回收期越短越值得投资。*(讲义 p35)*

**(3) 纯收入**——整个生命周期内累计经济效益（折合现在值）与投资之差。*(讲义 p35)*

**(4) 投资回收率**——衡量投资效益大小，可与年利率相比较。*(讲义 p35)*

## 三、系统流程图 ⭐

**定义**：系统流程图是概括地描绘**物理系统**的传统工具。用图形符号以黑盒子形式描绘组成系统的每个部件（程序、文档、数据库、人工过程等）。*(讲义 p29)*

**关键辨析**：系统流程图表达的是**数据在系统各部件之间流动的情况**，而不是对数据进行加工处理的控制过程。尽管某些符号和程序流程图形式相同，但它却是**物理数据流图而不是程序流程图**。*(讲义 p29)*

> Caveat (单选2): 系统流程图描述的是**物理系统**，不是逻辑系统、程序系统或体系结构。*(题目单选2)*

**基本符号**：*(讲义 p29)*

| 符号 | 名称 | 说明 |
|------|------|------|
| ▭（矩形） | 处理 | 能改变数据值或数据位置的加工或部件 |
| ◇（菱形） | 输入输出 | 广义的不指明具体设备的输入/输出符号 |
| ○（圆圈） | 连接 | 转到图的另一部分或从另一部分转来（同一页） |
| ▽（换页连接） | 换页连接 | 转到另一页图上或由另一页转来 |
| →（箭头） | 数据流 | 指明数据流动方向 |

**系统符号**（部分）：*(讲义 p30)*

| 符号 | 名称 | 说明 |
|------|------|------|
| 矩形 | 穿孔卡片 | 穿孔卡片输入/输出或文件 |
| 文档形状 | 文档 | 打印输出或打印终端输入 |
| 圆圈 | 磁带 | 磁带输入输出或磁带文件 |
| 联机存储形状 | 联机存储 | 任何种类的联机存储（磁盘、磁鼓、软盘等） |
| 圆柱 | 磁盘 | 磁盘输入输出或磁盘文件/数据库 |
| 显示器形状 | 显示 | CRT 终端或类似显示部件 |
| 人工输入形状 | 人工输入 | 人工输入数据的脱机处理 |
| 人工操作形状 | 人工操作 | 人工完成的处理 |
| 正方形 | 辅助操作 | 使用设备进行的脱机操作 |
| 波浪线 | 通信链路 | 通过远程通信线路传送数据 |

**分层描绘**：面对复杂系统时，先用一张高层次系统流程图描绘总体概貌，再分别把每个关键功能扩展到适当详细程度画在单独的页上。*(讲义 p31)*

## 四、数据流图（DFD） ⭐⚠

**定义**：数据流图（DFD）是一种图形化技术，它描绘信息流和数据从输入移动到输出的过程中所经受的变换。数据流图中没有任何具体的物理部件，它只描绘数据在软件中流动和被处理的**逻辑过程**。*(讲义 p31)*

**特点**：*(讲义 p31)*
- 是系统**逻辑功能**的图形表示，是分析员与用户之间极好的**通信工具**。
- 设计时只需考虑系统必须完成的基本逻辑功能，完全不需要考虑怎样具体实现。

**四种基本符号**：*(讲义 p31)*

| 符号 | 含义 | 说明 |
|------|------|------|
| 正方形 | 数据的源点或终点 | 软件系统外部环境中的实体（人员、组织或其他软件系统） |
| 圆角矩形 | 变换数据的处理 | — |
| 开口矩形 | 数据存储 | 可以是数据库文件或任何形式的数据组织 |
| 箭头 | 数据流 | 特定数据的流动方向 |

> Caveat (单选4): 在数据流图中，**不能由计算机处理**的成分是**数据源/终点**（它们是外部环境中的实体）。*(题目单选4)*

> Caveat (单选10): 数据流图是**功能模型**工具，**不是数据模型**工具；类图、ER 图、数据字典才是数据模型工具。*(题目单选10)*

**分层描述**：*(讲义 p31-32)*
- 任何系统的基本模型都由若干数据源点/终点以及一个处理组成。
- 从高层次的基本系统模型开始画，再逐层分解。
- **顶层数据流图**列出了所有给定的数据源点/终点，描述了系统的**输入与输出**。

> Caveat (单选3): 顶层数据流图描述了系统的**输入与输出**。*(题目单选3)*

**数据流图的用途**：*(讲义 p32)*
1. 作为交流信息的工具。
2. 作为分析和设计的工具。
3. 辅助物理系统设计时，以图中不同处理的定时要求为指南，画出自动化边界，每组边界可能意味着一个不同的物理系统。

## 五、数据字典 ⭐⚠

**定义**：数据字典是关于数据的信息的集合，也就是对数据流图中包含的所有元素的定义的集合。*(讲义 p33)*

**数据字典由 4 类元素的定义组成**：*(讲义 p33)*

| 元素 | 说明 |
|------|------|
| 数据流 | — |
| 数据流分量 | 即数据项 |
| 数据存储 | — |
| 处理 | — |

> Caveat (单选7): 数据字典由 4 类元素组成，**不包括外部实体**（外部实体是数据流图的成分）。*(题目单选7)*

**数据字典中记录的信息**：一般信息（名字、别名、描述等）、定义（数据类型、长度、结构等）、使用特点（值的范围、使用频率、使用方式）、控制信息（来源、用户、使用程序、改变权、使用权等）、分组信息（父结构、从属结构、物理位置等）。*(讲义 p33)*

**别名的 3 个原因**：*(讲义 p33)*
1. 不同用户对同样数据使用了不同名字。
2. 一个分析员在不同时期对同一数据使用了不同名字。
3. 两个分析员分别分析同一数据流时使用了不同名字。

**由数据元素组成数据的 4 种基本方式**：*(讲义 p33-34)*

| 方式 | 说明 |
|------|------|
| 顺序 | 以确定次序连接两个或多个分量 |
| 选择 | 从两个或多个可能的元素中选取一个 |
| 重复 | 把指定的分量重复零次或多次 |
| 可选 | 一个分量是可有可无的（当重复上下限为 1 和 0 时的特例，单独列出使字典更清晰） |

**数据字典符号**：*(讲义 p34)*

| 符号 | 含义 |
|------|------|
| `=` | 等价于（或定义为） |
| `+` | 和（连接两个分量） |
| `[ ]` | 或（方括弧中的分量选一个） |
| `{ }` | 重复（重复花括弧中的分量） |
| `( )` | 可选（圆括弧里的分量可有可无） |

**示例**——标识符的定义：*(讲义 p34)*
```
标识符 = 字母字符 + 字母数字串
字母数字串 = 0 {字母或数字} 7
字母或数字 = [字母字符 | 数字字符]
```

**数据字典的用途**：*(讲义 p34)*
1. 最重要的用途是作为分析阶段的工具。
2. 包含的每个数据元素的控制信息很有价值。
3. 是开发数据库的第一步，而且是很有价值的一步。

## 六、需求分析

### 1. 需求分析的任务 ⭐⚠

**定义**：需求分析是软件定义时期的最后一个阶段，它的基本任务是准确地回答"系统必须做什么"这个问题。*(讲义 p35)*

> Caveat (判断7): 需求分析只解决"做什么"的问题，**不考虑"怎么做"**（怎么做是设计阶段的任务）。*(题目判断7)*

**需求分析的 4 条准则**：*(讲义 p35)*

| 准则 | 对应模型 |
|------|----------|
| 必须理解并描述问题的信息域 | 建立数据模型 |
| 必须定义软件应完成的功能 | 建立功能模型 |
| 必须描述作为外部事件结果的软件行为 | 建立行为模型 |
| 必须对描述信息、功能和行为的模型进行分解 | 用层次的方式展示细节 |

**需求分析阶段的 4 项主要任务**：*(讲义 p35-36)*

**(1) 确定对系统的综合要求**——8 个方面：*(讲义 p35-36)*

| 需求类型 | 说明 |
|----------|------|
| 功能需求 | 指定系统必须提供的服务 |
| 性能需求 | 定时约束或容量约束（速度/响应时间、信息量速率、主存容量、磁盘容量、安全性等） |
| 可靠性和可用性需求 | 定量指定系统可靠性；可用性量化用户可使用系统的程度 |
| 出错处理需求 | 系统对环境错误（非本身造成的）应怎样响应 |
| 接口需求 | 与环境通信的格式（用户接口、硬件接口、软件接口、通信接口） |
| 约束 | 设计或实现时应遵守的限制条件（精度、工具和语言约束、设计约束、标准、硬件平台） |
| 逆向需求 | 说明软件系统**不应该做什么** |
| 将来可能提出的要求 | 为将来扩充和修改预做准备 |

> Caveat (单选12): "软件产品必须能够在3秒内对用户请求作出响应"属于**性能需求**（定时约束）。*(题目单选12)*

**(2) 分析系统的数据要求**——利用数据字典全面准确定义数据，利用图形工具辅助描绘数据结构以提高可理解性。*(讲义 p36)*

**(3) 导出系统的逻辑模型**——通常用数据流图、实体-联系图、状态转换图、数据字典和主要的处理算法描述。*(讲义 p36)*

**(4) 修正系统开发计划**——根据分析过程中获得的更深入具体的了解，较准确地估计成本和进度。*(讲义 p36)*

### 2. 与用户沟通获取需求的方法 ⭐

**4 种方法**：*(讲义 p36-37)*

| 方法 | 要点 |
|------|------|
| 访谈 | 最早且仍广泛使用；正式访谈（事先准备具体问题）和非正式访谈（开放性问题鼓励自由回答）；常配合**情景分析技术** |
| 面向数据流自顶向下求精 | 结构化分析方法；从数据流图输出端着手，沿数据流图从输出端往输入端回溯，确定每个数据元素的来源 |
| 简易的应用规格说明技术 | 面向团队的需求收集方法；开发者与用户不分彼此、密切合作；典型过程：初步访谈→各自写产品需求→开会讨论→制定小型规格说明→起草完整需求规格说明 |
| 快速建立软件原型 | 快速建立旨在演示主要功能的可运行程序；特性：快速、容易修改；工具：第四代技术、可重用软件构件、形式化规格说明和原型环境 |

**情景分析技术的 2 个用处**：*(讲义 p36)*
1. 能在某种程度上演示目标系统的行为，便于用户理解，可能进一步揭示分析员目前还不知道的需求。
2. 保证用户在需求分析过程中始终扮演积极主动的角色。

> Caveat (单选14): 结构化分析方法就是**面向数据流**自顶向下逐步求精进行需求分析的方法。*(题目单选14)*

### 3. 分析建模与规格说明 ⭐

**模型定义**：模型就是为了理解事物而对事物作出的一种抽象，是对事物的一种无歧义的书面描述。*(讲义 p37)*

**3 种模型及其工具**：*(讲义 p37-38)*

| 模型 | 描述内容 | 对应图形工具 |
|------|----------|-------------|
| 数据模型 | 数据对象及数据对象之间的关系 | 实体-联系图（ER 图） |
| 功能模型 | 系统应完成的功能 | 数据流图 |
| 行为模型 | 系统的各种行为模式和状态间转换 | 状态转换图 |

**软件需求规格说明**：需求分析阶段得出的**最主要的文档**。通常用自然语言完整、准确、具体地描述系统的数据要求、功能需求、性能需求、可靠性和可用性要求、出错处理需求、接口需求、约束、逆向需求以及将来可能提出的要求。*(讲义 p38)*

> Caveat (单选8): 需求分析阶段最重要的技术文档是**需求规格说明书**。*(题目单选8)*

> Caveat (多选5): 需求规格说明书的作用包括：软件验收的依据、用户与开发人员交流的工具、软件设计的依据；**不是**可行性研究的依据（可行性研究在需求分析之前）。*(题目多选5)*

## 七、实体联系图（ER 图） ⭐

**定义**：概念性数据模型（信息模型）是一种面向问题的数据模型，按照用户的观点对数据建立的模型，描述从用户角度看到的数据，反映用户的现实环境，与在软件系统中的实现方法无关。*(讲义 p38)*

**数据模型的 3 种信息**：*(讲义 p38)*
- **数据对象**：对软件必须理解的复合信息的抽象（外部实体、事物、行为、事件、角色、单位、地点或结构等）。
- **属性**：定义数据对象的性质。必须把一个或多个属性定义为"标识符"（关键字/键）。
- **联系（关系）**：数据对象彼此间相互连接的方式。

**联系的 3 种类型**：*(讲义 p38)*

| 类型 | 说明 |
|------|------|
| 一对一联系（1∶1） | — |
| 一对多联系（1∶N） | — |
| 多对多联系（M∶N） | — |

**ER 图的 3 种基本成分及符号**：*(讲义 p38)*

| 成分 | 符号 |
|------|------|
| 实体（即数据对象，实体集） | 矩形框 |
| 联系（联系集） | 菱形框，连接相关实体 |
| 属性 | 椭圆形或圆角矩形，用直线与实体/联系连接 |

> Caveat (单选13): ER 图中，**菱形**表示**联系**。*(题目单选13)*

## 八、状态转换图 ⭐

**定义**：状态转换图（简称状态图）通过描绘系统的**状态**及引起系统状态转换的**事件**，来表示系统的**行为**。还指明了作为特定事件的结果系统将做哪些动作。*(讲义 p39)*

**3 个核心概念**：*(讲义 p39)*

| 概念 | 定义 |
|------|------|
| 状态 | 任何可以被观察到的系统行为模式；状态规定了系统对事件的响应方式 |
| 事件 | 在某个特定时刻发生的事情，是对引起系统做动作或从一个状态转换到另一个状态的外界事件的抽象 |
| 符号 | 见下表 |

**状态图符号**：*(讲义 p39)*

| 状态类型 | 符号 | 数量限制 |
|----------|------|----------|
| 初态（初始状态） | 实心圆 | 只能有 **1** 个 |
| 终态（最终状态） | 一对同心圆（内圆为实心圆） | 可以有 **0 至多个** |
| 中间状态 | 圆角矩形 | — |

**中间状态的 3 部分**（用两条水平横线分隔）：*(讲义 p39)*
- 上面部分：状态名称（必须有的）
- 中间部分：状态变量的名字和值（可选）
- 下面部分：活动表（可选）

**活动表语法**：*(讲义 p39)*
```
事件名(参数表)/动作表达式
```

**3 种标准事件**：*(讲义 p39)*

| 事件 | 含义 |
|------|------|
| `entry` | 指定**进入**该状态的动作 |
| `exit` | 指定**退出**该状态的动作 |
| `do` | 指定在该状态**下**的动作 |

> Caveat (单选20): 在状态图中，用**圆角矩形**表示**中间状态**。*(题目单选20)*

> Caveat (单选11): 状态图是需求分析**行为模型**的工具；数据流图是功能模型工具；ER 图和数据字典是数据模型工具。*(题目单选11)*

## 九、需求分析的其他图形工具

### 1. 层次方框图

用**树形结构**的一系列多层次的矩形框描绘数据的层次结构。*(讲义 p40)*

### 2. Warnier 图

也用树形结构描绘信息，但比层次方框图提供更丰富的描绘手段。可以表明信息的逻辑组织——指出一类信息或一个信息元素是**重复出现**的，也可以表示特定信息在某一类信息中是**有条件地出现**的。*(讲义 p40-41)*

### 3. IPO 图

输入、处理、输出图的简称，由美国 IBM 公司发展完善，能够方便地描绘输入数据、对数据的处理和输出数据之间的关系。*(讲义 p41)*

## 十、验证软件需求 ⭐

需求分析阶段的工作结果是开发软件系统的重要基础。一旦对目标系统提出一组要求之后，必须严格验证这些需求的正确性。*(讲义 p41)*

**从 4 个方面验证**：*(讲义 p41)*

| 验证方面 | 含义 |
|----------|------|
| 一致性 | 所有需求必须是一致的，任何一条需求不能和其他需求互相矛盾 |
| 完整性 | 需求必须是完整的，规格说明书应该包括用户需要的每一个功能或性能 |
| 现实性 | 指定的需求应该是用现有的硬件技术和软件技术基本上可以实现的 |
| 有效性 | 必须证明需求是正确有效的，确实能解决用户面对的问题 |

> Caveat (单选15): "需求确实能解决用户的问题"属于需求验证的**有效性**。*(题目单选15)*

## 十一、面向对象分析（OOA） ⭐

**定义**：面向对象分析（OOA）的关键是识别出问题域内的类与对象，并分析它们相互间的关系，最终建立起问题域的简洁、精确、可理解的正确模型。*(讲义 p41)*

**3 种模型**：*(讲义 p41)*

| 模型 | 性质 | 描述内容 | 建模工具 |
|------|------|----------|----------|
| 对象模型 | 静态的、结构化的"数据"性质 | 模拟客观世界实体的对象及对象彼此间的关系，描述系统的静态结构 | UML 类图 |
| 动态模型 | 瞬时的、行为化的"控制"性质 | 系统的行为 | 序列图、活动图、状态图 |
| 功能模型 | 变化的"功能"性质 | 系统应该"做什么"，直接反映用户需求 | 数据流图、用例图 |

> Caveat (单选16): 面向对象分析需要建立 3 种模型：对象模型、动态模型、功能模型。**ER 模型不属于**这 3 种模型。*(题目单选16)*

> Caveat (判断10): 顺序图是建立**动态模型**的工具，不是功能模型。*(题目判断10)*

**对象模型是最基本、最重要、最核心的模型。***(讲义 p41)*

复杂系统的对象模型通常由 **5 个层次**组成：*(讲义 p42)*
1. 主题层
2. 类与对象层
3. 结构层
4. 属性层
5. 服务层

### 类图的基本符号

类图包含 3 部分：类名、属性、服务。接口用 `<<interface>>` 标识。*(讲义 p42)*

### 类与类之间的 4 种关系 ⭐

| 关系 | 说明 | 表示符号 |
|------|------|----------|
| 关联 | 类与类之间存在连接关系；普通关联用连接两个类之间的**直线**表示 | 直线 |
| 泛化（继承） | 通用元素和具体元素之间的分类关系；具体元素完全拥有通用元素的信息并可附加其他信息 | 一端为**空心三角形**的连线，三角形顶角紧挨通用元素 |
| 依赖 | 两个模型元素之间的语义连接：一个独立，另一个依赖于它；独立元素改变将影响依赖元素 | 虚线箭头（可标 `<<友元>>` 等） |
| 细化/实现 | 对同一事物在不同抽象层次上描述时，描述之间具有细化关系 | 虚线箭头（可标 `<<interface>>`） |

> Caveat (单选19): 在类图中，**无箭头实线**线段代表的是**关联**。*(题目单选19)*

**聚集（聚合）**——关联的特例，表示整体与部分的关系：*(讲义 p42-43)*

| 类型 | 说明 |
|------|------|
| 共享聚集 | 部分方对象可同时参与多个整体方对象的构成 |
| 组合聚集（组成） | 部分类完全隶属于整体类，部分与整体共存，整体不存在则部分也随之消失 |

### 动态模型工具

**序列图（Sequence Diagram）**：对象之间基于时间顺序的动态交互，显示随着时间变化对象之间如何通讯。主要用途之一是更详细地描述用例表达的需求。*(讲义 p44)*

序列图中的消息类型：*(讲义 p44)*
- 调用消息
- 异步消息
- 返回消息
- 对象生命线
- 对象销毁（`<<create>>`、`destroy`）

**活动图**：描述系统行为的模型视图，用来描述系统的工作流程和并发行为，展现参与系统某一行为的类进行的各种活动的顺序关系。*(讲义 p44)*

活动图元素：开始、动作、并发分支、分支（监护表达式如 `[not complete]`、`[else]`）、并发汇合、结束。*(讲义 p44)*

### 功能模型与用例图 ⭐

功能模型表示变化的系统的"功能"性质，指明系统应该"做什么"。在 UML 中用**用例图**是进行需求分析和建立功能模型的强有力工具。*(讲义 p44)*

**用例定义**：一个用例是可以被行为者感受到的、系统的一个完整的功能。在 UML 中把用例定义成系统完成的一系列动作，动作的结果能被特定的行为者察觉到。*(讲义 p44-45)*

**关键规则**：一个用例必须至少与一个行为者相关联。*(讲义 p45)*

**用例图中的 4 种关系**：*(讲义 p45)*

| 关系类型 | 说明 | 表示符号 |
|----------|------|----------|
| 关联 | 参与者与用例之间的关系 | 实线 |
| 泛化 | 参与者之间或用例之间的关系 | 实线箭头 |
| 包含 | 用例之间的关系 | 虚线箭头，标 `<<includes>>` |
| 扩展 | 用例之间的关系 | 虚线箭头，标 `<<extends>>` |

> Caveat (单选17): 用例图是从用户使用系统的角度描述系统功能的图形表达方法。*(题目单选17)*

> Caveat (单选18): 用例图中涉及的关系有：关联、泛化、包含、扩展。**关联**描述的是行为者和用例之间的关系，不是用例之间的关系。*(题目单选18)*

## 十二、重点对比与易混辨析

### 数据流图 vs 系统流程图

| 对比项 | 系统流程图 | 数据流图 |
|--------|-----------|----------|
| 描绘对象 | **物理系统** | **逻辑过程** |
| 是否含物理部件 | 是（以黑盒子形式描绘物理部件） | 否（没有任何具体物理部件） |
| 表达内容 | 数据在系统各部件之间流动 | 数据在软件中流动和被处理的逻辑过程 |
| 主要用途 | 概括描绘物理系统 | 分析员与用户之间的通信工具；分析和设计工具 |

### 三种模型与工具映射总表

| 模型 | 性质 | 结构化分析工具 | 面向对象分析工具 |
|------|------|---------------|-----------------|
| 数据模型 | 静态/结构 | 数据字典、ER 图 | 类图（对象模型） |
| 功能模型 | 变化/功能 | 数据流图 | 用例图 |
| 行为模型 | 瞬时/控制 | 状态转换图 | 状态图、序列图、活动图 |

> Caveat: 数据流图在结构化分析中是**功能模型**工具，不是数据模型工具。*(题目单选10)*

## 十三、高频考点与易错提醒

1. **可行性研究目的**：用最小代价在尽可能短时间内确定问题**是否能够解决**，不是规划如何开发。*(题目判断1)*
2. **可行性研究 3 方面**：技术、经济、操作（必要时含法律、社会效益）。*(题目判断2)*
3. **系统流程图**描绘的是**物理系统**，不是逻辑系统。*(题目单选2)*
4. **数据流图 4 种符号**：正方形=源点/终点，圆角矩形=处理，开口矩形=存储，箭头=数据流。*(题目单选5-6)*
5. **顶层数据流图**描述系统的**输入与输出**。*(题目单选3)*
6. **数据字典 4 类元素**：数据流、数据流分量、数据存储、处理——**不含外部实体**。*(题目单选7)*
7. **需求分析**只解决"做什么"，**不考虑"怎么做"**。*(题目判断7)*
8. **需求规格说明书**是需求分析阶段最重要的技术文档，是软件验收依据、交流工具、设计依据，但**不是可行性研究依据**。*(题目单选8, 多选5)*
9. **行为模型工具**是状态图；功能模型工具是数据流图/用例图；数据模型工具是 ER 图/数据字典/类图。*(题目单选10-11)*
10. **ER 图符号**：矩形=实体，菱形=联系，椭圆/圆角矩形=属性。*(题目单选13)*
11. **结构化分析**=面向**数据流**自顶向下逐步求精。*(题目单选14)*
12. **验证需求 4 方面**：一致性、完整性、现实性、有效性。"确实能解决用户问题"=有效性。*(题目单选15)*
13. **OOA 三模型**：对象模型（类图）、动态模型（状态图/序列图/活动图）、功能模型（用例图/数据流图）。**ER 模型不属于**。*(题目单选16)*
14. **用例图 4 种关系**：关联（参与者↔用例）、泛化、包含、扩展。*(题目单选18)*
15. **类图中无箭头实线**=关联。*(题目单选19)*
16. **状态图中间状态**用圆角矩形表示；初态用实心圆，终态用同心圆。*(题目单选20)*
17. **顺序图**是建立**动态模型**的工具，不是功能模型。*(题目判断10)*

## 十四、一句话记忆要点

- **可行性研究**：最小代价、最短时间，确定"能不能解、值不值得解"——技术/经济/操作三方面。
- **系统流程图**：描绘**物理系统**，黑盒子画部件，是物理数据流图而非程序流程图。
- **数据流图**：4 符号（方=源/终、圆角=处理、开口=存储、箭头=流），逻辑功能图形表示，顶层图=输入输出，是**功能模型**工具。
- **数据字典**：4 元素（流、分量、存储、处理），**不含外部实体**；5 符号（=、+、[]、{}、()）。
- **成本/效益**：$F = P(1+i)^n$；投资回收期越短越好；纯收入=累计效益现值−投资。
- **需求分析**：只回答"做什么"；4 准则→数据/功能/行为 3 模型+层次分解；8 项综合要求。
- **ER 图**：矩形=实体、菱形=联系、椭圆=属性；1∶1、1∶N、M∶N。
- **状态图**：实心圆=初态（仅1个）、同心圆=终态（0~多个）、圆角矩形=中间状态；entry/exit/do。
- **验证需求**：一致性、完整性、现实性、有效性。
- **OOA**：对象模型（类图，最核心）、动态模型（序列图/活动图/状态图）、功能模型（用例图/数据流图）。
- **类图 4 关系**：关联（直线）、泛化（空心三角）、依赖（虚线）、细化/实现。
- **用例图 4 关系**：关联、泛化、包含、扩展。
