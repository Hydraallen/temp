## 一、本章概览

系统设计是软件工程中承上启下的核心阶段，回答"系统应该如何实现"的问题。本章分为三大板块：**总体设计**（概要设计，确定软件结构、模块划分与接口）、**详细设计**（确定每个模块的具体实现算法与程序"蓝图"）、**面向对象设计**（OO 设计准则、基本原则、设计模式与软件重用）。总体设计包含设计过程九步骤、五大设计原理（模块化/抽象/逐步求精/信息隐藏/模块独立）、七条启发规则、面向数据流的设计方法（变换分析/事务分析）；详细设计涵盖结构程序设计、人机界面设计、六种过程设计工具、面向数据结构的设计方法（Jackson 法）以及程序复杂度定量度量（McCabe/Halstead）；面向对象设计部分覆盖六大准则、六条启发规则、五大基本原则、23 种设计模式及软件重用体系。本章与第二章（需求分析，产出数据流图）紧密衔接，为后续编码与测试阶段奠定基础。

## 二、知识点详解

### 1. 总体设计的任务 ⭐(练习1,2,多选1,判断1)

- **定义**：总体设计的基本目的就是回答"概括地说，系统应该如何实现"这个问题，因此，总体设计又称为概要设计或初步设计。
- **要点**：
  - 总体设计阶段的另一项重要任务是**设计软件的结构**，也就是要确定系统中每个程序是由哪些模块组成的，以及这些模块相互间的关系、接口。
  - 数据库设计属于总体设计的基本任务之一。
  - 总体设计过程通常由两个主要阶段组成：**系统设计阶段**（确定系统的具体实现方案）和**结构设计阶段**（确定软件结构）。

---

### 2. 总体设计过程

- **定义**：总体设计过程通常由两个主要阶段组成：系统设计阶段，确定系统的具体实现方案；结构设计阶段，确定软件结构。
- **九个步骤**：

  | 步骤 | 内容 |
  |------|------|
  | 1. 设想供选择的方案 | 以数据流图为出发点，设想把 DFD 中的处理分组的各种可能方法，抛弃技术上行不通的分组方法 |
  | 2. 选取合理的方案 | 至少选取低成本、中等成本和高成本 3 种方案；每个合理方案准备 4 份资料：系统流程图、组成系统的物理元素清单、成本/效益分析、实现进度计划 |
  | 3. 推荐最佳方案 | 用户和技术专家审查，使用部门负责人审批后进入结构设计 |
  | 4. 功能分解 | 从实现角度把复杂功能进一步分解为一系列比较简单的功能 |
  | 5. 设计软件结构 | 把模块组织成良好的层次系统，顶层调用下层，最下层完成最具体功能 |
  | 6. 设计数据库 | 在需求分析阶段确定的系统数据需求基础上，进一步设计数据库 |
  | 7. 制定测试计划 | 在软件开发早期阶段考虑测试问题，促使设计时注意提高可测试性 |
  | 8. 书写文档 | 系统说明、用户手册、测试计划、详细的实现计划、数据库设计结果 |
  | 9. 审查和复审 | 技术审查通过后由客户从管理角度进行复审 |

### 3. 总体设计原理 ⭐⚠(练习4,5,6,7,8,9,多选2,4,判断2,3,4,5)

#### 3.1 模块化

- **定义**：模块是由边界元素限定的相邻程序元素（例如，数据说明，可执行的语句）的序列，而且有一个总体标识符代表它。模块是构成程序的基本构件。
- 模块化就是把程序划分成独立命名且可独立访问的模块，每个模块完成一个子功能，集成起来构成整体完成指定功能。
- 模块化是为了使一个复杂的大型程序能被人的智力所管理，是软件应该具备的唯一属性。
- **论证**（数学不等式）：
  - 设 $C(x)$ 定义问题 $x$ 的复杂程度，$E(x)$ 确定解决问题 $x$ 需要的工作量。对于两个问题 $P_1$ 和 $P_2$，如果 $C(P_1)>C(P_2)$，显然 $E(P_1)>E(P_2)$。
  - $C(P_1+P_2)>C(P_1)+C(P_2)$
  - $E(P_1+P_2)>E(P_1)+E(P_2)$
  - 这个不等式导致"各个击破"的结论——把复杂的问题分解成许多容易解决的小问题。
- **注意**：无限分割不成立——模块数目增加时单个模块规模减小、成本减少，但模块间接口工作量增加。每个程序都有一个最适当的模块数目 $M$，使系统开发成本最小。

#### 3.2 抽象

- 抽象就是抽出事物的本质特性而暂时不考虑它们的细节。
- 软件工程过程的每一步都是对软件解法的抽象层次的一次精化。由总体设计向详细设计过渡时，抽象程度随之减少；源程序写出后达到抽象的最低层。

#### 3.3 逐步求精

- **定义**：为了能集中精力解决主要问题而尽量推迟对问题细节的考虑。
- 最初由 Niklaus Wirth 提出，是一种自顶向下的设计策略。
- 求精实际上是细化过程，抽象与求精是一对互补的概念。

#### 3.4 信息隐藏和局部化

- **信息隐藏原理**：应该这样设计和确定模块，使得一个模块内包含的信息（过程和数据）对于不需要这些信息的模块来说，是不能访问的。
- **局部化**：把一些关系密切的软件元素物理地放得彼此靠近。
- 在测试期间和以后的软件维护期间需要修改软件时，使用信息隐藏原理作为模块化系统设计的标准会带来极大好处。

#### 3.5 模块独立 ⭐⚠(练习6,7,8,9,多选2,4,判断2,3,4,5)

- 模块独立的概念是模块化、抽象、信息隐藏和局部化概念的直接结果。
- 模块独立的重要性：有效的模块化的软件比较容易开发出来；独立的模块比较容易测试和维护。
- 模块的独立程度由两个定性标准度量：**内聚**和**耦合**。

##### （1）耦合

耦合是对一个软件结构内不同模块之间互连程度的度量。耦合强弱取决于模块间接口的复杂程度，进入或访问一个模块的点，以及通过接口的数据。

| 耦合类型 | 程度 | 说明 |
|----------|------|------|
| 数据耦合 | 低耦合 | 两个模块彼此间通过参数交换信息，且交换的信息仅仅是数据。系统中至少必须存在这种耦合 |
| 控制耦合 | 中等程度 | 传递的信息中有控制信息（尽管有时以数据形式出现） |
| 特征耦合 | — | 当把整个数据结构作为参数传递而被调用的模块只需要使用其中一部分数据元素时 |
| 公共环境耦合 | 随模块个数变化 | 两个或多个模块通过一个公共数据环境相互作用（全程变量、共享通信区、内存公共覆盖区、文件、物理设备等）。只有两个模块时：一个送一个取=比较松散；都送都取=介于数据耦合和控制耦合之间 |
| 内容耦合 | 最高程度 | ①一个模块访问另一个模块的内部数据；②一个模块不通过正常入口而转到另一个模块的内部；③两个模块有一部分程序代码重叠（只可能出现在汇编程序中）；④一个模块有多个入口（意味着一个模块有几种功能） |

- **设计原则**：尽量使用数据耦合，少用控制耦合和特征耦合，限制公共环境耦合的范围，完全不用内容耦合。

##### （2）内聚

内聚衡量一个模块内部各个元素彼此结合的紧密程度。理想内聚的模块只做一件事情。内聚和耦合密切相关，模块内的高内聚往往意味着模块间的松耦合。

| 类别 | 内聚类型 | 说明 |
|------|----------|------|
| 低内聚 | 偶然内聚 | 一个模块完成一组任务，这些任务彼此间即使有关系，关系也是很松散 |
| 低内聚 | 逻辑内聚 | 一个模块完成的任务在逻辑上属于相同或相似的一类 |
| 低内聚 | 时间内聚 | 一个模块包含的任务必须在同一段时间内执行 |
| 中内聚 | 过程内聚 | 一个模块内的处理元素是相关的，而且必须以特定次序执行 |
| 中内聚 | 通信内聚 | 模块中所有元素都使用同一个输入数据和（或）产生同一个输出数据 |
| 高内聚 | 顺序内聚 | 一个模块内的处理元素和同一个功能密切相关，且必须顺序执行（通常一个处理元素的输出数据作为下一个的输入数据） |
| 高内聚 | 功能内聚 | 模块内所有处理元素属于一个整体，完成一个单一的功能。**最高程度的内聚** |

- 按紧密程度从低到高排列：偶然内聚 → 逻辑内聚 → 时间内聚 → 过程内聚 → 通信内聚 → 顺序内聚 → 功能内聚。

### 4. 总体设计启发规则 ⭐(练习10,多选3,判断6,7)

1. **改进软件结构提高模块独立性**：通过模块分解或合并，力求降低耦合提高内聚。
2. **模块规模应适中**：最好能写在一页纸内（通常不超过 60 行语句）。
3. **深度、宽度、扇出和扇入都应适当**：
   - **深度**：软件结构中控制的层数。
   - **宽度**：软件结构内同一个层次上的模块总数的最大值。
   - **扇出**：一个模块直接控制（调用）的模块数目。
   - **扇入**：一个模块被多少个上级模块直接调用的数目。
4. **模块的作用域应该在控制域之内**：
   - **作用域**：受该模块内一个判定影响的所有模块的集合。
   - **控制域**：模块本身以及所有直接或间接从属于它的模块的集合。
   - 修改方法：①把做判定的点往上移；②把在作用域内但不在控制域内的模块移到控制域内。
5. **力争降低模块接口的复杂程度**：模块接口复杂是软件发生错误的一个主要原因。
6. **设计单入口单出口的模块**：不要使模块间出现内容耦合。
7. **模块功能应该可以预测**：只要输入的数据相同就产生同样的输出（黑盒子）。但也要防止模块功能过分局限。带有内部"存储器"的模块功能可能不可预测。

### 5. 描绘软件结构的图形工具 ⭐(练习11,多选5,判断8)

#### 5.1 层次图和 HIPO 图

- **层次图**：用来描绘软件的层次结构，很适于在自顶向下设计软件的过程中使用。
- **HIPO 图**：美国 IBM 公司发明的"层次图加输入/处理/输出图"的英文缩写。为了使 HIPO 图具有可追踪性，在 H 图（层次图）里除了最顶层的方框之外，每个方框都加了编号。

#### 5.2 结构图

- Yourdon 提出的结构图是进行软件结构设计的工具。
- 一个方框代表一个模块，框内注明模块的名字或主要功能；方框之间的箭头（或直线）表示模块的调用关系。
- 尾部是**空心圆**表示传递的是**数据**，**实心圆**表示传递的是**控制信息**。

### 6. 面向数据流的设计方法 ⭐(练习12,13)

- **定义**：面向数据流的设计方法定义了一些不同的"映射"，利用这些映射可以把数据流图变换成软件结构。通常所说的结构化设计方法（简称 SD 方法）也就是基于数据流的设计方法。

#### 6.1 信息流的两种类型

| 类型 | 特征 |
|------|------|
| **变换流** | 信息沿输入通路进入系统，由外部形式变换成内部形式，进入系统的信息通过变换中心，经加工处理后再沿输出通路变换成外部形式离开软件系统 |
| **事务流** | 数据沿输入通路到达一个处理 T，这个处理根据输入数据的类型在若干个动作序列中选出一个来执行 |

- **事务中心**完成三项任务：①接收输入数据（事务）；②分析每个事务以确定它的类型；③根据事务类型选取一条活动通路。

#### 6.2 变换分析（7 个步骤）

1. 复查基本系统模型
2. 复查并精化数据流图
3. 确定数据流图具有变换特性还是事务特性
4. 确定输入流和输出流的边界，从而孤立出变换中心
5. 完成"第一级分解"：
   - 顶层控制模块 $C_m$ 协调下述从属控制功能：
   - 输入信息处理控制模块 $C_a$：协调对所有输入数据的接收
   - 变换中心控制模块 $C_t$：管理对内部形式的数据的所有操作
   - 输出信息处理控制模块 $C_e$：协调输出信息的产生过程
6. 完成"第二级分解"：把数据流图中的每个处理映射成软件结构中一个适当的模块
   - 从变换中心边界逆着输入通路向外移动，映射到 $C_a$ 控制下的低层模块
   - 沿输出通路向外移动，映射到 $C_e$ 控制下的低层模块
   - 变换中心内的每个处理映射到 $C_t$ 控制下的模块
7. 使用设计度量和启发式规则对第一次分割得到的软件结构进一步精化

#### 6.3 事务分析

- 数据流具有明显的事务特点时采用。设计步骤和变换分析大部分相同或类似，主要差别在于由数据流图到软件结构的映射方法不同。
- 由事务流映射成的软件结构包括一个**接收分支**和一个**发送分支**。

#### 6.4 设计优化

- 设计人员应该致力于开发能够满足所有功能和性能要求，而且按照设计原理和启发式设计规则衡量是值得接收的软件。

### 7. 详细设计的任务 (练习3)

- **定义**：详细设计阶段的根本目标是确定应该怎样具体地实现所要求的系统，经过这个阶段的设计工作，应该得出对目标系统的精确描述，从而在编码阶段可以把这个描述直接翻译成用某种程序设计语言书写的程序。
- 详细设计阶段的任务不是具体地编写程序，而是要设计出程序的"蓝图"。详细设计的结果基本上决定了最终的程序代码的质量。
- 目标不仅仅是逻辑上正确地实现每个模块的功能，更重要的是设计出的处理过程应该尽可能简明易懂。
- **结构程序设计技术**是实现上述目标的关键技术，是详细设计的逻辑基础。

### 8. 结构程序设计 ⭐(练习14,多选6)

- 1965 年由 E.W.Dijkstra 提出：程序的质量与程序中所包含的 GO TO 语句的数量成反比。
- 1966 年 Bohm 和 Jacopini 证明了只用"顺序"、"选择"和"循环"控制结构就能实现任何单入口单出口的程序。

- **经典定义**：如果一个程序的代码块仅仅通过顺序、选择和循环这 3 种基本控制结构进行连接，并且每个代码块只有一个入口和一个出口，则称这个程序是结构化的。
- **更全面的定义**：结构程序设计是尽可能少用 GO TO 语句的程序设计方法。最好仅在检测出错误时才使用 GO TO 语句，而且应该总是使用前向 GO TO 语句。

- **三种层次**：

  | 层次 | 允许的控制结构 |
  |------|----------------|
  | 经典的结构程序设计 | 顺序、IF-THEN-ELSE 型分支、DO-WHILE 型循环 |
  | 扩展的结构程序设计 | 上述 3 种 + DO-CASE 型多分支结构 + DO-UNTIL 型循环结构 |
  | 修正的结构程序设计 | 上述 5 种 + LEAVE（或 BREAK）结构 |

### 9. 人机界面设计

- 人机界面设计是接口设计的一个重要组成部分。
- 设计人机界面时几乎总会遇到 4 个问题：

| 问题 | 要点 |
|------|------|
| **系统响应时间** | 两个重要属性：**长度**（过长紧张，过短易犯错）和**易变性**（相对于平均响应时间的偏差） |
| **用户帮助设施** | 分为集成的和附加的两类。需解决：何时获得帮助、怎样请求帮助（帮助菜单/特殊功能键/HELP命令）、怎样显示、怎样返回、怎样组织（平面结构/层次结构/超文本结构） |
| **出错信息处理** | ①用用户可理解的术语描述问题；②提供建设性意见；③指出负面后果；④伴随听觉/视觉提示；⑤不能带有指责色彩 |
| **命令交互** | ①是否每个菜单选项都有对应命令；②采用何种命令形式（控制序列/功能键/输入命令）；③学习记忆难度；④是否可定制或缩写。提供"命令宏机制" |

### 10. 过程设计的工具 ⭐(练习15,16,17,多选7,8)

#### 10.1 程序流程图

- 又称程序框图，使用最广泛的描述过程设计的方法。
- 箭头代表**控制流**。
- **主要缺点**：
  1. 本质上不是逐步求精的好工具，诱使程序员过早考虑控制流程而不考虑全局结构。
  2. 用箭头代表控制流，程序员不受约束，可以随意转移控制。
  3. 不易表示数据结构。

#### 10.2 盒图（N-S 图）

- 由 Nassi 和 Shneiderman 提出。
- **特点**：
  1. 功能域明确，一眼看出。
  2. 不可能任意转移控制。
  3. 很容易确定局部和全程数据的作用域。
  4. 很容易表现嵌套关系，也可以表示模块的层次结构。

#### 10.3 PAD 图（问题分析图）

- PAD 是 Problem Analysis Diagram 的英文缩写，用二维树形结构表示程序的控制流。
- **主要优点**：
  1. 使用 PAD 符号设计出来的程序必然是结构化程序。
  2. 所描绘的程序结构十分清晰。
  3. 易读、易懂、易记。
  4. 容易将 PAD 图转换成高级语言源程序，可用软件工具自动完成。
  5. 既可表示程序逻辑，也可描绘数据结构。
  6. 支持自顶向下、逐步求精方法的使用。

#### 10.4 判定表

- 能够清晰地表示复杂的条件组合与应做的动作之间的对应关系。
- 由 4 部分组成：左上部列出所有条件、左下部是所有可能做的动作、右上部是表示各种条件组合的矩阵、右下部是和每种条件组合相对应的动作。
- 右半部每一列实质上是一条规则。

#### 10.5 判定树

- 判定表的变种，也能清晰地表示复杂的条件组合与应做的动作之间的对应关系。

#### 10.6 过程设计语言（PDL / 伪码）

- 用正文形式表示数据和处理过程的设计工具。
- **特点**：①关键字的固定语法；②自然语言的自由语法；③数据说明的手段；④模块定义和调用的技术。
- **优点**：①可作为注释直接插在源程序中间；②可用普通正文编辑程序书写编辑；③已有自动处理程序可自动生成代码。
- **缺点**：不如图形工具形象直观，描述复杂条件组合与动作间对应关系时不如判定表清晰简单。

### 11. 面向数据结构的设计方法 ⚠(判断9)

- **核心思想**：数据结构既影响程序的结构又影响程序的处理过程。重复出现的数据通常由具有循环控制结构的程序来处理，选择数据要用带有分支控制结构的程序来处理，层次的数据组织通常和使用这些数据的程序的层次结构十分相似。
- **最终目标**：得出对程序处理过程的描述（不是给出软件结构的系统化途径）。这种方法最适合于在详细设计阶段使用。
- **代表方法**：Jackson 方法和 Warnier 方法。

#### Jackson 图

逻辑数据结构只有 3 类：**顺序结构**、**选择结构**、**循环结构**。

#### Jackson 法（5 个步骤）

1. 分析并确定输入数据和输出数据的逻辑结构，并用 Jackson 图描绘这些数据结构。
2. 找出输入数据结构和输出数据结构中有对应关系的数据单元。
3. 从描绘数据结构的 Jackson 图导出描绘程序结构的 Jackson 图。
4. 列出所有操作和条件（包括分支条件和循环结束条件），并且把它们分配到程序结构图的适当位置。
5. 用伪码表示程序。

### 12. 程序复杂程度的定量度量 ⭐(练习18)

#### 12.1 McCabe 方法

- 根据程序控制流的复杂程度定量度量程序的复杂程度，度量结果称为**环形复杂度**。

##### （1）流图

- 流图实质上是"退化了的"程序流程图，仅描绘程序的控制流程，完全不表现对数据的具体操作以及分支或循环的具体条件。
- 一个圆代表一条或多条语句；一个顺序结构可以合并一个结点；箭头线称为边，代表控制流；一条边必须终止于一个结点。
- **复合条件**（包含布尔运算符 OR、AND、NAND、NOR）应分解为若干个简单条件，每个简单条件对应流图中一个结点。

##### （2）计算环形复杂度的 3 种方法

$$V(G) = \text{流图中线性无关的区域数}$$

$$V(G) = E - N + 2$$

其中 $E$ 是流图中边的条数，$N$ 是结点数。

$$V(G) = P + 1$$

其中 $P$ 是流图中判定结点的数目。

##### （3）环形复杂度的用途

- 取决于程序控制流的复杂程度，是对测试难度的一种定量度量，也能对软件最终的可靠性给出某种预测。
- **模块规模以 $V(G) \leq 10$ 为宜**，即 $V(G)=10$ 是模块规模的一个更科学更精确的上限。

#### 12.2 Halstead 方法

- 根据程序中运算符和操作数的总数来度量程序的复杂程度。
- $N_1$ = 程序中运算符出现的总次数，$N_2$ = 操作数出现的总次数。
- 程序长度：

$$N = N_1 + N_2$$

- $n_1$ = 不同运算符（包括关键字）的个数，$n_2$ = 不同操作数（变量和常数）的个数。
- 预测程序长度：

$$H = n_1 \log_2 n_1 + n_2 \log_2 n_2$$

- 预测程序中包含错误的个数：

$$E = \frac{N \log_2(n_1 + n_2)}{3000}$$

### 13. 面向对象设计的准则 (练习5)

1. **模块化**：对象就是模块，把数据结构和操作这些数据的方法紧密地结合在一起。
2. **抽象**：面向对象方法不仅支持过程抽象，而且支持数据抽象。类是一种抽象数据类型，公共接口构成类的规格说明。某些语言还支持**参数化抽象**（如 C++ 的"模板"机制）。
3. **信息隐藏**：通过对象的封装性实现，类分离了接口与实现。
4. **弱耦合**：
   - **交互耦合**：通过消息连接实现，应尽可能松散（降低消息连接复杂程度、减少消息数）。
   - **继承耦合**：应提高继承耦合程度，特殊类应尽量多继承并使用一般化类的属性和服务。
5. **强内聚**：
   - **服务内聚**：一个服务应该完成一个且仅完成一个功能。
   - **类内聚**：一个类应该只有一个用途，属性和服务应该是高内聚的。
   - **一般/特殊内聚**：设计出的一般-特殊结构应该符合多数人的概念，是对领域知识的正确抽取。
6. **可重用**：尽量使用已有的类，设计新类时考虑将来的可重复使用性。

---

### 14. 面向对象设计的启发规则 (多选9)

1. **设计结果应该清晰易懂**：提高软件可维护性和可重用性的重要措施。
2. **一般-特殊结构的深度应适当**：一个中等规模（约 100 个类）的系统中，类等级层次数应保持为 **7±2**。
3. **设计简单的类**：①避免包含过多的属性；②有明确的定义；③尽量简化对象之间的合作关系；④不要提供太多服务。
4. **使用简单的协议**：消息中的参数不要超过 **3** 个。
5. **使用简单的服务**：尽量避免使用复杂的服务。
6. **把设计变动减至最小**：设计早期变动较大，随时间推移日趋成熟，改动越来越小。

### 15. 面向对象设计的基本原则 ⚠(判断10)

| 原则 | 核心内容 |
|------|----------|
| **单一职责原则** | 一个类最好只做一件事，是低耦合、高内聚在面向对象原则上的引申 |
| **开放封闭原则** | 对扩展开放，对修改封闭。核心思想是对抽象编程而不对具体编程，通过继承和多态实现扩展 |
| **Liskov 替换原则（里氏替换原则）** | 子类必须能够替换其基类；基类替换为子类，程序行为不发生变化；反过来不成立——基类不一定能替换子类 |
| **依赖倒置原则** | 高层模块不依赖于底层模块，二者都同依赖于抽象；抽象不依赖于具体，具体依赖于抽象 |
| **接口隔离原则** | 使用多个小的专门的接口，而不要使用一个大的总接口；接口应该是内聚的，避免"胖"接口；不要强迫依赖不用的方法 |

### 16. 设计模式 ⭐(练习19,20,多选10)

- **定义**：设计模式是对面向对象设计中反复出现的问题的解决方案。
- 常见设计模式共 **23** 种：创建型 **5** 种、结构型 **7** 种、行为型 **11** 种。

#### 16.1 创建型模式（5 种）

抽象工厂模式、生成器模式、工厂方法模式、原型模式、**单例模式**。

- **单例模式**：保证一个类仅有一个实例，并提供一个访问它的全局访问点。

#### 16.2 结构型模式（7 种）

适配器模式、桥接模式、**组合模式**、装饰模式、外观模式、享元模式、代理模式。

- **适配器模式**：将一个类的接口转换成客户希望的另外一个接口，使得原本接口不兼容的类可以一起工作。
- **组合模式**：将对象组合成树形结构以表示"整体-部分"的层次结构，使得用户对单个对象和组合对象的使用具有一致性。
- **外观模式**：为子系统中的一组接口提供一个一致的界面，定义了一个高层接口，使得子系统更加容易使用。
- **代理模式**：为其他对象提供一种代理以控制对这个对象的访问。

#### 16.3 行为型模式（11 种）

**观察者模式**、模板方法模式、命令模式、**状态模式**、职责链模式、解释器模式、中介者模式、访问者模式、策略模式、备忘录模式、**迭代器模式**。

- **观察者模式**：定义对象间一种一对多的依赖关系，当一个对象的状态发生变化时，所有依赖于它的对象都得到通知并被自动更新。
- **状态模式**：允许一个对象在其内部状态改变时改变它的行为，让对象看起来似乎修改了它的类。
- **迭代器模式**：提供一种方法顺序访问一个聚合对象中的各个元素，而又不许暴露该对象的内部表示。

### 17. 软件重用 (练习21)

- **定义**：重用也叫再用或复用，是指同一事物不作修改或稍加改动就多次重复使用。
- **广义软件重用 3 个层次**：知识重用、方法和标准的重用、软件成分的重用。

- **软件成分重用 3 个级别**：

  | 级别 | 内容 |
  |------|------|
  | 代码重用 | 源代码剪贴、源代码包含、继承 |
  | 设计结果重用 | 重用某个软件系统的设计模型（求解域模型） |
  | 分析结果重用 | 更高级别的重用，重用某个系统的分析模型 |

- **典型的可重用软件成分**（10 种）：项目计划、成本估计、体系结构、需求模型和规格说明、设计、源代码、用户文档和技术文档、用户界面、数据、测试用例。

- **类构件的重用方式**（3 种）：
  1. **实例重用**
  2. **继承重用**
  3. **多态重用**

### 模块化论证不等式

$$C(P_1+P_2) > C(P_1) + C(P_2)$$

$$E(P_1+P_2) > E(P_1) + E(P_2)$$

### McCabe 环形复杂度

$$V(G) = \text{线性无关的区域数}$$

$$V(G) = E - N + 2 \quad (E\text{=边数},\ N\text{=结点数})$$

$$V(G) = P + 1 \quad (P\text{=判定结点数})$$

- 模块规模上限：$V(G) \leq 10$

### Halstead 方法

$$N = N_1 + N_2 \quad (N_1\text{=运算符总次数},\ N_2\text{=操作数总次数})$$

$$H = n_1 \log_2 n_1 + n_2 \log_2 n_2 \quad (n_1\text{=不同运算符个数},\ n_2\text{=不同操作数个数})$$

$$E = \frac{N \log_2(n_1 + n_2)}{3000}$$

## 四、重点对比与易混辨析

### 4.1 耦合类型对比（由低到高）

| 耦合类型 | 程度 | 典型特征 | 设计建议 |
|----------|------|----------|----------|
| 数据耦合 | 低 | 仅通过参数交换数据 | 尽量使用 |
| 控制耦合 | 中 | 传递控制信息 | 少用 |
| 特征耦合 | — | 整个数据结构作为参数传递，但只需一部分 | 少用 |
| 公共环境耦合 | 随模块数变化 | 通过公共数据环境相互作用 | 限制范围 |
| 内容耦合 | 最高 | 访问内部数据/不通过正常入口/代码重叠/多入口 | 完全不用 |

### 4.2 内聚类型对比（由低到高）

| 类别 | 内聚类型 | 紧密程度 | 典型特征 |
|------|----------|----------|----------|
| 低内聚 | 偶然内聚 | 最低 | 任务间关系松散 |
| 低内聚 | 逻辑内聚 | — | 任务逻辑上属于相同或相似的一类 |
| 低内聚 | 时间内聚 | — | 任务必须在同一段时间内执行 |
| 中内聚 | 过程内聚 | — | 处理元素相关，必须以特定次序执行 |
| 中内聚 | 通信内聚 | — | 所有元素使用同一输入/产生同一输出 |
| 高内聚 | 顺序内聚 | — | 处理元素与同一功能密切相关，顺序执行 |
| 高内聚 | 功能内聚 | 最高 | 所有元素属于一个整体，完成单一功能 |

### 4.3 过程设计工具对比

| 工具 | 全称/别名 | 特点 | 缺点 |
|------|-----------|------|------|
| 程序流程图 | 程序框图 | 使用最广泛；箭头代表控制流 | 不是逐步求精好工具；可随意转移控制；不易表示数据结构 |
| 盒图 | N-S 图 | 功能域明确；不可能任意转移控制；易确定作用域；易表现嵌套 | [原文未述] |
| PAD 图 | 问题分析图 | 二维树形结构；必然结构化；清晰易读；可自动转换代码；支持逐步求精 | [原文未述] |
| 判定表 | — | 清晰表示条件组合与动作对应关系；每列是一条规则 | [原文未述] |
| 判定树 | — | 判定表的变种，清晰表示条件组合与动作对应关系 | [原文未述] |
| PDL | 伪码 | 正文形式；可作注释插入源程序；可自动生成代码 | 不如图形直观；复杂条件不如判定表清晰 |

### 4.4 变换流 vs 事务流

| 特征 | 变换流 | 事务流 |
|------|--------|--------|
| 输入处理方式 | 信息沿输入通路进入，经变换中心加工后沿输出通路离开 | 数据沿输入通路到达处理 T，T 根据类型选择一个动作序列执行 |
| 映射结构 | 控制输入、变换、输出三个模块 | 接收分支 + 发送分支 |
| 设计方法 | 变换分析 | 事务分析 |

### 4.5 面向对象设计五大原则对比

| 原则 | 一句话概括 | 关键点 |
|------|-----------|--------|
| 单一职责原则 | 一个类最好只做一件事 | 低耦合高内聚的引申 |
| 开放封闭原则 | 对扩展开放，对修改封闭 | 对抽象编程，通过继承多态扩展 |
| 里氏替换原则 | 子类必须能替换基类 | 反过来不成立 |
| 依赖倒置原则 | 依赖于抽象，不依赖于具体 | 高层底层都依赖抽象 |
| 接口隔离原则 | 用多个小接口，不用大总接口 | 避免胖接口，接口应内聚 |

### 4.6 结构程序设计三种层次

| 层次 | 允许的控制结构 |
|------|----------------|
| 经典 | 顺序、IF-THEN-ELSE、DO-WHILE |
| 扩展 | 经典 + DO-CASE + DO-UNTIL |
| 修正 | 扩展 + LEAVE（或 BREAK） |

## 五、高频考点与易错提醒

1. ⚠ **总体设计 vs 详细设计的任务边界**是高频易错点——总体设计建立软件结构、设计数据库；详细设计确定功能实现算法（练习1,2,3,多选1,判断1）。
2. ⚠ **内聚和耦合是度量模块独立性的指标**，不是度量模块复杂性的指标——判断题4的陷阱（判断4）。
3. ⚠ **内容耦合是最高程度耦合**，特征包括访问内部数据、不通过正常入口转入、代码重叠、多入口——反复考查（练习8,9）。
4. ⚠ **内聚从低到高排列顺序**（偶然→逻辑→时间→过程→通信→顺序→功能）需准确记忆（多选4）。
5. ⚠ **扇入 vs 扇出定义易混**——扇入是被多少上级模块调用，扇出是直接控制多少下级模块（练习10,判断6）。
6. ⚠ **作用域应在控制域之内**——判定影响的所有模块应从属于做判定的模块（判断7）。
7. ⚠ **HIPO 图、层次图、结构图**是描绘软件结构的工具；**数据流图不是详细设计工具**（练习11,15,多选5,判断8）。
8. ⚠ **结构化设计（SD 方法）以数据流图为基础**——面向数据流的设计方法（练习12）。
9. ⚠ **事务流的特征**——"数据沿输入通路到达处理 T，根据类型选择动作序列"（练习13）。
10. ⚠ **三种基本控制结构**：顺序、选择、循环（练习14,多选6）。
11. ⚠ **程序流程图箭头代表控制流**，不是数据流（练习16）。
12. ⚠ **McCabe 环形复杂度计算**：$V(G)=E-N+2$ 或 $V(G)=P+1$，模块规模上限 $V(G)\leq10$（练习18）。
13. ⚠ **面向数据结构的设计方法目标是得出程序处理过程描述**，不是给出软件结构途径——判断题9的陷阱（判断9）。
14. ⚠ **依赖倒置原则**：应依赖抽象而非具体——判断题10的陷阱（判断10）。
15. ⚠ **设计模式分类与数量**：创建型5种、结构型7种、行为型11种，共23种；单例模式限制实例为一个，组合模式表示"整体-部分"层次结构（练习19,20,多选10）。
16. ⚠ **类构件重用方式**：实例重用、继承重用、多态重用——不包括"消息重用"（练习21）。
17. ⚠ **模块应高内聚低耦合**，不应加强模块间联系——判断题2的陷阱（判断2）。

## 六、一句话记忆要点

1. **总体设计**=概要设计，回答"系统如何实现"，确定软件结构与模块关系。
2. **总体设计九步骤**：设想方案→选取方案→推荐最佳→功能分解→设计结构→设计数据库→制定测试计划→书写文档→审查复审。
3. **五大设计原理**：模块化（各个击破）、抽象（抽本质忽略细节）、逐步求精（推迟细节）、信息隐藏（不需则不可访问）、模块独立（内聚+耦合度量）。
4. **耦合**从低到高：数据→控制→特征→公共环境→内容；**内聚**从低到高：偶然→逻辑→时间→过程→通信→顺序→功能。
5. **七条启发规则**：提高独立性、规模适中、深度宽度扇出扇入适当、作用域在控制域内、降低接口复杂度、单入单出、功能可预测。
6. **变换流**=输入→变换中心→输出；**事务流**=到达处理T→选类型→选通路。
7. **变换分析七步**：复查模型→精化DFD→判断特性→定边界孤立变换中心→第一级分解($C_m$/$C_a$/$C_t$/$C_e$)→第二级分解→精化。
8. **详细设计**=设计程序"蓝图"，不是写代码；结构程序设计是其逻辑基础。
9. **结构程序设计**：顺序+选择+循环（经典），+DO-CASE+DO-UNTIL（扩展），+LEAVE/BREAK（修正）。
10. **过程设计六工具**：程序流程图（箭头=控制流）、盒图/N-S图（不可转移控制）、PAD图（二维树形，可自动转代码）、判定表（条件组合×动作矩阵）、判定树（判定表变种）、PDL/伪码（正文形式）。
11. **面向数据结构设计**：目标=程序处理过程描述（非软件结构），Jackson法5步，数据结构3类=顺序/选择/循环。
12. **McCabe**：$V(G)=E-N+2=P+1$，上限≤10；**Halstead**：$N=N_1+N_2$，$H=n_1\log_2 n_1+n_2\log_2 n_2$，$E=N\log_2(n_1+n_2)/3000$。
13. **OO设计六准则**：模块化、抽象（过程+数据+参数化）、信息隐藏（封装）、弱耦合（交互松/继承紧）、强内聚（服务/类/一般-特殊）、可重用。
14. **OO设计五原则**：单一职责、开放封闭（扩展开放修改封闭）、里氏替换（子类可替基类）、依赖倒置（依赖抽象）、接口隔离（小接口非胖接口）。
15. **23种设计模式**：创建型5（含单例）、结构型7（含适配器/组合/外观/代理）、行为型11（含观察者/状态/迭代器）。
16. **软件重用**：3层次（知识/方法标准/软件成分），3级别（代码/设计/分析），类构件3方式（实例/继承/多态）。
