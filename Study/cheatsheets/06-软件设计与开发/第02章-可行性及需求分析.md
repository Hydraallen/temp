# 软件设计与开发 · 第二章 可行性及需求分析 — 复习资料

> 来源：6-3-软件设计与开发计算机类讲义.pdf 物理页 [28, 52]（印刷页 23–47）
> 说明：仅摘录「知识点」正文；⭐=高频考点，⚠=易错点（依据本章专项练习与答案标注）。

---

## 一、本章概览

本章覆盖软件生命周期中**可行性研究**与**需求分析**两大阶段，是软件定义时期的核心内容。可行性研究旨在用最小代价、在最短时间内确定问题是否能够解决，涉及技术/经济/操作三方面可行性、系统流程图、数据流图、数据字典及成本/效益分析。需求分析则准确回答"系统必须做什么"，涉及需求获取方法、分析建模（数据模型/功能模型/行为模型）、实体联系图、状态转换图、需求验证，以及面向对象分析（对象模型/动态模型/功能模型）。本章知识点多、图形工具密集，是考试高频出题区域。

---

## 二、知识点详解

### 1. 可行性研究的任务 ⭐ (练习单选1, 判断1-2)

- **定义**：可行性研究的目的，就是用最小的代价在尽可能短的时间内确定问题是否能够解决。
- **要点**：并非任何问题都有简单明显的解决办法，许多问题不可能在预定的系统规模或时间期限之内解决。如果问题没有可行的解，那么花费在这项工程上的任何时间、人力、软硬件资源和经费，都是无谓的浪费。
- **三个方面**：
  - **技术可行性**：使用现有的技术能实现这个系统吗？度量一个特定技术信息系统解决方案的实用性及技术资源的可用性。
  - **经济可行性**：这个系统的经济效益能超过它的开发成本吗？
  - **操作可行性**：系统的操作方式在这个用户组织内行得通吗？
- 必要时还应该从法律、社会效益等更广泛的方面研究。
- 可行性研究成本一般只是预期的工程总成本的 **5%～10%**。

### 2. 可行性研究过程 ⭐ (练习多选1)

- **定义**：典型的可行性研究过程有下述 **8 个步骤**。
- **8 个步骤**：
  1. **复查系统规模和目标**——确保分析员正在解决的问题确实是要求他解决的问题。
  2. **研究目前正在使用的系统**——现有系统是信息的重要来源。新系统必须能完成旧系统的基本功能，也必须能解决旧系统中存在的问题。没有系统在"真空"中运行。
  3. **导出新系统的高层逻辑模型**——从现有物理系统出发→导出现有系统逻辑模型→设想目标系统逻辑模型→建造新的物理系统。
  4. **进一步定义问题**——前4个步骤构成循环：定义问题→分析→导出试探性解→再定义→再分析→修改解，直到逻辑模型完全符合系统目标。
  5. **导出和评价供选择的解法**——从逻辑模型导出若干较高层次物理解法；考虑操作可行性→去掉不可接受方案；考虑经济可行性→估计开发成本和运行费用；为可行方案制定实现进度表。
  6. **推荐行动方针**——关键问题：是否继续进行这项开发工程？选择最好的解法并说明理由。客户主要根据经济上是否划算决定是否投资，因此必须进行成本/效益分析。
  7. **草拟开发计划**——制定工程进度表，估计人员和资源需要，估计各阶段成本，给出下一阶段（需求分析）的详细进度表和成本估计。
  8. **书写文档提交审查**——写成清晰文档，请用户、负责人及评审组审查。

### 3. 系统流程图 ⭐ (练习单选2, 判断3)

- **定义**：系统流程图是概括地描绘**物理系统**的传统工具。基本思想是用图形符号以黑盒子形式描绘组成系统的每个部件（程序、文档、数据库、人工过程等）。
- **关键区分**：系统流程图表达的是**数据在系统各部件之间流动的情况**，而不是对数据进行加工处理的控制过程。因此尽管某些符号和程序流程图相同，它却是**物理数据流图**而不是程序流程图。
- **基本符号**：

| 符号 | 名称 | 说明 |
|------|------|------|
| 矩形 | 处理 | 能改变数据值或数据位置的加工或部件（程序、处理机、人工加工等） |
| 菱形 | 输入输出 | 表示输入或输出（或既输入又输出），广义的不指明具体设备的符号 |
| 圆圈 | 连接 | 指出转到图的另一部分或从图的另一部分转来，通常在同一页上 |
| 倒三角 | 换页连接 | 指出转到另一页图上或由另一页图转来 |
| 箭头 | 数据流 | 用来连接其他符号，指明数据流动方向 |

- **系统符号**（部分）：穿孔卡片、文档（打印输出）、磁带、联机存储、磁盘、磁鼓、显示（CRT终端）、人工输入、人工操作、辅助操作、通信链路。
- **分层描绘**：面对复杂系统时，先用一张高层次系统流程图描绘系统总体概貌，然后分别把每个关键功能扩展到适当详细程度，画在单独的页上。便于从抽象到具体逐步深入了解。

### 4. 数据流图（DFD） ⭐⚠ (练习单选3-6, 多选2-3, 判断4-5)

- **定义**：数据流图(DFD)是一种图形化技术，它描绘信息流和数据从输入移动到输出的过程中所经受的变换。在数据流图中没有任何具体的物理部件，它只是描绘数据在软件中流动和被处理的**逻辑过程**。
- **性质**：数据流图是系统**逻辑功能**的图形表示，即使不是专业计算机技术人员也容易理解，是分析员与用户之间极好的**通信工具**。设计时只需考虑系统必须完成的基本逻辑功能，不需要考虑怎样具体实现。
- **四种基本符号**：

| 符号 | 含义 |
|------|------|
| 正方形 | 数据的源点或终点——软件系统外部环境中的实体（包括人员、组织或其他软件系统） |
| 圆角矩形 | 变换数据的处理 |
| 开口矩形 | 数据存储——可以是数据库文件或任何形式的数据组织 |
| 箭头 | 数据流，即特定数据的流动方向 |

- **分层描述**：复杂系统的数据流图可采用分层的描述方式。任何系统的基本模型都由若干个数据源点/终点以及一个处理组成。从基本系统模型（顶层）开始画数据流图，顶层上是否列出了所有数据源点/终点一目了然。然后逐步分解到功能级、更细的层次。
- **数据流图的用途**：
  1. 作为交流信息的工具。
  2. 作为分析和设计的工具。
  3. 辅助物理系统设计时，以图中不同处理的定时要求为指南，能够在数据流图上画出许多组自动化边界，每组自动化边界可能意味着一个不同的物理系统。

### 5. 数据字典 ⭐⚠ (练习单选7, 多选4)

- **定义**：数据字典是关于数据的信息的集合，也就是对数据流图中包含的所有元素的定义的集合。
- **四类元素**：数据字典由对下列 4 类元素的定义组成：
  1. 数据流
  2. 数据流分量
  3. 数据存储
  4. 处理

> ⚠ 注意：数据字典**不包括外部实体**（外部实体是数据流图中的概念）。(练习单选7)

- **数据字典中记录的信息**：一般信息（名字、别名、描述等）、定义（数据类型、长度、结构等）、使用特点（值的范围、使用频率、使用方式——输入、输出、本地，条件值等）、控制信息（来源、用户、使用它的程序、改变权、使用权等）和分组信息（父结构、从属结构、物理位置——记录、文件和数据库等）。
- **别名产生的原因**（3个）：
  1. 对于同样的数据，不同的用户使用了不同的名字。
  2. 一个分析员在不同时期对同一个数据使用了不同的名字。
  3. 两个分析员分别分析同一个数据流时，使用了不同的名字。
- **数据元素组成数据的方式**（4种基本类型）：
  1. **顺序**：以确定次序连接两个或多个分量。
  2. **选择**：从两个或多个可能的元素中选取一个。
  3. **重复**：把指定的分量重复零次或多次。
  4. **可选**：一个分量是可有可无的（当重复的上下限分别为1和0时可用重复算符表示，但单独列出更清晰）。
- **数据字典符号**：

| 符号 | 含义 |
|------|------|
| `=` | 等价于（或定义为） |
| `+` | 和（连接两个分量） |
| `[ ]` | 或（方括弧中的分量选一个） |
| `{ }` | 重复（重复花括弧中的分量） |
| `( )` | 可选（圆括弧里的分量可有可无） |

- **示例**（标识符定义）：
  ```
  标识符 = 字母字符 + 字母数字串
  字母数字串 = 0{字母或数字}7
  字母或数字 = [字母字符 | 数字字符]
  ```
- **数据字典的用途**：
  1. 最重要的用途是作为分析阶段的工具。
  2. 包含的每个数据元素的控制信息是很有价值的。
  3. 是开发数据库的第一步，而且是很有价值的一步。

### 6. 成本/效益分析 (练习判断6)

- **定义**：软件开发成本主要表现为人力消耗（乘以平均工资则得到开发费用）。成本估计不是精确的科学，应该使用几种不同的估计技术以便相互检验。
- **三种估算技术**：
  1. **代码行技术**：比较简单的定量估算方法，把开发每个软件功能的成本和实现这个功能需要用的源代码行数联系起来。用每行代码的平均成本乘以行数确定软件成本。每行代码的平均成本主要取决于软件的复杂程度和工资水平。
  2. **任务分解技术**：把软件开发工程分解为若干个相对独立的任务，分别估计每个任务的成本，最后累加。最常用的办法是按开发阶段划分任务。
  3. **自动估计成本技术**：采用自动估计成本的软件工具，需要长期搜集的大量历史数据为基础，且需要良好的数据库系统支持。
- **成本/效益分析的四个方面**：
  1. **货币的时间价值**：通常用利率的形式表示。公式：$F = P(1 + i)^n$（其中 $i$ 为年利率，$P$ 为现在存入的金额，$n$ 为年数，$F$ 为 $n$ 年后得到的钱数）。
  2. **投资回收期**：使累计的经济效益等于最初投资所需要的时间。回收期越短越值得投资。
  3. **纯收入**：在整个生命周期之内系统的累计经济效益（折合成现在值）与投资之差。
  4. **投资回收率**：衡量投资效益的大小，可以和年利率相比较。
- **核心原则**：投资是现在进行的，效益是将来获得的，不能简单地比较成本和效益，应该考虑货币的时间价值。

### 7. 需求分析的任务 ⭐⚠ (练习单选9,12, 多选7, 判断7)

- **定义**：需求分析是软件定义时期的最后一个阶段，它的基本任务是准确地回答"**系统必须做什么**"这个问题。
- **前提**：对软件需求的深入理解是软件开发工作获得成功的前提条件。
- **四条准则**（所有结构化分析方法都遵守）：
  - 必须理解并描述问题的信息域 → 建立**数据模型**
  - 必须定义软件应完成的功能 → 建立**功能模型**
  - 必须描述作为外部事件结果的软件行为 → 建立**行为模型**
  - 必须对描述信息、功能和行为的模型进行分解，用层次的方式展示细节

> ⚠ 需求分析只回答"做什么"，不回答"怎么做"。(练习判断7)

- **主要任务**：
  1. **确定对系统的综合要求**（8个方面）：

| 序号 | 需求类型 | 说明 |
|------|----------|------|
| (1) | 功能需求 | 指定系统必须提供的服务 |
| (2) | 性能需求 | 定时约束或容量约束（速度/响应时间、信息量速率、主存容量、磁盘容量、安全性等） |
| (3) | 可靠性和可用性需求 | 定量指定系统可靠性；可用性量化用户可使用系统的程度 |
| (4) | 出错处理需求 | 系统对环境错误应该怎样响应（非应用系统本身造成的错误） |
| (5) | 接口需求 | 应用系统与环境的通信格式（用户接口、硬件接口、软件接口、通信接口） |
| (6) | 约束 | 设计或实现时应遵守的限制条件（精度、工具和语言约束、设计约束、标准、硬件平台） |
| (7) | 逆向需求 | 说明软件系统**不应该**做什么 |
| (8) | 将来可能提出的要求 | 不属于当前开发范畴但将来很可能提出的要求，为扩充和修改预做准备 |

  2. **分析系统的数据要求**：系统必须处理的信息和应该产生的信息在很大程度上决定了系统的面貌。利用数据字典全面准确地定义数据，但不够形象直观，常利用图形工具辅助描绘数据结构。
  3. **导出系统的逻辑模型**：用数据流图、实体-联系图、状态转换图、数据字典和主要的处理算法描述。
  4. **修正系统开发计划**：根据更深入具体的了解，比较准确地估计成本和进度，修正开发计划。

### 8. 与用户沟通获取需求的方法 ⭐ (练习单选14, 多选9, 判断9)

- **四种方法**：

#### 8.1 访谈
- 最早开始使用、至今仍广泛使用的需求分析技术。
- 两种基本形式：
  - **正式访谈**：提出事先准备好的具体问题。
  - **非正式访谈**：提出开放性问题，鼓励被访问人员说出自己的想法。
- **情景分析技术**：对用户将来使用目标系统解决某个具体问题的方法和结果进行分析。
  - 用处一：能在某种程度上演示目标系统的行为，便于用户理解，可能进一步揭示出分析员目前还不知道的需求。
  - 用处二：较为用户所理解，能保证用户在需求分析过程中始终扮演积极主动的角色。

#### 8.2 面向数据流自顶向下求精
- 结构化分析方法就是**面向数据流自顶向下逐步求精**进行需求分析的方法。
- 通过可行性研究已得出目标系统的高层数据流图，需求分析的目标之一是把数据流和数据存储定义到元素级。
- 通常从数据流图的**输出端**着手分析（系统的基本功能是产生输出，输出数据决定了系统必须具有的最基本的组成元素），沿数据流图从输出端往输入端回溯，确定每个数据元素的来源，初步定义有关的算法。
- 数据流图是帮助复查的极好工具，借助数据流图、数据字典和 IPO 图向用户解释输入数据是怎样一步一步地转变成输出数据的。

#### 8.3 简易的应用规格说明技术
- 解决传统方法中用户处于被动地位、与开发者区分"彼此"的问题。
- 典型过程：
  1. 进行初步的访谈
  2. 开发者和用户分别写出"产品需求"
  3. 开会讨论，各自展示需求列表
  4. 得出意见一致后，为需求列表制定小型规格说明
  5. 根据会议结果，起草完整的软件需求规格说明
- 优点：开发者与用户不分彼此、密切合作；即时讨论并求精；有能导出规格说明的具体步骤。

#### 8.4 快速建立软件原型
- 快速原型就是快速建立起来的旨在演示目标系统主要功能的可运行的程序。
- **两个特性**：①快速；②容易修改。
- 三种方法和工具：
  1. **第四代技术**：快速生成可执行的代码，较理想的快速原型工具。
  2. **可重用的软件构件**：使用已有的软件构件（组件）来装配原型。
  3. **形式化规格说明和原型环境**：形式化语言调用自动工具把规格说明翻译成可执行的程序代码。

### 9. 分析建模与规格说明 ⭐⚠ (练习单选8,10,11, 多选5-6)

- **模型定义**：模型就是为了理解事物而对事物作出的一种抽象，是对事物的一种无歧义的书面描述。
- **三种模型**（从不同角度抽象目标系统特性）：

| 模型 | 作用 | 图形工具 |
|------|------|----------|
| 数据模型 | 描绘数据对象及数据对象之间的关系 | 实体联系图（ER图） |
| 功能模型 | 建立功能模型的基础 | 数据流图 |
| 行为模型 | 描绘系统的各种行为模式和在不同状态间转换的方式 | 状态转换图 |

> ⚠ 数据流图是功能模型工具，不是数据模型工具。(练习单选10)
> ⚠ 状态图是行为模型工具。(练习单选11)

- **软件需求规格说明**：需求分析阶段得出的最主要的文档。通常用自然语言完整、准确、具体地描述系统的数据要求、功能需求、性能需求、可靠性和可用性要求、出错处理需求、接口需求、约束、逆向需求以及将来可能提出的要求。
- 为消除自然语言可能存在的不一致、歧义、含糊不完整及抽象层次混乱等问题，有些人主张用**形式化方法**描述用户需求。

### 10. 实体联系图（ER图） ⭐ (练习单选13, 多选8)

- **定义**：概念性数据模型（也称信息模型），是一种面向问题的数据模型，按照用户的观点对数据建立的模型。描述从用户角度看到的数据，反映用户的现实环境，与在软件系统中的实现方法无关。
- **数据模型包含3种信息**：
  - **数据对象**：对软件必须理解的复合信息的抽象（外部实体、事物、行为、事件、角色、单位、地点或结构等）。
  - **属性**：定义数据对象的性质。必须把一个或多个属性定义为"标识符"（即"关键字"/"键"）。
  - **联系（关系）**：数据对象彼此之间相互连接的方式。
- **联系的三种类型**：
  - 一对一联系（1∶1）
  - 一对多联系（1∶N）
  - 多对多联系（M∶N）
- **ER图三种基本成分及符号**：

| 成分 | 符号 |
|------|------|
| 实体（实体集） | 矩形框 |
| 联系（联系集） | 菱形框（连接相关实体） |
| 属性 | 椭圆形或圆角矩形（用直线与实体/联系连接） |

- **实体集**：共享相同性质或属性的、具有相同类型的实体的集合。
- **联系集**：相同类型联系的集合。

### 11. 状态转换图 ⭐ (练习单选20)

- **定义**：状态转换图（简称状态图）通过描绘系统的**状态**及引起系统状态转换的**事件**，来表示系统的**行为**。还指明了作为特定事件的结果系统将做哪些动作。
- **状态**：任何可以被观察到的系统行为模式，一个状态代表系统的一种行为模式，规定了系统对事件的响应方式。
  - 三种状态：**初态**（初始状态）、**终态**（最终状态）、**中间状态**。
  - 一张状态图中只能有**一个初态**，终态可以有 **0 至多个**。
- **事件**：在某个特定时刻发生的事情，是对引起系统做动作或（和）从一个状态转换到另一个状态的外界事件的抽象。
- **符号**：

| 状态 | 符号 |
|------|------|
| 初态 | 实心圆 |
| 终态 | 一对同心圆（内圆为实心圆） |
| 中间状态 | 圆角矩形 |

- 中间状态可用两条水平横线分成上、中、下三部分：
  - 上面：状态的名称（必须有的）
  - 中间：状态变量的名字和值（可选）
  - 下面：活动表（可选）
- **活动表语法格式**：`事件名(参数表)/动作表达式`
- **三种标准事件**：
  - `entry`：指定进入该状态的动作
  - `exit`：指定退出该状态的动作
  - `do`：指定在该状态下的动作

### 12. 需求分析的其他图形工具

#### 12.1 层次方框图
- 用树形结构的一系列多层次的矩形框描绘数据的层次结构。

#### 12.2 Warnier图
- 也用树形结构描绘信息，但比层次方框图提供更丰富的描绘手段。
- 可以表明信息的逻辑组织：指出一类信息或一个信息元素是重复出现的，也可以表示特定信息在某一类信息中是有条件地出现的。

#### 12.3 IPO图
- 输入、处理、输出图的简称，由美国IBM公司发展完善。
- 能够方便地描绘输入数据、对数据的处理和输出数据之间的关系。

### 13. 验证软件需求 ⭐ (练习单选15, 多选10)

- **定义**：需求分析阶段的工作结果是开发软件系统的重要基础。为了提高软件质量、确保软件开发成功、降低开发成本，一旦对目标系统提出一组要求之后，必须严格验证这些需求的正确性。
- **四个方面验证**：

| 方面 | 含义 |
|------|------|
| 一致性 | 所有需求必须是一致的，任何一条需求不能和其他需求互相矛盾 |
| 完整性 | 需求必须是完整的，规格说明书应该包括用户需要的每一个功能或性能 |
| 现实性 | 指定的需求应该是用现有的硬件技术和软件技术基本上可以实现的 |
| 有效性 | 必须证明需求是正确有效的，确实能解决用户面对的问题 |

### 14. 面向对象分析 ⭐⚠ (练习单选16-19, 判断8,10)

- **定义**：面向对象分析(OOA)的关键是识别出问题域内的类与对象，并分析它们相互间的关系，最终建立起问题域的简洁、精确、可理解的正确模型。
- **三种模型**：

| 模型 | 性质 | 描述 | 图形工具 |
|------|------|------|----------|
| 对象模型 | 静态的、结构化的系统的"数据"性质 | 对模拟客观世界实体的对象及对象彼此间关系的映射，描述系统的静态结构 | UML 类图 |
| 动态模型 | 瞬时的、行为化的系统的"控制"性质 | — | 序列图、活动图、状态图 |
| 功能模型 | 变化的系统的"功能"性质 | 指明系统应该"做什么"，更直接地反映用户对目标系统的需求 | 数据流图、用例图 |

> ⚠ 对象模型是三种模型中最基本、最重要、最核心的。
> ⚠ OOA 三种模型是对象模型、动态模型、功能模型，**不包括 ER 模型**。(练习单选16)
> ⚠ 序列图是建立**动态模型**的工具，不是功能模型。(练习判断10)

#### 14.1 对象模型
- 复杂系统的对象模型通常由 **5 个层次**组成：
  1. 主题层
  2. 类与对象层
  3. 结构层
  4. 属性层
  5. 服务层

- **类图基本符号**：类名、属性、服务（接口用 `<<interface>>` 标注）。

- **类与类之间的 4 种关系**：

| 关系 | 说明 | 符号 |
|------|------|------|
| 关联 | 类与类之间存在连接关系 | 连接两个类之间的直线（无箭头实线） |
| 泛化（继承） | 通用元素和具体元素之间的分类关系，具体元素完全拥有通用元素的信息并可附加其他信息 | 一端为空心三角形的连线，三角形顶角紧挨通用元素 |
| 依赖 | 两个模型元素之间的语义连接：一个独立，另一个依赖于它；独立的改变将影响依赖于它的 | 虚线（可标注如 `<<友元>>`） |
| 细化/实现 | 对同一个事物在不同抽象层次上描述时，这些描述之间具有细化关系 | 虚线空心三角形 |

> ⚠ 无箭头实线段代表的是**关联**。(练习单选19)

- **聚集（聚合）**：关联的特例，表示整体与部分的关系。两种特殊形式：
  - **共享聚集**：部分方对象可同时参与多个整体方对象的构成。
  - **组合聚集（组成）**：部分类完全隶属于整体类，部分与整体共存，整体不存在了部分也随之消失。

#### 14.2 动态模型
- 在UML中使用**序列图、活动图、状态图**建立动态模型。
- **序列图（Sequence Diagram）**：对象之间基于时间顺序的动态交互，显示随着时间变化对象之间如何进行通讯。主要用途之一是更详细地描述用例表达的需求。
  - 包含的元素：对象生命线、调用消息、异步消息、返回消息、对象销毁等。
- **活动图**：描述系统行为的模型视图，可用来描述系统的工作流程和并发行为，展现参与系统某一行为的类进行的各种活动的顺序关系。
  - 包含的元素：开始、动作、并发分支、分支、监护表达式、并发汇合、结束等。

#### 14.3 功能模型
- 可以用一组数据流图描述。在UML中提供的**用例图**是进行需求分析和建立功能模型的强有力工具。
- **用例**：可以被行为者感受到的、系统的一个完整的功能。在UML中定义为系统完成的一系列动作，动作的结果能被特定的行为者察觉到。
- **行为者（参与者）**：用例通过关联与行为者连接，一个用例必须至少与一个行为者相关联。
- **用例图包含的模型元素**：系统、参与者、用例，以及它们之间的各种关系。
- **用例图中的 4 种关系**：

| 关系 | 说明 | 表示符号 |
|------|------|----------|
| 关联 | **参与者与用例之间**的关系 | 实线 |
| 泛化 | 参与者之间或用例之间的关系 | 实线箭头 |
| 包含 | 用例之间的关系 | 虚线箭头 `<<includes>>` |
| 扩展 | 用例之间的关系 | 虚线箭头 `<<extends>>` |

> ⚠ **关联**是参与者与用例之间的关系，**不是**用例之间的关系。(练习单选18)

## 三、公式 / 算法 / 定理速查表

| 名称 | 公式 / 内容 | 适用条件 |
|------|-------------|----------|
| 货币的时间价值 | $F = P(1 + i)^n$ | $i$=年利率，$P$=现在存入金额，$n$=年数，$F$=n年后得到的钱数 |
| 可行性研究成本占比 | 预期工程总成本的 5%～10% | 一般情况 |
| 数据字典标识符定义 | `标识符 = 字母字符 + 字母数字串`；`字母数字串 = 0{字母或数字}7`；`字母或数字 = [字母字符 \| 数字字符]` | 标识符长度不超过8个字符，首字符必须是字母 |
| 状态图活动表语法 | `事件名(参数表)/动作表达式` | 三种标准事件：entry、exit、do |
| 对象模型5层 | 主题层 → 类与对象层 → 结构层 → 属性层 → 服务层 | 复杂系统的对象模型 |

## 四、重点对比与易混辨析

### 4.1 系统流程图 vs 数据流图

| 对比项 | 系统流程图 | 数据流图（DFD） |
|--------|------------|-----------------|
| 描绘对象 | **物理系统** | **逻辑过程** |
| 基本思想 | 用图形符号以黑盒子形式描绘组成系统的每个部件 | 描绘信息流和数据从输入移动到输出所经受的变换 |
| 是否有物理部件 | 有（程序、文档、数据库、人工过程等） | 没有任何具体的物理部件 |
| 性质 | 物理数据流图 | 系统逻辑功能的图形表示 |
| 通信工具 | — | 分析员与用户之间极好的通信工具 |

### 4.2 需求分析三种模型对比

| 模型 | 描绘内容 | 对应准则 | 图形工具 |
|------|----------|----------|----------|
| 数据模型 | 问题的信息域 | 理解并描述问题的信息域 | 实体联系图（ER图）、数据字典、类图 |
| 功能模型 | 软件应完成的功能 | 定义软件应完成的功能 | 数据流图、用例图 |
| 行为模型 | 外部事件结果的软件行为 | 描述作为外部事件结果的软件行为 | 状态转换图 |

### 4.3 面向对象分析三种模型对比

| 模型 | 性质 | 核心地位 | UML图形工具 |
|------|------|----------|-------------|
| 对象模型 | 静态的、结构化的"数据"性质 | **最基本、最重要、最核心** | 类图 |
| 动态模型 | 瞬时的、行为化的"控制"性质 | — | 序列图、活动图、状态图 |
| 功能模型 | 变化的"功能"性质 | — | 数据流图、用例图 |

### 4.4 数据流图四种基本符号

| 符号图形 | 含义 |
|----------|------|
| 正方形 | 数据的源点或终点 |
| 圆角矩形 | 变换数据的处理 |
| 开口矩形 | 数据存储 |
| 箭头 | 数据流 |

### 4.5 ER图三种基本成分

| 成分 | 符号 |
|------|------|
| 实体（实体集） | 矩形框 |
| 联系（联系集） | 菱形框 |
| 属性 | 椭圆形或圆角矩形 |

### 4.6 状态图三种状态符号

| 状态 | 符号 | 数量限制 |
|------|------|----------|
| 初态 | 实心圆 | 只能有1个 |
| 终态 | 一对同心圆（内圆为实心圆） | 0至多个 |
| 中间状态 | 圆角矩形 | — |

### 4.7 用例图四种关系

| 关系 | 描述的对象 | 表示符号 |
|------|-----------|----------|
| 关联 | 参与者**与**用例之间 | 实线 |
| 泛化 | 参与者之间**或**用例之间 | 实线箭头 |
| 包含 | 用例之间 | 虚线箭头 `<<includes>>` |
| 扩展 | 用例之间 | 虚线箭头 `<<extends>>` |

### 4.8 类图四种关系

| 关系 | 说明 | 符号 |
|------|------|------|
| 关联 | 类与类之间的连接关系 | 直线（无箭头实线） |
| 泛化（继承） | 通用元素与具体元素的分类关系 | 空心三角形连线 |
| 依赖 | 一个独立，另一个依赖于它 | 虚线 |
| 细化/实现 | 同一事物不同抽象层次的描述之间 | 虚线空心三角形 |

### 4.9 聚集的两种特殊形式

| 类型 | 说明 |
|------|------|
| 共享聚集 | 部分方对象可同时参与多个整体方对象的构成 |
| 组合聚集（组成） | 部分类完全隶属于整体类，部分与整体共存，整体不存在则部分随之消失 |

## 五、高频考点与易错提醒

1. ⚠ **可行性研究的目的**是"用最小的代价在尽可能短的时间内确定问题是否能够解决"，不是"规划如何开发项目"。(练习判断1, 单选1)
2. ⚠ **可行性研究三个方面**：技术可行性、经济可行性、操作可行性（不是"规划"或"设计"）。(练习判断2)
3. ⚠ **系统流程图描绘的是物理系统**，不是逻辑系统或程序系统。它表达数据在系统各部件之间流动的情况，是物理数据流图而不是程序流程图。(练习单选2, 判断3)
4. ⚠ **数据流图四种基本符号**是高频考点：正方形=源点/终点，圆角矩形=处理，开口矩形=数据存储，箭头=数据流。务必记准符号与含义的对应关系。(练习单选3-6, 多选2-3)
5. ⚠ **数据流图中不能由计算机处理的成分**是数据源/终点（外部实体）。(练习单选4)
6. ⚠ **顶层数据流图**描述了系统的输入与输出（由若干数据源点/终点和一个处理组成）。(练习单选3)
7. ⚠ **数据字典不包括外部实体**，只包括数据流、数据流分量、数据存储、处理四类元素。(练习单选7, 多选4)
8. ⚠ **需求分析阶段最重要的技术文档**是需求规格说明书，不是可行性分析报告或设计说明书。(练习单选8)
9. ⚠ **需求分析只回答"做什么"**，不回答"怎么做"。"怎么做"是设计阶段的任务。(练习判断7, 单选9)
10. ⚠ **数据流图是功能模型工具**，不是数据模型工具。数据模型工具是ER图、数据字典、类图。(练习单选10)
11. ⚠ **状态图是行为模型工具**，不是功能模型或数据模型工具。(练习单选11)
12. ⚠ **"3秒内响应"属于性能需求**，不是功能需求或设计约束。(练习单选12)
13. ⚠ **ER图中菱形表示联系**，矩形表示实体，椭圆/圆角矩形表示属性。(练习单选13)
14. ⚠ **结构化分析方法是面向数据流**的自顶向下逐步求精，不是面向目标、功能或对象。(练习单选14, 判断9)
15. ⚠ **需求验证的四个方面**：一致性、完整性、现实性、有效性。"确实能解决用户面对的问题"对应有效性。(练习单选15, 多选10)
16. ⚠ **OOA三种模型是对象模型、动态模型、功能模型**，不包括ER模型。(练习单选16)
17. ⚠ **用例图**是从用户使用系统的角度描述系统功能的图形表达方法。(练习单选17)
18. ⚠ **关联不是用例之间的关系**，而是参与者与用例之间的关系。用例之间的关系是泛化、包含、扩展。(练习单选18)
19. ⚠ **类图中无箭头实线段代表关联**。(练习单选19)
20. ⚠ **状态图中圆角矩形表示中间状态**，实心圆表示初态，同心圆表示终态。(练习单选20)
21. ⚠ **序列图（顺序图）是动态模型工具**，不是功能模型工具。(练习判断10)
22. ⚠ **面向对象的类相当于ER模型的实体集**。(练习判断8)
23. ⚠ **需求规格说明书不是可行性研究的依据**（可行性研究在需求分析之前）。需求规格说明书的作用是：软件验收的依据、用户与开发人员交流的工具、软件设计的依据。(练习多选5)
24. ⚠ **确定功能实现算法不属于需求分析阶段的任务**，而是详细设计阶段的任务。(练习多选7)

## 六、一句话记忆要点

1. **可行性研究**：最小代价、最短时间，确定问题能否解决；技术、经济、操作三方面。
2. **可行性研究过程**：8步——复查目标→研究现有系统→导出高层逻辑模型→定义问题→导出评价解法→推荐方针→草拟计划→书写文档。
3. **系统流程图**：描绘物理系统，数据在部件间流动，是物理数据流图非程序流程图。
4. **数据流图**：四种符号——正方形(源点/终点)、圆角矩形(处理)、开口矩形(数据存储)、箭头(数据流)；逻辑功能图形表示，通信工具。
5. **数据字典**：四类元素(数据流、数据流分量、数据存储、处理)，不含外部实体；五种符号(=、+、[]、{}、())。
6. **成本/效益分析**：三种估算(代码行、任务分解、自动估计)；四个方面(货币时间价值 $F=P(1+i)^n$、投资回收期、纯收入、投资回收率)。
7. **需求分析任务**：回答"做什么"；四准则→三模型(数据/功能/行为)；八项综合要求。
8. **获取需求方法**：访谈、面向数据流自顶向下求精、简易应用规格说明、快速原型。
9. **分析建模**：数据模型→ER图，功能模型→数据流图，行为模型→状态图；需求规格说明是最主要文档。
10. **ER图**：矩形=实体，菱形=联系，椭圆/圆角矩形=属性；三种联系(1:1, 1:N, M:N)。
11. **状态转换图**：初态(实心圆,唯一)、终态(同心圆,0~多个)、中间状态(圆角矩形)；活动表语法`事件名(参数表)/动作表达式`；三种标准事件entry/exit/do。
12. **其他图形工具**：层次方框图(树形矩形框)、Warnier图(更丰富,可表示重复和条件)、IPO图(输入-处理-输出)。
13. **验证软件需求**：一致性、完整性、现实性、有效性。
14. **面向对象分析**：三模型(对象模型最核心→类图、动态模型→序列图/活动图/状态图、功能模型→用例图)；类图四关系(关联-直线、泛化-空心三角、依赖-虚线、细化)；用例图四关系(关联-参与者与用例间、泛化、包含、扩展-用例间)；聚集两形式(共享聚集、组合聚集)。
