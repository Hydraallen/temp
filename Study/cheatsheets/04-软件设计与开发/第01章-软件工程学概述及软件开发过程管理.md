## 一、本章概览

本章是软件工程学的总纲，解决"什么是软件工程、为什么要诞生软件工程、软件如何从无到有再到废弃"这一主线问题。全章包含五大块：**软件危机与软件工程的诞生**（知识点1）、**软件工程基本概念与方法学**（知识点2–5）、**面向对象方法学**（知识点6–9）、**软件生命周期**（知识点10）、**软件过程与过程模型**（知识点11–12）。其中过程模型种类最多（瀑布、快速原型、增量、螺旋、V模型、喷泉、RUP、敏捷/XP、DevOps），是考试最高频区域；面向对象基本概念与UML建模次之。本章为后续各章（需求分析、设计、测试、项目管理等）奠定术语和框架基础。

## 二、知识点详解

### 1. 软件危机和软件工程 ⭐（练习1、2，多选1，判断1–4）

- **定义**：软件危机是指在计算机软件的开发和维护过程中所遇到的一系列严重问题。这些问题绝不仅仅是不能正常运行的软件才具有的，实际上，几乎所有软件都不同程度地存在这些问题。
- **两方面概括**：
  1. 如何开发软件，以满足对软件日益增长的需求；
  2. 如何维护数量不断膨胀的已有软件。
- **典型表现（7条）**：
  1. 对软件开发成本和进度的估计常常很不准确。
  2. 用户对"已完成的"软件系统不满意的现象经常发生。
  3. 软件产品的质量往往靠不住。
  4. 软件常常是不可维护的。
  5. 软件通常没有适当的文档资料。
  6. 软件成本在计算机系统总成本中所占的比例逐年上升。
  7. 软件开发生产率提高的速度，远远跟不上计算机应用迅速普及深入的趋势。
- **产生原因**：
  - 与软件本身特点有关：
    1. 软件是逻辑部件而非物理部件，缺乏"可见性"，管理和控制开发过程相当困难。
    2. 软件在运行过程中不会因为使用时间过长而被"用坏"；维护通常意味着改正或修改原来的设计，客观上使软件较难维护。
    3. 软件规模庞大，程序复杂性随规模增加呈指数上升。
  - 与软件开发与维护的方法不正确有关：
    1. 忽视软件需求分析的重要性，认为软件开发就是写程序并设法使之运行。
    2. 只重视程序而忽视软件配置其余成分。
    3. 轻视维护是一个最大的错误。统计数据表明，用于软件维护的费用占软件总费用的 $55\%\sim70\%$。
- **修改代价随阶段变化**：在早期引入变动涉及面少，代价较低；开发中期代价剧增；软件"已经完成"时再引入变动需付出更高代价。
- **消除软件危机的途径**：
  1. 对计算机软件有正确认识——软件是程序、数据及相关文档的完整集合。
  2. 充分认识到软件开发是各类人员协同配合的工程项目，吸收和借鉴工程项目的原理、概念、技术和方法。
  3. 应该开发和使用更好的软件工具。
  - 总之，既要有技术措施（方法和工具），又要有必要的组织管理措施。软件工程正是从管理和技术两方面研究如何更好地开发和维护计算机软件的一门新兴学科。

### 2. 软件工程的定义 ⭐（练习3）

- **1968年第一届NATO会议早期定义**："软件工程就是为了经济地获得可靠的且能在实际机器上有效地运行的软件，而建立和使用完善的工程原理。"
- **1993年IEEE定义**："软件工程是：①把系统的、规范的、可度量的途径应用于软件开发、运行和维护过程，也就是把工程应用于软件；②研究①中提到的途径。"

### 3. 软件工程的本质特性 ⭐⚠（练习3，多选2，判断5）

人们普遍认为软件工程具有下述本质特性：

1. 软件工程关注于大型程序的构造。
2. 软件工程的中心课题是控制复杂性。
3. 软件经常变化。⚠（练习判断5：在大部分项目中需求往往是不稳定的，"需求稳定不变"是错误说法）
4. 开发软件的效率非常重要。
5. 和谐地合作是开发软件的关键。
6. 必须有效地支持它的用户。
7. 在软件工程领域中通常由具有一种文化背景的人替具有另一种文化背景的人创造产品。

### 4. 软件工程的基本原理

B.W.Bohm 提出了软件工程的 7 条基本原理（确保软件产品质量和开发效率的原理的最小集合）：

1. 用分阶段的生命周期计划严格管理。
2. 坚持进行阶段评审。
3. 实行严格的产品控制。
4. 采用现代程序设计技术。
5. 结果应能清楚地审查。
6. 开发小组的人员应该少而精。
7. 承认不断改进软件工程实践的必要性。

### 5. 软件工程方法学 ⭐⚠（判断6）

- **三要素**：方法、工具和过程。⚠（练习判断6：题目说"方法、工具和模型"是错误的，应为"方法、工具和过程"）
  - **方法**：完成软件开发的各项任务的技术方法，回答"怎样做"。
  - **工具**：为运用方法而提供的自动的或半自动的软件工程支撑环境。
  - **过程**：为了获得高质量的软件所需要完成的一系列任务的框架，规定了完成各项任务的工作步骤。
- **传统方法学**（生命周期方法学/结构化范型）：
  - 采用结构化技术（结构化分析、结构化设计、结构化实现）完成软件开发各项任务。
  - 把软件生命周期全过程依次划分为若干个阶段，顺序完成每个阶段的任务。
  - 每个阶段开始和结束都有严格标准；前一阶段的结束标准就是后一阶段的开始标准。
  - 每个阶段结束前都必须进行正式严格的技术审查和管理复审。
  - 审查的一条主要标准：每个阶段都应该交出"最新式的"高质量的文档资料。
- **面向对象方法学**：
  - 以数据为主线，把数据和对数据的操作紧密结合。
  - 4个要点：
    1. 把对象作为融合了数据及在数据上的操作行为的统一的软件构件。
    2. 把所有对象都划分成类。
    3. 按照父类与子类的关系，把若干个相关类组成一个层次结构的系统。
    4. 对象彼此间仅能通过发送消息互相联系。
  - 使问题空间（问题域）与解空间（求解域）在结构上尽可能一致。
  - 继承性和多态性进一步提高了可重用性。

### 6. 面向对象方法学的优点 ⭐

1. **与人类习惯的思维方式一致**：按照人类习惯的思维方法建立问题域的模型。
2. **稳定性好**：软件结构根据问题领域的模型建立，而非基于功能分解；功能需求变化时往往仅需局部修改。
3. **可重用性好**：封装和信息隐藏使对象内部实现与外界隔离，具有较强的独立性，对象是理想的模块和可重用的软件成分。
4. **较易开发大型软件产品**：每个对象像微型程序，可把大型软件分解为一系列相互独立的小产品来处理。
5. **可维护性好**：稳定性好、比较容易修改、比较容易理解、易于测试和调试。

### 7. 面向对象的基本特性 ⭐（多选3）

1. **抽象**：忽略一个主题中与当前目标无关的方面，以便更充分地注意与当前目标有关的方面。包括过程抽象和数据抽象。
2. **封装**：把过程和数据包围起来，对数据的访问只能通过已定义的界面。对象通过受保护的接口访问其他对象。
3. **继承**：一种联结类的层次模型，允许和鼓励类的重用。新类从现有类中派生，新类称为派生类（子类），原始类称为基类（父类）。派生类可继承基类的方法和实例变量，并可修改或增加新的方法。
4. **多态性**：允许不同类的对象对同一消息作出响应。包括参数化多态性和包含多态性。具有灵活、抽象、行为共享、代码共享的优势。

### 8. 面向对象的基本概念 ⭐⚠（练习4–6，判断7）

1. **对象**：由描述该对象属性的数据以及可以对这些数据施加的所有操作封装在一起构成的统一体。
2. **类**：对具有相同数据和相同操作的一组相似对象的定义。
3. **实例**：由某个特定的类所描述的一个具体的对象。
4. **消息**：要求某个对象执行在定义它的那个类中所定义的某个操作的规格说明。例：`MyCircle.Show(GREEN)`。
5. **方法**：对象所能执行的操作，也就是类中所定义的服务。
6. **属性**：类中所定义的数据，对客观世界实体所具有的性质的抽象。
7. **封装**：把数据和实现操作的代码集中起来放在对象内部。访问控制：共有（public）、私有（private）、受保护（protect）。
8. **继承**：子类自动地共享基类中定义的数据和方法的机制。类等级为树形结构时为**单继承**；允许一个类有多个父类时为**多重继承**。⚠（练习4：子类可以定义和父类同名的操作——这是覆盖，不是"不能定义"）
9. **多态**：子类对象可以像父类对象那样使用，同样的消息既可以发送给父类对象也可以发送给子类对象。
   - **静态绑定**（Static Binding）：在编译时即可确定调用哪个函数或方法。
   - **动态绑定**（Dynamic Binding）：在运行时才能确定调用哪个函数或方法，多态属于动态绑定。⚠（练习6，判断7：动态绑定在运行时才确定，不是编译时）
10. **重载**：在同一作用域内的若干个参数特征不同的函数可以使用相同的函数名字。⚠（练习5：一个类具有多个同名而参数类型列表不同的方法称为重载）
11. **覆盖**：在子类中重新定义（覆盖）了父类中具有相同名称和参数列表的方法，是实现多态性的关键机制之一。
12. **接口**：一种规则，是对行为的抽象。在单继承中，一个类只能有一个父类，但可以实现多个接口。

### 9. 面向对象建模 ⭐⚠（练习7–9，多选4）

- **UML**（统一建模语言）：一种可视化的面向对象建模语言，是用来对真实世界物理进行建模的标准标记，用图形方式表现典型的面向对象系统的整个结构。支持从需求分析开始的软件开发的全过程。
- **UML三要素**：UML 的基本构造块、支配这些构造块如何放置在一起的规则、用于整个语言的公共机制。
- **UML词汇表（三种构造块）**：
  - **事物**：结构事物（类、接口、协作、活动类、用例、组件、节点等）、行为事物（消息、状态等）、分组事物（包）、注释事物。
  - **关系**：依赖、关联、泛化、实现。
  - **图**。
- **三种模型**：
  - **功能模型**：从用户角度展示系统功能，包括**用例图**。
  - **对象模型**：采用对象、属性、操作、关联等概念展示系统结构和基础，包括**类图、对象图**。⚠（练习9：描述系统数据结构的是对象模型）对象模型始终是最重要、最基本、最核心的。
  - **动态模型**：展现系统内部行为，包括**序列图、活动图、状态图**。
- **UML主要图例（10种）**：

| 图名 | 功用 |
|------|------|
| 用例图 | 描述角色以及角色与用例之间的连接关系；说明谁要使用系统及可以使用系统做什么 |
| 类图 | 描述系统中的类及各个类之间的关系的静态视图 |
| 对象图 | 类图的实例，显示类的多个对象实例，描述对象之间的关系 |
| 活动图 | 描述用例要求所要进行的活动及活动间的约束关系，有利于识别并行活动 |
| 状态图 | 描述类的对象所有可能的状态及事件发生时状态的转移条件 |
| 序列图（时序图） | 显示参与者如何以一系列顺序步骤与系统对象交互，强调消息序列 |
| 协作图 | 显示对象间的动态合作关系，可看成类图和顺序图的交集；强调上下级关系时选择协作图 |
| 构件图（组件图） | 描述代码构件的物理结构及各种构建之间的依赖关系 |
| 包图 | 用类似文件夹的符号表示模型元素的组合 |
| 部署图（配置图） | 用于建模系统的物理部署，如计算机和设备及其连接方式 |

  - 序列图与协作图合称为**交互图**。强调时间和顺序用序列图，强调上下级关系用协作图。
  - ⚠（练习8，多选4：数据流图**不属于**UML图例）

### 10. 软件生命周期 ⭐⚠（练习10–13，多选5–6，判断8）

- **定义**：软件生命周期是软件的产生直到报废或停止使用的生命周期。
- **三大时期、八个阶段**：

| 时期 | 阶段 | 关键问题/任务 |
|------|------|--------------|
| **软件定义时期**（系统分析） | ①问题定义 | "要解决的问题是什么？" |
| | ②可行性研究 | "对于上一个阶段所确定的问题有行得通的解决办法吗？"研究问题范围，探索是否值得去解 |
| | ③需求分析 | "目标系统必须做什么？"确定目标系统必须具备哪些功能；用数据流图、数据字典和简要算法表示系统逻辑模型；产出规格说明书 |
| **软件开发时期** | ④总体设计（概要设计） | "概括地说，应该怎样实现目标系统？"设计几种可能方案（低、中、高成本），推荐最佳方案；设计程序体系结构（模块组成及模块间关系） |
| | ⑤详细设计 | "应该怎样具体地实现这个系统？"设计每个模块的详细规格说明，确定算法和数据结构（程序流程图、PAD图、N-S图等） |
| | ⑥编码和单元测试 | 写出正确的容易理解容易维护的程序模块；仔细测试每一个模块 |
| | ⑦综合测试 | 通过集成测试、验收测试等方法进一步测试检验 |
| **运行维护时期**（软件维护） | ⑧软件维护 | 通过各种必要的维护活动使系统持久地满足用户需要 |

  - 其中④⑤又称**系统设计**，⑥⑦又称**系统实现**。
  - ⚠（练习12，多选5，判断8：软件定义时期的三个阶段依次是**问题定义→可行性研究→需求分析**；需求分析是软件定义时期的最后一个阶段）
- **四类维护活动**：

| 维护类型 | 含义 | 工作量占比 |
|----------|------|-----------|
| 改正性维护 | 诊断和改正在使用过程中发现的软件错误 | $17\%\sim21\%$ |
| 适应性维护 | 修改软件以适应环境的变化 | $18\%\sim25\%$ |
| 完善性维护 | 根据用户的要求改进或扩充软件使它更完善 | $50\%\sim66\%$ |
| 预防性维护 | 修改软件，为将来的维护活动预先做准备 | $4\%$ |

  - 维护工作量在软件生命周期中所占比例：**60%以上**。⚠（练习11：工作量最大的阶段是软件维护；练习13：改进或扩充软件属于完善性维护）

### 11. 软件过程的定义 ⭐（多选7）

- **定义**：软件过程是为了获得高质量软件所需要完成的一系列任务的框架，它规定了完成各项任务的工作步骤。
- 过程定义了：运用方法的顺序、应该交付的文档资料、为保证软件质量和协调变化所需要采取的管理措施、标志软件开发各个阶段任务完成的里程碑。
- 没有一个适用于所有软件项目的任务集合。科学、有效的软件过程应该定义一组适合于所承担的项目特点的任务集合。

### 12. 过程模型 ⭐⚠（练习14–20，多选8–10，判断9–10）

#### 12.1 瀑布模型 ⭐⚠（练习15、16，多选9，判断9）

- **特点**：
  1. **阶段间具有顺序性和依赖性**：①必须等前一阶段工作完成才能开始后一阶段；②前一阶段的输出文档就是后一阶段的输入文档。
  2. **推迟实现的观点**：编码之前设置系统分析与系统设计阶段，主要考虑逻辑模型，不涉及物理实现。
  3. **质量保证的观点**：①每个阶段必须完成规定的文档；②每个阶段结束前对文档进行评审。
- 实际的瀑布模型是带"反馈环"的。
- **优点**：
  1. 可强迫开发人员采用规范的方法（如结构化技术）。
  2. 严格地规定了每个阶段必须提交的文档。
  3. 要求每个阶段交出的所有产品都必须经过质量保证小组的仔细验证。
- **缺点**：
  1. 只能通过文档了解产品，不经过实践的需求是不切实际的。
  2. 实际项目很少按照该模型给出的顺序进行。
  3. 用户常常难以清楚地给出所有需求。
  4. 用户必须有耐心，等到系统开发完成。
- **适用**：需求是预知的；软件实现方法是成熟的；项目周期较短。
- ⚠（练习15：瀑布模型突出缺点是不适应用户需求的变动；练习16："规定了每个阶段必须完成的文档"是优点而非缺点；判断9：瀑布模型是文档驱动的）

#### 12.2 快速原型模型

- **快速原型**：快速建立起来的可以在计算机上运行的程序，所能完成的功能往往是最终产品功能的一个子集。
- **主要优点**：不带反馈环，软件产品开发基本上是线性顺序进行。
  - 原因：①原型系统已通过与用户交互得到验证，规格说明文档正确描述了用户需求；②开发人员通过建立原型系统学到了许多东西，设计和编码阶段错误可能性较小。

#### 12.3 增量模型 ⭐（判断10）

- 也称渐增模型。把软件产品作为一系列的增量构件来设计、编码、集成和测试。
- 每个构件由多个相互作用的模块构成，能够完成特定的功能。
- 第一个增量构件往往实现软件的基本需求，提供最核心的功能。
- ⭐（判断10：增量模型能在较短时间内向用户提交可完成部分工作的产品）

#### 12.4 螺旋模型 ⭐（练习17）

- **主要优势**：风险驱动的。
- **弱点**：除非软件开发人员具有丰富的风险评估经验和专门知识，否则将出现真正的风险——当项目实际上正在走向灾难时，开发人员可能还认为一切正常。
- **适用**：庞大而复杂的、高风险的系统。
- 螺旋模型包含简化螺旋模型和完整螺旋模型，每个阶段之前都增加了风险分析过程。
- ⭐（练习17：强调风险分析的软件过程模型是螺旋模型）

#### 12.5 V模型

- 将软件开发和测试阶段对称地放置在一个V形图中。
- **重要意义**：强调开发和测试同等重要，非常明确地表明测试过程中存在的不同级别，清晰描述测试阶段和开发阶段的对应关系。

#### 12.6 喷泉模型

- "喷泉"体现面向对象软件开发过程**迭代**和**无缝**的特性。
- 不同阶段的圆圈相互重叠表示两个活动之间存在交迭。
- 面向对象方法在概念和表示方法上的一致性，保证了各项开发活动之间的**无缝过渡**。
- 阶段内的向下箭头代表该阶段内的迭代（或求精）。
- 较小的圆圈代表维护，象征采用面向对象范型后维护时间缩短。
- **适用**：面向对象软件开发。

#### 12.7 Rational统一过程（RUP） ⭐（练习18、19，多选10）

- **6条最佳实践**：
  1. 迭代式开发
  2. 管理需求
  3. 使用基于构件的体系结构
  4. 可视化建模（与UML紧密联系）
  5. 验证软件质量
  6. 控制软件变更
- **二维生命周期模型**：纵轴代表核心工作流，横轴代表时间。
- **9个核心工作流**：
  - 前6个**核心过程工作流**：业务建模、需求、分析与设计、实现、测试、部署。
  - 后3个**核心支持工作流**：配置与变更管理、项目管理、环境。
- **4个连续阶段**（每阶段有明确目标和里程碑）：

| 阶段 | 目标 |
|------|------|
| 初始阶段 | 建立业务模型，定义最终产品视图，确定项目范围 |
| 精化阶段 | 设计并确定系统的体系结构，制定项目计划，确定资源需求 |
| 构建阶段 | 开发出所有构件和应用程序，集成为客户需要的产品，详尽测试所有功能 |
| 移交阶段 | 把开发出的产品提交给用户使用 |

  - ⚠（练习18：RUP的4个阶段是初始、精化、构建、移交，"确认阶段"不属于RUP；练习19：空格处填"精化阶段"）
- 每次循环经历完整生命周期，每次循环结束向用户交付产品的一个可运行版本。强调迭代和渐增方式开发软件。

#### 12.8 敏捷过程与极限编程 ⭐（练习20）

- **敏捷软件开发宣言（4个价值观）**：
  1. 个体和交互胜过过程和工具
  2. 可以工作的软件胜过面面俱到的文档
  3. 客户合作胜过合同谈判
  4. 响应变化胜过遵循计划
- **敏捷过程特点**：以人为核心、迭代、循序渐进的开发方法；强调程序员团队与业务专家紧密协作、面对面沟通、频繁交付新版本、紧凑而自我组织型的团队、能适应需求变化。
- **适用**：商业竞争环境下小型项目的有限资源和有限开发时间约束；可作为对RUP的补充和完善。
- **局限**：作为一种软件过程模式，远不如RUP全面和完整。
- **极限编程（XP）**：敏捷过程中最富盛名的一个，"极限"指把好的开发实践运用到极致。广泛应用于需求模糊且经常改变的场合。
  - **实践准则**：客户作为开发团队的成员、使用用户素材、短交付周期、验收测试、结对编程、测试驱动开发、集体所有、持续集成、可持续的开发速度、开放的工作空间、及时调整计划、简单的设计、重构、使用隐喻。
- **其他敏捷模型**：Scrum、Crystal、AUP、Kanban等。⚠（练习20：增量模型不属于敏捷过程）

#### 12.9 DevOps

- **定义**：Development 和 Operations 的合成词，用于促进开发、技术运营和质量保障（QA）部门之间的沟通、协作与整合。重视"软件开发人员"和"IT运维技术人员"之间沟通合作的文化、运动或惯例。
- 在整个软件供应链中应用敏捷和精益开发原则，完善了敏捷开发存在的短板，实现真正的闭环。
- **5个持续循环阶段**：
  1. **持续开发**：将软件可交付成果分解到多次冲刺中开发；开发编写功能代码→提交到仓库（Git、SVN）→构建，把代码打包到可执行文件（ant、maven）。
  2. **持续测试**：使用自动化测试工具（testing、Junit、unittest、pytest、selenium、appium）对新功能进行测试。
  3. **持续集成**：测试通过的新功能代码与原先代码合并（Jenkins）；每次集成都需自动化构建进行验证，包含自动化测试。
  4. **持续部署**：测试通过的部署到服务器上（Kubernetes、docker），让不同环境一致。
  5. **持续监控**：部署上线后持续监控行为、性能（ELK）。

## 四、重点对比与易混辨析

### 4.1 传统方法学 vs 面向对象方法学

| 维度 | 传统方法学（生命周期方法学/结构化范型） | 面向对象方法学 |
|------|----------------------------------------|---------------|
| 核心主线 | 功能分解 | 数据为主线，数据与操作紧密结合 |
| 技术手段 | 结构化分析、结构化设计、结构化实现 | 对象、类、继承、消息 |
| 阶段关系 | 严格划分阶段，顺序完成 | 分析设计编码间无明显边界，无缝过渡 |
| 可重用性 | [原文未述] | 继承性和多态性提高可重用性 |
| 稳定性 | [原文未述] | 功能需求变化时往往仅需局部修改 |

### 4.2 面向对象基本特性 vs 基本概念

| 类别 | 内容 |
|------|------|
| 基本特性（4个） | 抽象、封装、继承、多态性 |
| 基本概念（12个） | 对象、类、实例、消息、方法、属性、封装、继承、多态、重载、覆盖、接口 |

### 4.3 重载 vs 覆盖

| 维度 | 重载 | 覆盖 |
|------|------|------|
| 作用域 | 同一作用域内 | 子类与父类之间 |
| 参数特征 | 参数特征不同 | 相同名称和参数列表 |
| 目的 | 同名函数处理不同参数 | 子类修改继承方法的行为，实现多态性 |

### 4.4 静态绑定 vs 动态绑定

| 维度 | 静态绑定 | 动态绑定 |
|------|---------|---------|
| 确定时机 | 编译时 | 运行时 |
| 典型应用 | 静态方法 | 多态 |

### 4.5 软件过程模型对比

| 模型 | 核心特点 | 优点 | 缺点 | 适用场景 |
|------|---------|------|------|---------|
| 瀑布模型 | 阶段间顺序性和依赖性；推迟实现；质量保证（文档+评审） | 强迫采用规范方法；严格规定文档；产品经质量保证小组验证 | 只能通过文档了解产品；实际项目很少按顺序进行；用户难以给出所有需求；用户须等到开发完成 | 需求预知；实现方法成熟；项目周期较短 |
| 快速原型模型 | 不带反馈环；线性顺序进行 | 原型已验证；开发人员已学到经验，减少错误 | [原文未述] | [原文未述] |
| 增量模型 | 一系列增量构件设计、编码、集成和测试；第一构件实现基本需求 | 能在较短时间内向用户提交可完成部分工作的产品 | [原文未述] | [原文未述] |

| 喷泉模型 | 迭代和无缝；阶段圆圈重叠表示活动交迭；阶段内向下箭头代表迭代 | 无缝过渡；维护时间缩短 | [原文未述] | 面向对象软件开发 |

## 五、高频考点与易错提醒

1. ⚠ **软件危机的定义**是高频考点——指开发和维护过程中遇到的一系列严重问题，不是病毒、犯罪或迷恋计算机。（练习1）
2. ⚠ **软件的组成**——程序、数据及相关文档的完整集合，"记录"不属于软件组成。（练习2）
3. ⚠ **软件工程本质特性**——"软件经常变化"是本质特性，"需求稳定不变"的说法是错误的。（练习3，判断5）
4. ⚠ **软件工程方法学三要素**——是方法、工具和**过程**，不是"模型"。（判断6）
5. ⚠ **继承与覆盖的区别**——子类可以定义和父类同名的操作（即覆盖），不是"不能定义"。（练习4）
6. ⚠ **重载的定义**——同一作用域内参数特征不同的函数使用相同函数名。（练习5）
7. ⚠ **静态绑定 vs 动态绑定**——动态绑定在运行时确定，多态属于动态绑定；静态绑定在编译时确定。（练习6，判断7）
8. ⚠ **UML图例辨析**——数据流图**不属于**UML图例；UML是可视化建模语言，不是程序设计语言。（练习7、8，多选4）
9. ⚠ **三种模型对应关系**——对象模型描述数据结构（类图、对象图），功能模型描述功能（用例图），动态模型描述内部行为（序列图、活动图、状态图）。（练习9）
10. ⚠ **软件定义时期阶段顺序**——问题定义→可行性研究→需求分析；需求分析是软件定义时期最后阶段。（练习12，多选5，判断8）
11. ⚠ **维护工作量最大**——软件维护占软件生命周期工作量60%以上。（练习11）
12. ⚠ **四类维护辨析**——完善性维护（改进/扩充，占比最高50%–66%）、适应性维护（适应环境变化）、改正性维护（改错）、预防性维护（为将来做准备，占比最低4%）。（练习13，多选6）
13. ⚠ **动态模型不是过程模型**——动态模型是面向对象开发的三种模型之一，不是软件过程模型。（练习14）
14. ⚠ **瀑布模型缺点辨析**——"规定了每个阶段必须完成的文档"是优点而非缺点；瀑布模型突出缺点是不适应需求变动；瀑布模型是文档驱动的。（练习15、16，判断9）
15. ⚠ **螺旋模型=风险分析**——强调风险分析的模型是螺旋模型，适用于庞大复杂高风险系统。（练习17）
16. ⚠ **RUP四阶段**——初始、精化、构建、移交；"确认阶段"不属于RUP。（练习18、19）
17. ⚠ **RUP核心过程工作流**——前6个为业务建模、需求、分析与设计、实现、测试、部署；后3个为配置与变更管理、项目管理、环境。（多选10）
18. ⚠ **敏捷过程辨析**——极限编程、Scrum、Crystal、AUP、Kanban属于敏捷；增量模型不属于敏捷过程。（练习20）
19. ⚠ **增量模型特点**——能在较短时间内向用户提交可完成部分工作的产品。（判断10）
20. ⚠ **瀑布模型是文档驱动的**——前一阶段输出文档就是后一阶段输入文档，每个阶段必须提交文档并评审。（判断9）

## 六、一句话记忆要点

1. **软件危机**＝开发+维护中的一系列严重问题；消除途径＝技术措施+组织管理措施。
2. **软件工程**＝用系统的、规范的、可度量的工程途径开发运行维护软件。
3. **方法学三要素**＝方法（怎么做）+工具（自动/半自动支撑）+过程（任务框架）。
4. **面向对象四特性**＝抽象、封装、继承、多态。
5. **重载**＝同名不同参（同一作用域）；**覆盖**＝同名同参（子类重定义父类方法）。
6. **静态绑定**编译时确定，**动态绑定**运行时确定，多态＝动态绑定。
7. **UML三模型**＝功能模型（用例图）+对象模型（类图/对象图，最核心）+动态模型（序列图/活动图/状态图）。
8. **软件生命周期**＝定义（问题定义→可行性研究→需求分析）+开发（总体设计→详细设计→编码单元测试→综合测试）+维护。
9. **维护占生命周期60%以上**，其中完善性维护占比最高（50%–66%）。
10. **瀑布模型**＝顺序依赖、推迟实现、文档驱动；缺点＝不适应需求变更。
11. **螺旋模型**＝风险驱动，适合庞大复杂高风险系统。
12. **RUP**＝4阶段（初始→精化→构建→移交）+9工作流（6过程+3支持）+6最佳实践。
13. **敏捷宣言**＝个体交互>过程工具、工作软件>文档、客户合作>合同谈判、响应变化>遵循计划。
14. **DevOps**＝5持续（开发→测试→集成→部署→监控），Dev与Ops全生命周期协作。
